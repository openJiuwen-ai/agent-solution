# Governance 配置指南

## 什么是 Governance

Governance（治理配置）是 EDPAgent 的核心配置体系，通过三层 YAML 文件定义 Agent 的角色定位、行为边界、执行约束和业务话术。所有业务逻辑的差异化都通过 Governance 配置实现，**无需修改 Java 代码**。

## 三层配置体系

| 配置文件 | 位置 | 职责 | 覆盖策略 |
|----------|------|------|----------|
| `planrule.yaml` | `engine/src/main/resources/governance/` | 定义 Agent 角色、职责边界、ReAct 协议、工具调用规约、Skill 路由规则 | 框架级默认；场景级可追加 additional_prompt、覆盖 scope/supplementary_prompt |
| `actrule.yaml` | `engine/src/main/resources/governance/` | 定义执行约束：最大子任务数、最大步数、允许的工具列表、工具调用次数上限、Skill 加载模式 | 框架级默认；场景级继承式覆盖 |
| `scriptconfig.yaml` | `engine/src/main/resources/governance/` | 定义话术配置：通用话术、思维链话术、ask_user 模板 | 框架级默认；场景级可覆盖特定话术 |

## planrule.yaml 详解

planrule.yaml 定义 Agent 是谁、负责什么、不能做什么、遵循什么协议。

### 完整结构

```yaml
planrule:
  # Agent 角色（继承式覆盖）
  role: "你的身份是通用动态规划智能体"

  # Agent 描述（继承式覆盖）
  description: "你的核心职责是负责任务规划、执行和结果总结。..."

  # 业务范围（替代式覆盖）
  scope:
    allowed: ""    # 允许的业务范围，空表示不限
    denied: ""     # 禁止的业务范围

  # 补充提示词（base_protocol 不可覆盖，additional_prompt 可追加）
  supplementary_prompt:
    base_protocol: |
      你采用「规划—执行—观察—反思」ReAct 循环处理用户请求。
      ## 任务规划协议（核心）
      ...（框架内置完整协议，不可修改）
    additional_prompt: ""  # 场景可追加特有规则

  # Skill 路由规则（场景级配置）
  skill_routing:
    - trigger: "用户首次请求推荐理财产品"
      skill: "product_recommend_skill"
      priority: 1
```

### 核心字段说明

#### role（角色）

定义 Agent 的身份。**注意**：框架默认要求自我介绍时必须说"你好，我是通用动态规划智能体"，场景不应覆盖此身份，只能在 scenario_description 中说明业务上下文。

✅ 正确：场景只配置 additional_prompt 增加业务规则，不修改 role

#### description（描述）

对 Agent 职责的详细描述，直接作为 System Prompt 的一部分发给 LLM。

#### scope（业务范围）

定义 Agent 能做什么、不能做什么：

- `allowed`：允许的业务范围列表，非空时 Agent 只能处理范围内的请求
- `denied`：禁止的业务范围列表，范围内的请求直接拒绝

示例（理财场景）：

```yaml
scope:
  allowed: "理财产品推荐、筛选、购买、银行账户余额查询、银行账户间转账、资金筹划"
  denied: "基金相关业务、股票相关业务、保险相关业务、贷款相关业务、信用卡相关业务"
```

当 scope 未配置（allowed 和 denied 均为空）时，Agent 通过 Skill 路由判断是否能处理请求。

#### supplementary_prompt（补充提示词）

分为两部分：

- `base_protocol`：框架内置的 ReAct 协议，**不可覆盖**，定义了核心的任务规划、状态机、工具调用规约
- `additional_prompt`：场景可追加的特有规则，追加到 base_protocol 之后

理财场景的 additional_prompt 示例：

```yaml
additional_prompt: |
  ## 理财场景强制规则（不可违反）
  - 当用户提出业务范围内的请求时，你**必须**立即调用 `todo_create` 工具创建任务列表，**禁止直接用文本回答**。
  - 每次只执行一个 tool_call，不要并发。
  - 优先使用理财场景预定义的 catalog_id。
```

#### skill_routing（Skill 路由）

定义触发条件与 Skill 的映射关系，按 priority 从小到大匹配：

```yaml
skill_routing:
  - trigger: "用户首次请求推荐理财产品"
    skill: "product_recommend_skill"
    priority: 1
  - trigger: "用户从推荐结果中选择产品"
    skill: "product_select_skill"
    priority: 2
```

| 字段 | 说明 |
|------|------|
| `trigger` | 触发条件描述，LLM 据此判断是否匹配 |
| `skill` | 对应的 Skill 目录名 |
| `priority` | 优先级，数字越小越优先 |

## actrule.yaml 详解

actrule.yaml 定义 Agent 的执行约束和工具限制。

### 完整结构

```yaml
actrule:
  # 单层规划中最多创建多少个子任务
  max_subtasks: 50

  # 整个会话最多执行多少步（工具调用+推理迭代）
  max_steps: 100

  # 是否启用任务循环（完成后可根据新意图重新规划）
  enable_task_loop: true

  # Skill 加载模式
  skill_mode: all  # all / auto_list / none

  # 允许调用的工具白名单
  allowed_tools:
    - bash
    - skill_tool
    - call_versatile
    - call_mcp
    - ask_user
    - todo_create
    - todo_modify
    - todo_list
    - todo_get
    - cancel_task

  # 单个工具的调用次数上限（防止死循环）
  tool_limits:
    call_versatile: 50
    call_mcp: 50
    ask_user: 50
    execute_cmd: 50
```

### 字段说明

#### max_subtasks（最大子任务数）

限制单次 todo_create 调用最多创建的子任务数量，防止 LLM 创建过多任务。默认 50。

#### max_steps（最大执行步数）

限制整个会话的最大执行步数（每轮 LLM 推理+工具调用算一步），防止死循环。默认 100。达到限制后任务强制结束。

#### enable_task_loop（任务循环）

是否允许任务循环。为 `true` 时，所有任务完成后如果用户有新意图，Agent 可以重新规划新任务；为 `false` 时，所有任务完成后会话结束。默认 `true`。

#### skill_mode（Skill 加载模式）

| 值 | 说明 |
|----|------|
| `all` | 列出所有 Skill，Agent 直接判断匹配（适合 Skill 数量少的场景，默认） |
| `auto_list` | 动态搜索，Agent 先调用 list_tool 查询匹配 Skill（适合 Skill 数量多的场景） |
| `none` | 关闭 Skill 系统，不加载任何 Skill |

#### allowed_tools（工具白名单）

Agent 可以调用的工具列表。**这是硬约束**，不在列表中的工具不会注册，LLM 也无法调用。

框架默认包含：
- DeepAgent 原生工具：`bash`、`skill_tool`、`todo_create`、`todo_modify`、`todo_list`、`todo_get`
- EDPAgent 业务工具：`call_versatile`、`call_mcp`、`ask_user`、`cancel_task`

场景配置可以**追加**工具到列表中（叠加合并），但不能移除框架默认工具。

#### tool_limits（工具调用次数限制）

对特定工具设置单会话调用次数上限，防止死循环：

```yaml
tool_limits:
  call_versatile: 50
  ask_user: 50
```

当某个工具调用次数达到上限时，ExecutionLimitRail 会拦截并返回错误。

## scriptconfig.yaml 详解

scriptconfig.yaml 定义 Agent 与人交互的话术配置。

### 完整结构

```yaml
scriptconfig:
  # 通用话术
  general_scripts:
    tool_start: "正在调用：{tool_name}"
    tool_end: "{tool_name} 执行完成"
    todo_start: "开始执行：{title}"
    todo_end: "{title} 已完成"
    todolist_start: "规划任务清单"
    todolist_end: "todolist规划完成"
    interrupt_start: "需要您确认以下信息"
    request_start: "您的请求已收到。"
    planning_start: "我们正在为您进行规划。"
    task_cancelled: "好的，已为您取消当前操作。如需其他帮助，请随时告诉我。"
    cancel_confirm: "确认要取消当前操作吗？"
    out_of_scope: "正在学习中，暂不支持该业务。"

  # 思维链话术配置
  think_chunk_scripts:
    think_chunk_mode: fixed_script  # real_stream / fixed_script
    think_chunk_fixed_scripts:
      enabled: true
      chars_per_frame: 4
      tokens_between_frames: 2
      min_interval_ms: 50

      default_scripts:
        - "正在分析您的需求..."

      query_patterns: []

      execution_scripts:
        - "正在分析执行结果..."

      resume_scripts:
        - "当前业务步骤已为您处理完毕"

  # ask_user 确认话术
  ask_user_confirm: {}
```

### 通用话术变量

| 话术 Key | 说明 | 可用变量 |
|----------|------|----------|
| `tool_start` | 工具调用开始时推送 | `{tool_name}` |
| `tool_end` | 工具调用完成时推送 | `{tool_name}` |
| `todo_start` | 任务开始执行时推送 | `{title}` |
| `todo_end` | 任务完成时推送 | `{title}` |
| `todolist_start` | 任务规划开始时推送 | - |
| `todolist_end` | 任务规划完成时推送 | - |
| `interrupt_start` | ask_user 中断时推送 | - |
| `request_start` | 收到用户请求时推送 | - |
| `planning_start` | 开始规划时推送 | - |
| `task_cancelled` | 任务取消完成时推送 | - |
| `cancel_confirm` | 取消确认提示 | - |
| `out_of_scope` | 业务范围外请求提示 | - |

### 思维链话术配置

`think_chunk_mode` 支持两种模式：

- `real_stream`：直接推送 LLM 的真实 thinking 流式分片（需要模型支持 thinking 输出）
- `fixed_script`：用预定义固定话术帧替代 LLM thinking，适合不希望暴露真实推理过程的场景

`fixed_script` 模式参数：

| 参数 | 说明 |
|------|------|
| `chars_per_frame` | 每帧字符数，控制逐字渲染效果，0=不切分整句推送 |
| `tokens_between_frames` | 每累积多少个 LLM token 推送一个子帧 |
| `min_interval_ms` | 子帧间最小间隔（毫秒），控制渲染速度 |

三阶段话术：

- `default_scripts`：planning 阶段（第1轮思考）使用，query_patterns 未命中时降级使用
- `query_patterns`：按用户 query 关键词匹配的话术组，首个命中生效
- `execution_scripts`：executing 阶段（工具调用后反思）使用
- `resume_scripts`：中断后续轮（用户回复"continue"或继续操作）使用

## 配置调试技巧

1. **查看启动日志**：启动时日志会输出加载的配置信息，检查是否有配置解析错误。

2. **DEBUG 日志**：设置 `EDP_AGENT_LOG_LEVEL=DEBUG`，可以看到完整的 Prompt 构建过程，包括 Governance 配置是如何拼接成 System Prompt 的。

3. **配置验证**：启动时 EdpConfigValidator 会校验配置格式，如果有 YAML 语法错误或字段类型错误，会在日志中报错。

4. **Redis 清理**：修改配置后重启服务时，如果遇到奇怪的状态问题，清理 Redis 中的旧数据：
   ```bash
   docker exec -it edp-redis redis-cli FLUSHDB
   ```

## 注意事项

1. **YAML 缩进**：YAML 对缩进敏感，使用 2 空格缩进，不要使用 Tab。

2. **特殊字符**：description 和 prompt 中包含中文冒号、引号等特殊字符时，建议用双引号包裹字符串。

3. **additional_prompt 不要写太长**：System Prompt 过长会影响 LLM 推理效果和成本，建议 additional_prompt 控制在 1000 字以内。

4. **tool_limits 不要设置太小**：正常业务流程可能需要多次调用工具（如 ask_user 可能会问2-3次），建议限制不低于 20。

5. **场景配置只写差异**：场景配置不需要写所有字段，只写与框架默认不同的部分即可，未配置的字段自动继承框架默认值。
