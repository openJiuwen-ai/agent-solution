上篇 · 生态定位与改造方案

1. EvoAgent 与生态定位

1.1 EvoAgent 是什么

EvoAgent 是基于 agent-core 的自进化元 Agent，面向平台或运维侧提供 Skill 优化编排能力：接收优化指令后，经 Adapter 远程触发业务对话、采集轨迹、热更 Skill， 并输出优化报告。它是 SkillOpt 方案在业务侧的使用层 / 编排层，不承载具体业务对话逻辑。

核心能力

  • Skill 自动优化：ReflACT 管线（reflect / aggregate / select / apply 等）
  • 远程 rollout：经 Adapter 触发业务 Agent 对话并收集轨迹
  • Skill 热更：经 Adapter 写回 Skill 正文，支撑下一轮迭代
  • 双入口：CLI（dataset.yaml）与 API（/optimize、SSE 进度）

边界（不负责）

  • 企业对话业务逻辑 → 业务 Agent（如 EDPAgent）
  • 原始日志结构化 → EvoAgentAdapter
  • Linux 沙箱生命周期 → jiuwenbox

1.2 四组件定位

组件角色典型职责
EvoAgent 优化编排方 下发优化任务；经 Adapter 发起对话、查询 traces、更新 Skill
EvoAgentAdapter Sidecar 集成面 轨迹采集（日志 → JSONL）；调用代理（SSE 转发）；Skill 存储与热更
EDPAgent 等业务 Agent 业务执行体 规则驱动动态规划（DPA）；SysOperation（read_file / shell）；sandbox 模式下在 jiuwenbox 内执行
jiuwenbox 沙箱运行时 bwrap + policy 管理沙箱；File API / exec；为业务 Agent 提供隔离文件系统

2. 整体工作流程

以「一次 Skill 优化迭代」为视角，四组件的典型协作顺序如下（§8 给出实验侧具体步骤）。

1
启动与注册EDPAgent 启动 → jiuwenbox 创建 sandbox → 上传/解压 Skill → 注册 SysOperation；日志写出 sandbox_id
2
Adapter 就绪轮询 process_*.log；若 skill_backend=jiuwenbox，解析 sandbox_id 并装配沙箱 SkillStore
3
rollout 对话EvoAgent → Adapter → EDPAgent；业务在沙箱内 read_file / 执行脚本
4
轨迹采集EDP 写 TAG 日志 → Adapter 增量解析 → EvoAgent 通过 traces API 取回
5
Skill 热更EvoAgent → Adapter update_skill → jiuwenbox upload 写沙箱 SKILL.md
6
优化闭环EvoAgent 基于 traces 与新 Skill 继续 reflect / apply，输出 OptimizeReport

3. 问题：旧挂载为何失效

旧假设：共享宿主机目录

Adapter / EDP / jiuwenbox 三方 bind 同一 HOST_SKILLS，再由 policy 映射进沙箱。

前提：沙箱看到的是同一块宿主机盘。

EDP sandbox 实际路径

启动时本地 zip → upload → 解压到 {SKILL_TARGET_PATH}（默认 /tmp）；因 zip 含 skills/ 前缀，skills 根为 {SKILL_TARGET_PATH}/skills（默认 /tmp/skills）。

运行时只读该 sandbox_id 的独立 FS，改宿主机文件不会进入沙箱。

在「纯 zip 上传、无 skills bind」部署下，Adapter 只写宿主机 skills 无法热更新沙箱内 Skill，必须改写到 jiuwenbox File API。

4. 虚拟机部署 → 沙箱容器改造对比

原虚拟机 / 裸机 / 共享挂载部署中，Adapter 与业务 Agent 共用宿主机 skills_dir 与 log_dir，Skill 热更即「写共享目录 → 下次 read_file 生效」。 当 EDPAgent 开启 sandbox 模式（SANDBOX_URL → jiuwenbox）后， Skill 运行时真源变为沙箱内文件系统，Adapter 必须改造 Skill 写路径； 三组件以独立容器组网联调。

6. 实验目标与验证范围

在「EDPAgent sandbox + jiuwenbox + Adapter」三容器拓扑下，验证 Adapter 轨迹采集、调用代理、Skill 热更是否仍可闭环；重点证明热更后的 Skill 能被后续对话真实读到（不仅文件层变更）。

  • 轨迹采集：能否从 EDPAgent process_*.log 抽出对话轨迹
  • 调用代理：经 Adapter HTTP 能否完成业务对话（ExampleLLM 模型）
  • Skill 热更：Adapter 经 jiuwenbox File API 写沙箱后，新会话 read_file 是否读到新正文

8.1 启动与初始化

  1. 创建 Docker 网络 evo-sandbox-net，启动 redis / jiuwenbox / edpagent / evo-adapter。

10. 配置说明与兼容性

方案为 opt-in：仅当 Agent 条目显式配置 skill_backend: jiuwenbox 时启用沙箱热更； 默认 local，虚拟机 / 共享挂载部署无需改配置。

10.1 Adapter agents[] 新增字段

字段默认说明
skill_backend local 沙箱热更时改为 jiuwenbox
jiuwenbox_url null jiuwenbox 模式下必填，如 http://jiuwenbox:8321
sandbox_id_resolve from_logs 选型优先级：from_logs > fixed > list_ready（运行时仅 from_logs→list_ready 自动回退）
sandbox_id null fixed 时必填
remote_skills_dir /tmp/skills 对齐 EDP SKILL_TARGET_PATH/skills

既有字段继续沿用：log_dir（轨迹与 from_logs）、 skills_dir（jiuwenbox 模式下主要存 .meta）、 agent_url / project_id / agent_id（调用代理）。

10.2 沙箱场景 YAML 示例

agents:
  - name: edp_agent
    log_dir: /data/logs/edp_agent
    skills_dir: /data/skills/edp_agent
    agent_url: http://edpagent:8090
    project_id: default
    agent_id: edp_agent
    skill_backend: jiuwenbox
    jiuwenbox_url: http://jiuwenbox:8321
    sandbox_id_resolve: from_logs
    remote_skills_dir: /tmp/skills

10.3 EDPAgent 侧环境变量

变量说明
SANDBOX_URL非空则 sandbox 模式，如 http://jiuwenbox:8321
SKILL_TARGET_PATH默认 /tmp；skills 根为 {该值}/skills
LOG_DIR需设置以产出 process_*.log（轨迹 + from_logs）

10.4 兼容性

  • 默认 skill_backend: local，与改前行为一致。
  • 仅当出现 jiuwenbox 后端时才装配新 SkillStore；否则仍走原工厂逻辑。
  • 官方默认配置不强制新增字段 → 现有部署零配置变更。
  • 轨迹采集、调用代理 HTTP 契约不变；新字段只影响 Skill 存储后端。

11. 运维要点与边界

  • LOG_DIR：EDPAgent 需显式设置，否则可能无 process_*.log，轨迹与 from_logs 都会失败。
  • 端口：宿主机 8900 被占用时，实验 Adapter 使用 18900。
  • jiuwenbox 权限：Docker Desktop 下 isolated 网络沙箱建议 --privileged。
  • sandbox_id：EDP 重启会换新 id；Adapter 宜重启或清缓存，确保与最新日志对齐。
  • frontmatter：Skill 正文/脚本可热更；name/description 等注册元数据仍可能需进程重启（既有行为）。

13. 总结

生态：EvoAgent 负责 Skill 优化编排；Adapter 作为 Sidecar 统一提供轨迹、代理与热更； 业务 Agent（如 EDPAgent）执行对话；jiuwenbox 提供沙箱隔离与文件真源。优化闭环经 Adapter HTTP 契约贯通。

改造：从虚拟机共享挂载迁到沙箱容器后，轨迹与代理路径可保持不变； Skill 热更须切换为 skill_backend: jiuwenbox，写入与 EDP 相同的 sandbox_id。 本方案仅改 EvoAgentAdapter 源码；EDP / jiuwenbox 沿用既有 API，部署侧仍需配置 SANDBOX_URL、LOG_DIR 等。

实证：三容器联调 8 / 8 PASS；前后两轮对话分别读出不同 HOTUPDATE_MARKER，证明热更已进入运行时可见的沙箱文件系统。
