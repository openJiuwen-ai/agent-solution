# 服务启动失败排错指南

## 问题描述

EDPAgent-Java服务（Spring Boot应用）启动过程中失败，进程退出或容器立即退出，无法正常提供服务。EDPAgent基于Spring Boot 3 + Java 21构建，启动时会依次完成配置加载、Redis连接初始化、Governance配置加载、Skill加载、工具注册、Web服务启动等步骤，任何一步失败都可能导致启动失败。

**重要提示**：API Key无效不会导致启动失败，但会导致所有用户请求失败。

## 常见症状

1. **容器立即退出**：`docker run`后容器状态为Exited，`docker ps`看不到运行中的容器
2. **JVM启动失败**：出现`Error: Could not create the Java Virtual Machine`、`UnsupportedClassVersionError`等错误
3. **端口被占用**：日志显示`Port 8190 is already in use`、`Address already in use`
4. **配置解析错误**：YAML语法错误、配置文件找不到、环境变量格式错误
5. **Redis连接失败导致启动失败**：启动时Redis健康检查不通过，Spring容器初始化失败
6. **启动后无响应**：进程在但端口未监听，健康检查失败
7. **ClassNotFoundException/NoSuchMethodError**：依赖缺失或版本冲突
8. **OOM启动失败**：堆内存不足，JVM无法启动

## 可能原因

| 分类 | 可能原因 | 关键特征 |
|------|----------|----------|
| 端口冲突 | 端口被占用 | `Address already in use`、`Port 8190 was already in use` |
| Redis依赖 | Redis不可达 | `Redis health check failed`、`Connection refused` |
| 配置文件 | YAML配置语法错误 | `ScannerException`、`while parsing a block mapping`、缩进问题 |
| 配置文件 | 配置文件路径错误 | `FileNotFoundException`、`edp-config.yaml not found` |
| Java版本 | JDK版本错误 | `UnsupportedClassVersionError`、需要Java 21+ |
| 内存配置 | 堆内存不足 | `OutOfMemoryError: Java heap space`、`Could not reserve enough space` |
| 依赖问题 | Jar包损坏/依赖冲突 | `ClassNotFoundException`、`NoSuchMethodError` |
| 场景配置 | 场景目录不存在 | `Scenario home not found`、`governance/planrule.yaml missing` |
| 权限问题 | 文件权限不足 | `Permission denied`、无法读取配置文件 |
| Docker问题 | 镜像构建失败/架构不匹配 | `exec format error`、`no such file or directory` |
| 环境变量 | 环境变量格式错误 | `Duration parse error`、`NumberFormatException` |

**注意**：以下问题**不会**导致启动失败，但会导致运行时错误：
- `EDP_AGENT_MODEL_API_KEY`无效/未设置
- Versatile服务不可达（未调用call_versatile时不影响）
- MCP服务不可达（未调用call_mcp时不影响）

## 排查步骤

### 步骤1：使用docker logs查看完整启动日志

这是排查启动失败的第一步：

```bash
# 查看容器日志（即使容器已退出）
docker logs edp-agent

# 查看最后200行日志
docker logs edp-agent --tail 200

# 实时查看日志
docker logs edp-agent -f
```

本地启动（java -jar或mvn spring-boot:run）：
- 日志直接输出到控制台
- 如果配置了文件日志，查看`engine/logs/`目录

**关键日志节点（正常启动顺序）**：
```
  .   ____          _            __ _ _
 /\\ / ___'_ __ _ _(_)_ __  __ _ \ \ \ \
( ( )\___ | '_ | '_| | '_ \/ _` | \ \ \ \
 \/  ___)| |_)| | | | | || (_| |  ) ) ) )
  '  |____| .__|_| |_|_| |_\__, | / / / /
 =========|_|==============|___/=/_/_/_/
 :: Spring Boot ::                (v3.x.x)
```

找到第一个`ERROR`或`Exception`，那里就是启动失败的根本原因。

### 步骤2：检查端口占用

如果日志显示`Address already in use`或`Port 8190 was already in use`：

**Windows：**
```powershell
# 查看8190端口占用
netstat -ano | findstr :8190
# 查看对应进程
tasklist | findstr <PID>
# 结束占用进程（谨慎）
taskkill /F /PID <PID>
```

**Linux/macOS：**
```bash
# 查看8190端口占用
lsof -i :8190
netstat -tlnp | grep 8190
# 结束占用进程
kill -9 <PID>
```

**Docker端口映射冲突：**
```bash
# 查看端口映射
docker ps
# 如果主机8190被占用，映射到其他端口
docker run -p 8191:8190 ...  # 主机8191→容器8190
```

或修改EDPAgent监听端口：
```bash
SERVER_PORT=8080
```

### 步骤3：验证Java版本

EDPAgent要求**JDK 21或更高版本**。

检查Java版本：
```bash
java -version
# 预期输出包含 openjdk version "21.x.x" 或更高
```

如果版本不对：
- 本地开发：安装JDK 21，设置`JAVA_HOME`
- Docker：使用正确的基础镜像（查看deploy/Dockerfile）

典型错误：
```
Exception in thread "main" java.lang.UnsupportedClassVersionError: 
  com/huawei/ascend/edp/EdpaApplication has been compiled by a more recent version of the Java Runtime
```
→ 这表示运行时JRE版本低于编译时JDK版本（需要21+）。

### 步骤4：检查Redis连接是否正常

启动时Redis健康检查失败会导致启动失败：
```
[EDPA-DIAG] REDIS_HEALTH CHECK FAILED: Connection refused
Caused by: java.lang.IllegalStateException: Redis health check failed
```

**快速排查：**
1. 确认Redis已启动（参考Redis连接问题.md）
2. 本地开发：确保Redis运行在localhost:6379
3. Docker：确认使用`host.docker.internal`（Win/Mac）或宿主机IP
4. 先手动测试Redis连接：`redis-cli ping`返回PONG

**最小启动依赖**：EDPAgent启动阶段**必须**能连接到Redis（健康检查）。

### 步骤5：检查YAML配置语法

YAML对缩进和语法要求严格，常见错误：

1. **缩进不一致**（必须用空格，不能用Tab）
2. **冒号后缺少空格**：`key:value` ❌ → `key: value` ✅
3. **特殊字符未转义**：`password: p@ss:w0rd` 可能需要加引号
4. **列表项格式错误**
5. **多行字符串缩进错误**

验证YAML语法：
- 使用在线YAML验证工具：https://yaml-online-parser.appspot.com/
- 使用IDE的YAML插件检查
- 使用Python验证：`python -c "import yaml; yaml.safe_load(open('your-file.yaml'))"`

关键配置文件：
- `deploy/config/edp-config.yaml`
- `engine/src/main/resources/application.yml`
- `scenarios/*/governance/planrule.yaml`
- `scenarios/*/governance/actrule.yaml`
- `scenarios/*/governance/scriptconfig.yaml`

典型错误日志：
```
Caused by: org.yaml.snakeyaml.scanner.ScannerException: while scanning a simple key
  in 'reader', line 5, column 1:
    key:value
    ^
could not find expected ':'
```

### 步骤6：检查配置文件路径

验证`EDPA_AGENT_CONFIG_PATH`和`EDP_AGENT_SCENARIO_HOME`路径正确：

| 环境变量 | Docker默认值 | 说明 |
|----------|-------------|------|
| `EDPA_AGENT_CONFIG_PATH` | `/app/config/edp-config.yaml` | 主配置文件路径 |
| `EDP_AGENT_SCENARIO_HOME` | `/app/scenarios/wealth-demo` | 场景目录路径，**必填** |

检查文件是否存在：
```bash
# Docker容器内检查
docker exec edp-agent ls -la /app/config/
docker exec edp-agent ls -la /app/scenarios/
docker exec edp-agent ls -la /app/scenarios/wealth-demo/governance/

# 预期能看到：
# edp-config.yaml
# planrule.yaml、actrule.yaml、scriptconfig.yaml
```

本地开发检查：
```bash
ls -la src/main/resources/edp-config.yaml
ls -la ../scenarios/wealth-demo/governance/
```

如果文件不存在：
1. 检查挂载路径是否正确（Docker -v参数）
2. 确认场景目录名称拼写正确
3. 确认配置文件已复制到镜像中（检查.dockerignore和Dockerfile）

### 步骤7：检查内存配置

如果JVM无法分配足够内存，会启动失败：

错误特征：
```
Error occurred during initialization of VM
Could not reserve enough space for object heap
OutOfMemoryError: Java heap space
```

排查和解决：
1. **检查Docker内存限制**：Docker Desktop默认内存可能不足，建议设置≥4GB
2. **调整JVM堆内存参数**：
   ```bash
   # Docker运行时设置
   docker run -e JAVA_OPTS="-Xms512m -Xmx1024m" ...
   
   # 或直接在JAVA_TOOL_OPTIONS中设置
   JAVA_TOOL_OPTIONS="-Xms256m -Xmx512m"
   ```
3. **检查系统可用内存**：`free -h`（Linux）或任务管理器（Windows）
4. **降低内存使用**：开发环境可以减小堆内存

### 步骤8：查看容器退出码

```bash
# 查看容器退出码
docker inspect edp-agent --format='{{.State.ExitCode}}'
```

常见退出码：
| 退出码 | 含义 | 可能原因 |
|--------|------|----------|
| 0 | 正常退出 | 应用主动退出（可能是启动后立即关闭） |
| 1 | 通用错误 | 应用异常，查看日志 |
| 137 | OOM被Kill | 内存不足，被Docker/K8s OOM Killer杀死 |
| 139 | Segmentation Fault | JVM崩溃、 native代码问题 |
| 143 | SIGTERM | 被正常终止（如docker stop） |

退出码137→**内存不足**，增大Docker内存或减小JVM堆。

### 步骤9：检查Docker镜像和架构问题

如果是`exec format error`：
```
exec /usr/local/bin/java: exec format error
```
这表示镜像架构与宿主机不匹配（如在ARM Mac上运行x86镜像，或反之）。

解决：
1. 确认Dockerfile基础镜像支持当前架构
2. 构建镜像时指定平台：`docker build --platform linux/amd64 ...`
3. 或在对应架构机器上重新构建

检查镜像：
```bash
docker inspect edp-agent:latest --format='{{.Architecture}}'
uname -m  # 查看宿主机架构
```

### 步骤10：检查环境变量格式

环境变量格式错误会导致配置解析失败：

1. **Duration格式错误**（如VERSATILE_TIMEOUT）：
   - 正确：`30s`、`60s`、`2m`
   - 错误：`30`、`60seconds`（需要单位）
   
2. **数字格式错误**：
   - PORT必须是数字：`8190` ✅，`8190abc` ❌
   - TIMEOUT毫秒必须是数字

3. **特殊字符未转义**：
   - 密码含特殊字符时用引号包裹：`-e "EDPA_REDIS_PASSWORD=p@ss:w0rd"`

4. **空值处理**：
   - 不需要密码时不要设置或设置为空字符串：`EDPA_REDIS_PASSWORD=`
   - 不要设置为空格：`EDPA_REDIS_PASSWORD=" "` ❌

查看实际环境变量：
```bash
docker exec edp-agent env | sort
```

### 步骤11：检查文件权限

Linux/Docker环境下可能出现权限问题：
```
Permission denied
java.io.FileNotFoundException: /app/config/edp-config.yaml (Permission denied)
```

解决：
```bash
# 检查文件权限
ls -la deploy/config/

# 构建镜像时确保权限正确（参考Dockerfile）
# 或运行时指定用户（不推荐以root运行，但可临时排查）
docker run --user root ...
```

### 步骤12：检查依赖和Jar包完整性

如果出现`ClassNotFoundException`、`NoSuchMethodError`：
1. 重新构建项目：`mvn clean package -DskipTests`
2. 检查Maven依赖是否完整：`mvn dependency:tree`
3. 删除本地Maven仓库中损坏的包，重新下载
4. 确认target目录下jar包完整：
   ```bash
   ls -lh engine/target/edp-agent-engine-*.jar
   # 文件大小应该在几十MB
   ```

## 解决方案

### 方案1：快速启动最小可用环境

**本地开发快速启动（确保Redis已启动）：**
```bash
# 1. 启动Redis（Docker方式最简单）
docker run -d --name redis -p 6379:6379 redis:7-alpine

# 2. 进入engine目录
cd engine

# 3. 设置最小配置并启动
EDP_AGENT_MODEL_API_KEY=sk-your-key \
EDPA_REDIS_HOST=localhost \
EDP_AGENT_SCENARIO_HOME=../scenarios/wealth-demo \
mvn spring-boot:run
```

**Docker快速启动：**
```bash
# 1. 创建网络
docker network create edp-net

# 2. 启动Redis
docker run -d --name redis --network edp-net -p 6379:6379 redis:7-alpine

# 3. 启动EDPAgent
docker run -d --name edp-agent \
  --network edp-net \
  -p 8190:8190 \
  -e EDP_AGENT_MODEL_API_KEY=sk-your-key \
  -e EDPA_REDIS_HOST=redis \
  -e EDP_AGENT_SCENARIO_HOME=/app/scenarios/wealth-demo \
  edp-agent:latest

# 4. 查看日志
docker logs edp-agent -f
```

### 方案2：更换端口

如果8190端口被占用：

**方式1：映射到其他主机端口**
```bash
docker run -p 8191:8190 ...  # 主机8191→容器8190
```

**方式2：修改容器内监听端口**
```bash
docker run -e SERVER_PORT=8080 -p 8080:8080 ...
```

**本地启动：**
```bash
SERVER_PORT=8080 mvn spring-boot:run
```

### 方案3：修复Redis连接问题

参考[Redis连接问题](./Redis连接问题.md)进行详细排查。

**快速修复清单：**
1. Redis已启动：`redis-cli ping` → PONG
2. Docker连宿主机：`EDPA_REDIS_HOST=host.docker.internal`
3. 密码正确：无密码则`EDPA_REDIS_PASSWORD=`（空）
4. 端口正确：默认6379

### 方案4：修复YAML配置错误

YAML排错技巧：
1. **复制到在线验证工具**检查语法
2. **统一缩进**：使用2空格缩进，不要混用Tab
3. **对照示例配置**：参考`deploy/config/edp-config.yaml`
4. **逐步回滚**：最近修改的配置最可能是问题原因
5. **使用IDE**：IntelliJ IDEA/Eclipse有YAML语法检查

常见正确格式：
```yaml
edpa:
  agent:
    model:
      provider: OpenAI
      name: deepseek-chat
      base-url: https://api.deepseek.com/v1
    redis:
      host: localhost
      port: 6379
```

### 方案5：使用JDK 21

**本地开发：**
1. 下载JDK 21：https://adoptium.net/
2. 设置JAVA_HOME：
   ```bash
   # Windows
   set JAVA_HOME=C:\Program Files\Eclipse Adoptium\jdk-21.x.x-hotspot
   # Linux/macOS
   export JAVA_HOME=/Library/Java/JavaVirtualMachines/temurin-21.jdk/Contents/Home
   ```
3. 验证：`java -version`

**Docker：**
使用deploy/Dockerfile中的基础镜像（已配置JDK 21），不要修改基础镜像版本。

### 方案6：调整内存配置

**Docker内存调整：**
1. Docker Desktop → Settings → Resources → Memory → 调整为≥4GB
2. 调整JVM参数：
   ```bash
   docker run -e JAVA_OPTS="-Xms256m -Xmx512m" ...
   ```

**本地启动：**
```bash
# Maven启动
MAVEN_OPTS="-Xms256m -Xmx512m" mvn spring-boot:run

# java -jar启动
java -Xms256m -Xmx512m -jar edp-agent-engine.jar
```

### 方案7：修正配置路径和场景目录

**Docker挂载自定义配置：**
```bash
docker run -d --name edp-agent \
  -v /your/local/config:/app/config \
  -v /your/local/scenarios:/app/scenarios \
  -e EDPA_AGENT_CONFIG_PATH=/app/config/edp-config.yaml \
  -e EDP_AGENT_SCENARIO_HOME=/app/scenarios/your-scenario \
  -p 8190:8190 \
  edp-agent:latest
```

**检查路径注意事项：**
1. 容器内路径是绝对路径（/app/...）
2. 挂载时主机路径必须存在
3. 场景目录下必须有governance子目录，包含3个yaml文件

验证挂载：
```bash
docker exec edp-agent ls -la /app/config/
docker exec edp-agent ls -la /app/scenarios/
```

### 方案8：验证API Key（启动后验证）

API Key不会导致启动失败，但启动后需要验证是否有效：
```bash
# 启动后测试接口
curl -X POST http://localhost:8190/chat \
  -H "Content-Type: application/json" \
  -d '{"message":"你好"}'
```

如果返回401错误，参考[模型API错误](./模型API错误.md)修复API Key。

## 启动成功验证

服务启动成功后，通过以下方式验证：

1. **日志看到**：`Started EdpaApplication in XX.XXX seconds`
2. **端口在监听**：`netstat -tlnp | grep 8190`（Linux）或`netstat -ano | findstr :8190`（Windows）
3. **健康检查通过**：
   ```bash
   curl http://localhost:8190/.well-known/agent-card.json
   # 返回JSON包含agent信息
   ```
4. **Docker状态healthy**：
   ```bash
   docker inspect edp-agent --format='{{.State.Health.Status}}'
   # 等1-2分钟后应为healthy
   ```

健康检查端点说明：
- URL：`http://localhost:8190/.well-known/agent-card.json`
- Docker HEALTHCHECK配置：每10秒检查，5次失败标记unhealthy，启动等待60秒

## 相关配置/日志关键词

### 关键启动日志（按顺序）
1. Spring Boot Banner → Spring启动
2. `[EDPA-DIAG] REDIS_CONFIG` → Redis配置
3. `[EDPA-DIAG] REDIS_CHECKPOINTER registered` → Redis Checkpointer注册
4. `[EDPA-DIAG] REDIS_HEALTH CHECK passed` → Redis健康检查通过（失败则启动失败）
5. `Loading governance config` → 配置加载
6. `Loaded planrule/actrule/skill` → Governance和Skill加载
7. `Registered business tool` → 工具注册
8. `Tomcat started on port(s): 8190` → Web容器启动
9. `Started EdpaApplication` → **启动成功标志**

### 常见错误关键词
- `Address already in use` → 端口占用
- `Redis health check failed` → Redis连接失败
- `UnsupportedClassVersionError` → JDK版本错误
- `ScannerException`/`YAML` → YAML语法错误
- `FileNotFoundException` → 文件/路径错误
- `OutOfMemoryError`/`Could not reserve` → 内存不足
- `ClassNotFoundException` → 依赖问题
- `Permission denied` → 权限问题
- `exec format error` → 架构不匹配
- `Connection refused` → 连接被拒（Redis等）

### 配置项
- `SERVER_PORT` - 服务端口（默认8190）
- `EDPA_REDIS_*` - Redis配置（启动必须）
- `EDPA_AGENT_CONFIG_PATH` - 主配置路径
- `EDP_AGENT_SCENARIO_HOME` - 场景目录（必填）
- `EDP_AGENT_LOG_LEVEL` - 日志级别（DEBUG可看更多信息）

### 代码位置
- 应用入口：`engine/src/main/java/com/huawei/ascend/edp/EdpaApplication.java`
- Redis配置：`engine/src/main/java/com/huawei/ascend/edp/config/RedisConfig.java`
- Redis健康检查：`engine/src/main/java/com/huawei/ascend/edp/todo/RedisTodoStore.java:52-76`
- Dockerfile：`deploy/Dockerfile`
- 健康检查：`docs/operations/健康检查与日志.md`

## 预防措施

1. **启动前检查脚本**：编写启动前检查脚本，验证Redis、端口、配置
   ```bash
   # 示例预检脚本
   redis-cli ping || { echo "Redis not running"; exit 1; }
   ! netstat -tlnp | grep :8190 || { echo "Port 8190 in use"; exit 1; }
   ```

2. **使用Docker Compose**：编排Redis和EDPAgent，确保依赖顺序
   ```yaml
   services:
     redis:
       image: redis:7-alpine
     edp-agent:
       image: edp-agent:latest
       depends_on:
         redis:
           condition: service_started
       environment:
         - EDPA_REDIS_HOST=redis
   ```

3. **配置健康检查**：利用Docker HEALTHCHECK或K8s liveness/readiness探针

4. **配置中心化管理**：使用ConfigMap/Nacos/Apollo管理配置，避免手动修改YAML出错

5. **CI/CD验证**：
   - PR构建时自动验证配置文件语法
   - 自动化启动测试
   - 镜像扫描和架构验证

6. **资源配额**：
   - 生产环境设置合理的CPU/内存requests/limits
   - JVM堆内存设置为容器内存的50-70%
   - 预留足够内存给Metaspace和本地内存

7. **日志收集**：
   - 启动日志持久化存储
   - 监控启动失败告警（容器重启、CrashLoopBackOff）

8. **版本管理**：
   - 使用明确版本标签，不使用latest
   - 升级前在测试环境验证启动
   - 保留回滚镜像版本
