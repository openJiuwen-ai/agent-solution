# 话术管理特性

## 特性概述

话术管理特性实现了Agent与人交互过程中的统一话术配置、变量替换和场景级覆盖机制。通过分层话术体系（通用话术→场景话术→Skill话术），结合SafeFormat变量渲染和合规把关，确保Agent输出的话术一致、可控、合规。核心组件包括ScriptsRail（话术出口Rail）和ScriptResolver（话术解析工具类），支撑生命周期事件话术、业务结果话术、用户确认话术等多种场景。

核心能力包括：
- **general_scripts通用话术**：覆盖工具调用、任务状态、中断、取消等全生命周期
- **SKILL.yaml话术模板**：Skill级别的场景话术定义
- **话术变量替换**：{tool_name}、{title}等占位符自动替换，变量缺失时安全降级为空串
- **场景级话术覆盖**：继承式覆盖，场景配置合并到通用配置
- **ask_user_confirm确认话术**：用户中断确认场景的模板化话术
- **合规把关**：配置外话术自动替换为out_of_scope，防止不合规输出

## 配置方式

### 1. 通用话术配置（scriptconfig.yaml）

在`scriptconfig.yaml`的`general_scripts`中配置生命周期通用话术：

```yaml
scriptconfig:
  # ── UC-C02: interrupt 追问内容来源开关 ──
  # script（默认）：追问内容必须来自脚本，LLM 生成内容被拒绝
  # llm：追问内容可由 LLM 生成
  interrupt_source: "script"

  # 通用话术，用于业务流程状态（工具调用、中断、取消等）
  general_scripts:
    tool_start: "正在调用：{tool_name}"
    tool_end: "{tool_name} 执行完成"
    todo_start: "开始执行：{title}"
    todo_end: "{title} 已完成"
    todolist_start: "已生成任务规划"
    todolist_end: "任务规划完成"
    interrupt_start: "需要您确认以下信息"
    request_start: "您的请求已收到。"
    planning_start: "我们正在为您进行规划。"
    task_cancelled: "好的，已为您取消当前操作。如需其他帮助，请随时告诉我。"
    cancel_confirm: "确认要取消当前操作吗？"
    out_of_scope: "当前请求暂不在可处理范围内。"
```

**话术变量说明**：
| 变量名 | 适用话术 | 说明 |
|--------|---------|------|
| {tool_name} | tool_start、tool_end | 当前调用的工具名称 |
| {title} | todo_start、todo_end | 当前任务的标题（content） |

> **注意**：`interrupt_source` 控制 ask_user 追问内容来源：
> - `script`（默认）：脚本话术优先，LLM 生成内容仅作为未命中脚本时的兜底
> - `llm`：追问内容可由 LLM 生成，适用于灵活对话场景

### 5. 意图识别话术（query_intent_tool_text）

配置LLM在识别用户意图时输出的过渡话术，引导用户理解当前处理进度。话术key命名规则：`{intent}_text`，intent为`query_intent`工具返回的意图名称（小写）。

```yaml
scriptconfig:
  # 意图识别过渡话术（query_intent工具返回intent后自动渲染）
  query_intent_tool_text:
    recommend_finance: "正在为您推荐合适的理财产品，请稍候..."
    product_select: "正在为您处理产品选择，请稍候..."
    fund_planning: "正在为您进行资金筹划，请稍候..."
    interact_finance_rec: "正在为您重新筛选产品，请稍候..."
    out_of_scope: "您请求的业务暂不在处理范围内。"
```

**触发机制**：LLM调用query_intent工具后，根据返回的intent字段，自动查找`{intent}_text`话术并渲染为过渡帧。

### 6. 思考过程话术（think_chunk_scripts）

配置LLM思考过程的固定帧话术，用于在LLM进行长思考时输出过渡提示，避免用户长时间等待。话术key对应思考的不同阶段：

```yaml
scriptconfig:
  # 思考过程固定帧话术（LLM think阶段输出过渡提示）
  think_chunk_scripts:
    progress_hint: "我们正在为您处理..."        # 进度提示
    skip_tool: "让我检查一下相关文档..."        # 跳过工具调用时的提示
    plan_first_block: "正在生成任务规划..."     # 生成todo时的提示
    checkpoint_release: "请稍候，正在执行..."   # 检查点释放提示
```

**触发机制**：在LLM think阶段，根据思考进度自动渲染对应话术，通过artifactUpdate以增量文本流方式推送。

## 关键实现机制

### 1. 话术双层架构（A面+B面）

话术由两个Rail协作处理，通过ctx.getExtra()中的`_edp_response_template`键传递：

**A面：EdpaEventRail（priority=80）**
- 负责生命周期事件话术：tool_start/tool_end、todo_start/todo_end、todolist_start/todolist_end、interrupt_start
- 负责首轮开场：beforeInvoke发射request_start，afterModelCall发射planning_start
- 在事件发射时调用ScriptResolver渲染话术
- onToolException中解析ask_user的response_template参数（F3-fix承载点）
- afterInvoke中发射出口话术（interrupt_id="response_template"）

**B面：ScriptsRail（priority=50）**
- 负责cancel_task话术解析（beforeToolCall）
- 负责afterInvoke合规把关兜底（出口话术已被EdpaEventRail发射时的兜底）
- init时注入cancel规则和业务规则prompt

话术载体：`ctx.getExtra()`内的`_edp_response_template`键，同一HTTP请求内写+读+发射，不跨轮持久化。

代码位置：
- A面：`EdpaEventRail.java`（beforeInvoke发射request_start: 210-218，afterModelCall发射planning_start: 308-319，onToolException解析ask_user: 607-609，afterInvoke出口合规把关: 686-690）
- B面：`ScriptsRail.java:41-329`（cancel_task解析: 118-135，complianceGate兜底: 186-195，init注入Prompt: 201-211）

### 2. SafeFormat变量替换

ScriptResolver提供两种变量替换策略，适用于不同场景：

**resolve（UC-A06 保留占位符）**：
- 用于通用话术（tool_start/tool_end、todo_start/todo_end等）
- 使用正则`\{(\w+)\}`匹配{var}占位符
- 变量存在时替换为实际值
- **变量缺失时保留`{var}`占位符原样**（不替换为空串，便于排查缺失变量）
- template为null/空时返回空串

```java
// resolve 核心逻辑（UC-A06）
public static String resolve(String template, Map<String, String> vars) {
    if (template == null || template.isEmpty()) return "";
    if (vars == null || vars.isEmpty()) return template; // 保留占位符原样
    Matcher m = VAR.matcher(template);
    StringBuilder sb = new StringBuilder();
    while (m.find()) {
        String value = vars.get(m.group(1));
        if (value != null) {
            m.appendReplacement(sb, Matcher.quoteReplacement(value));
        } else {
            // UC-A06: 缺失变量保留 {var} 占位符原样
            m.appendReplacement(sb, Matcher.quoteReplacement(m.group(0)));
        }
    }
    m.appendTail(sb);
    return sb.toString();
}
```

**safeFormat（UC-A07 替换为空串）**：
- 用于ask_user话术模板渲染
- 变量缺失时替换为空串（UC-A07验收标准第5条）
- 适用于容错要求更高的业务确认场景

```java
// safeFormat 核心逻辑（UC-A07）
public static String safeFormat(String template, Map<String, String> vars) {
    if (template == null || template.isEmpty()) return "";
    if (vars == null || vars.isEmpty()) return VAR.matcher(template).replaceAll("");
    Matcher m = VAR.matcher(template);
    StringBuilder sb = new StringBuilder();
    while (m.find()) {
        String value = vars.getOrDefault(m.group(1), "");
        m.appendReplacement(sb, Matcher.quoteReplacement(value));
    }
    m.appendTail(sb);
    return sb.toString();
}
```

代码位置：`ScriptResolver.java:55-101`（resolve + safeFormat）

### 3. ask_user话术解析（F3-fix）

ScriptResolver.resolveAskUser()解析ask_user的话术参数：

**参数说明**：
- response_template_status：当前状态（confirm/missing_amount/missing_product等）
- response_template_keys：status→话术key的映射
- response_template_vars：变量键值对

**容错机制（coerceJsonMap）**：
1. 已是Map→直接转字符串键值
2. 字符串→去首尾引号→中文引号/冒号归一为ASCII→Jackson解析
3. JSON解析失败→正则逐对抓取key:value
4. 全部失败→返回空Map，不抛异常，不中断主流程

解析成功后写入：
- `_edp_response_template`：渲染后的话术文本
- `_edp_last_script_key`：命中的话术key（合规把关用）
- `_edp_selected_product`：confirm状态下保存选中的产品变量

代码位置：`ScriptResolver.java:552-579`（resolveAskUser方法）

### 4. 业务结果话术匹配

业务结果话术由LLM通过ask_user工具的`response_template_keys`参数动态指定，不再在代码中硬编码映射逻辑。

**机制说明**：
1. LLM根据工具返回结果（如productBuyResponse.buyStatus、productList等）判断业务状态
2. LLM在调用ask_user时通过`response_template_status`+`response_template_keys`指定要使用的话术键
3. ScriptResolver.resolveAskUser()解析参数，从配置中查找对应话术模板
4. 配合response_template_vars传入变量（如productName、amount）

**典型场景示例**：

```
# 购买成功（LLM识别productBuyResponse.buyStatus=1）
ask_user({
  response_template_status: "success",
  response_template_keys: {"success": "fund_planning_success"},
  ...
})
# → 渲染 fund_planning_success 话术

# 购买失败（LLM识别buyStatus!=1）
ask_user({
  response_template_status: "failed",
  response_template_keys: {"failed": "fund_planning_buy_failed"},
  ...
})
# → 渲染 fund_planning_buy_failed 话术
```

**优势**：业务键由SKILL.yaml动态定义，新增Skill只需新增SKILL.yaml的scripts字段，无需修改代码。

代码位置：话术键解析见 `ScriptResolver.java:552-579`（resolveAskUser），LLM通过Prompt引导见 `ScriptResolver.java:487-502`（businessRulesPrompt）

### 5. 合规把关（Compliance Gate）

ScriptsRail.complianceGate()实现话术合规检查：

- 检查`_edp_last_script_key`是否在scripts配置中
- 如果不在配置中（配置外话术），替换为out_of_scope
- 防止LLM编造话术或输出不合规内容

```java
private void complianceGate(AgentCallbackContext ctx) {
    Object k = ctx.getExtra().get(ScriptConstants.KEY_LAST_SCRIPT);
    String resolvedKey = k == null ? null : String.valueOf(k);
    if (scripts != null && resolvedKey != null && !scripts.has(resolvedKey)) {
        ctx.getExtra().put(ScriptConstants.KEY_RESPONSE_TEMPLATE,
                scripts.getScriptOrDefault("SCRIPT_OUT_OF_SCOPE", ""));
    }
}
```

EdpaEventRail.afterInvoke()也有相同的合规把关逻辑，双重保险。

代码位置：
- ScriptsRail：`ScriptsRail.java:186-195`
- EdpaEventRail：`EdpaEventRail.java:686-690`

### 6. Skill话术动态收集（SkillScriptsCollector）

`SkillScriptsCollector`在引擎启动时扫描skills目录，从各Skill的`SKILL.yaml`中收集`scripts`字段，合并到全局话术配置。

**收集流程**：
1. 扫描`scenarios/{scenario}/skills/`目录下所有子目录
2. 查找`v1/SKILL.yaml`或`SKILL.yaml`（优先v1版本）
3. 解析YAML，提取`scripts`字段（Map<String, String>）
4. 合并到全局话术配置（Skill话术覆盖通用话术同名key）

```
scenarios/wealth-demo/skills/
  ├── product_recommend_skill/SKILL.yaml  → scripts.product_recommend_success 等
  ├── product_select_skill/SKILL.yaml    → scripts.product_select_confirm 等
  ├── fund_planning_skill/SKILL.yaml     → scripts.fund_planning_success 等
  └── interact_finance_rec_skill/SKILL.yaml → scripts.mcp_result_empty 等
```

## 注意事项

1. **话术key必须在配置中存在**：合规把关机制会将配置外的话术自动替换为out_of_scope，不要在代码或prompt中引用未配置的话术key。

2. **变量缺失安全降级**：
   - 通用话术（resolve）：`{var}`变量不存在时**保留占位符原样**，便于排查缺失变量（UC-A06）
   - ask_user话术（safeFormat）：`{var}`变量不存在时替换为空串，不会抛出异常（UC-A07）
   - 配置模板时请注意变量完整性，通用话术保留占位符可能导致输出不完整。

3. **ask_user话术在onToolException解析**：因为AskUserTemplateRail(p=85)在beforeToolCall时抛出ToolInterruptException，ScriptsRail(p=50)的beforeToolCall不可达，所以ask_user话术解析放在EdpaEventRail.onToolException(p=80)中，必定触发。

4. **Skill话术继承覆盖**：Skill的scripts字段由`SkillScriptsCollector`在启动时动态收集，与全局general_scripts合并，相同key以Skill配置为准，实现场景级覆盖。修改SKILL.yaml中的scripts后需重启EDPA生效。

5. **interrupt_active时不发出口request_start**：当有活跃中断（ask_user等待用户输入）时，afterInvoke不发射request_start，因为interrupt_start.content已携带话术文本，避免冗余。

6. **话术载体不跨轮**：_edp_response_template只在同一HTTP请求内有效，afterInvoke发射后清除，不跨轮持久化。

7. **中文标点容错**：coerceJsonMap会将中文引号""、冒号：归一为ASCII字符，提高LLM输出JSON的容错率。

8. **结果话术不兜底未知业务**：对于没有配置话术的业务场景（如查余额、转账），ScriptsRail不强行兜底，由工具返回的原始结果决定最终输出，避免误导用户。

9. **interrupt_source控制追问来源**：
   - `script`（默认）：ask_user 追问内容优先使用配置的话术模板，LLM 生成内容仅在脚本未命中时作为兜底（优先级：versatile提示 > 脚本话术 > LLM question）
   - `llm`：追问内容可由LLM生成，适用于需要灵活对话的场景
   - 修改后需重启EDPA生效

10. **意图识别过渡话术**：query_intent_tool_text中的key命名规则为`{intent}_text`，intent必须与query_intent工具返回的intent字段（小写）一致，否则无法匹配。

11. **思考过程话术增量推送**：think_chunk_scripts中的话术通过artifactUpdate以增量文本流方式推送，不会阻塞主流程，适用于长思考场景的过渡提示。
