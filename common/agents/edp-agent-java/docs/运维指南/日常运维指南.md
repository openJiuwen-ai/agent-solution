# 日常运维操作

本文档介绍 EDPAgent 的日常运维操作，包括启停、日志查看、Redis 维护、配置更新等。

## 服务启停

### Docker 方式

#### 启动服务

```bash
# 如果使用 docker-compose
cd /opt/edp-agent
docker-compose up -d

# 或直接 docker run
docker run -d \
  --name edp-agent \
  --restart=always \
  -p 8190:8190 \
  --env-file /opt/edp-agent/.env \
  edp-agent-java:0.2.0
```

#### 停止服务

```bash
# 优雅停止（发送 SIGTERM，等待请求处理完成）
docker stop edp-agent

# docker-compose
docker-compose stop
```

#### 重启服务

```bash
# 重启容器
docker restart edp-agent

# 或 stop + start
docker stop edp-agent
docker start edp-agent

# docker-compose
docker-compose restart
```

#### 查看服务状态

```bash
# 查看容器状态
docker ps -a | grep edp-agent

# 查看健康状态
docker inspect edp-agent --format '{{.State.Status}} / {{.State.Health.Status}}'

# 查看资源占用
docker stats edp-agent --no-stream
```

### java -jar 方式（非容器部署）

```bash
# 启动（后台运行）
nohup java -jar edp-agent-engine-*.jar \
  --spring.config.location=file:./application.yml \
  > logs/edp-agent.out 2>&1 &

# 停止（查找进程并kill）
ps aux | grep edp-agent-engine
kill <pid>

# 或使用更优雅的方式（如果配置了shutdown hook）
kill -TERM <pid>
```

## 日志查看

### Docker 日志

```bash
# 查看实时日志
docker logs -f edp-agent

# 查看最近100行
docker logs --tail 100 edp-agent

# 查看最近30分钟的日志
docker logs --since 30m edp-agent

# 查看特定时间段日志
docker logs --since "2026-07-09T10:00:00" --until "2026-07-09T11:00:00" edp-agent

# 过滤错误日志
docker logs edp-agent 2>&1 | grep -i error

# 保存日志到文件
docker logs edp-agent > edp-agent-log-$(date +%Y%m%d).log
```

### 日志级别说明

| 级别 | 标识 | 说明 |
|------|------|------|
| ERROR | `ERROR` | 错误，需要关注和处理 |
| WARN | `WARN` | 警告，潜在问题但不影响运行 |
| INFO | `INFO` | 常规运行信息 |
| DEBUG | `DEBUG` | 调试详情，排查问题时使用 |

### 关键日志关键词

排查问题时搜索以下关键词：

| 关键词 | 说明 |
|--------|------|
| `Started EdpaApplication` | 启动成功 |
| `ERROR` | 错误发生 |
| `Exception` | 异常堆栈 |
| `Redis connection` | Redis 连接问题 |
| `Model API` | 大模型调用问题 |
| `Versatile` | Versatile 调用问题 |
| `MCP` | MCP 调用问题 |
| `Rail execution` | Rail 执行日志 |
| `Tool call` | 工具调用记录 |

### 日志轮转

Docker 部署时建议配置日志轮转（已在 Docker部署指南.md 推荐配置中说明）：

```json
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "100m",
    "max-file": "7"
  }
}
```

这会保留7个日志文件，每个最大100MB，总计约700MB。

## 动态调整日志级别

当前版本不支持运行时动态调整日志级别，需要通过环境变量调整后重启：

```bash
# 修改 .env 文件中的日志级别
EDP_AGENT_LOG_LEVEL=DEBUG

# 重启服务
docker restart edp-agent
```

## Redis 维护

### 连接检查

```bash
# 使用 redis-cli 连接
redis-cli -h <redis-host> -p <redis-port> -a <password>

# 测试连接
PING
# 返回 PONG 表示正常
```

### 查看 EDPAgent 相关 Key

```bash
# 查看所有 EDPAgent key（前缀 edpa:）
KEYS edpa:*

# 查看 Todo 数量
KEYS edpa:todo:* | wc -l

# 查看 Checkpoint 数量
KEYS edpa:checkpoint:* | wc -l
```

### 清理过期数据

EDPAgent 依赖 Redis TTL 自动过期清理数据，一般不需要手动清理。如需手动清理：

```bash
# 清理所有 EDPAgent 数据（会导致所有进行中的会话/任务丢失！）
# ⚠️ 谨慎操作，仅在开发环境或确定需要重置时使用
KEYS edpa:* | DEL

# 或只清理特定会话的数据
DEL edpa:todo:<conversationId>
DEL edpa:checkpoint:<conversationId>
```

### Redis 内存监控

```bash
# 查看 Redis 内存使用
redis-cli INFO memory | grep used_memory_human

# 查看 Key 统计
redis-cli INFO keyspace
```

## 配置更新

### 修改环境变量

配置修改后需要重启服务生效：

```bash
# 1. 编辑 .env 文件
vim /opt/edp-agent/.env

# 2. 如果是 docker run 启动，需要停止并重新创建容器
docker stop edp-agent
docker rm edp-agent
docker run -d \
  --name edp-agent \
  --restart=always \
  -p 8190:8190 \
  --env-file /opt/edp-agent/.env \
  edp-agent-java:0.2.0

# 3. 如果是 docker-compose，更简单：
docker-compose up -d  # 会自动重建配置变更的容器
```

### 切换业务场景

切换场景只需修改 `EDP_AGENT_SCENARIO_HOME` 环境变量：

```bash
# 编辑 .env
EDP_AGENT_SCENARIO_HOME=/app/scenarios/hz-zhidaitong

# 如果场景是自定义挂载的，确保挂载路径正确
# -v /path/to/hz-zhidaitong:/app/scenarios/hz-zhidaitong:ro

# 重启
docker-compose up -d
```

### 加载新 Skill

新 Skill 放入场景目录的 `skills/` 子目录后，重启服务生效：

```bash
# 将 Skill 目录复制到挂载的场景目录
cp -r /path/to/new-skill /opt/edp-agent/scenarios/wealth-demo/skills/

# 重启服务加载新 Skill
docker restart edp-agent

# 查看日志确认 Skill 加载
docker logs edp-agent 2>&1 | grep -i "loaded skill"
```

> ⚠️ **注意**：当前版本不支持热加载，任何配置、Skill、Governance YAML 的修改都需要重启服务才能生效。

## 版本升级

### 升级步骤

1. **备份配置和数据**
```bash
# 备份 .env 配置
cp /opt/edp-agent/.env /opt/edp-agent/.env.backup-$(date +%Y%m%d)

# Redis 数据备份（可选，Todo/Checkpoint 数据）
redis-cli BGSAVE
```

2. **拉取新镜像**
```bash
docker pull edp-agent-java:<new-version>
```

3. **停止旧容器**
```bash
docker stop edp-agent
docker rm edp-agent
```

4. **使用新镜像启动**
```bash
docker run -d \
  --name edp-agent \
  --restart=always \
  -p 8190:8190 \
  --env-file /opt/edp-agent/.env \
  edp-agent-java:<new-version>
```

5. **验证升级**
```bash
# 检查容器状态
docker ps

# 查看启动日志
docker logs --tail 100 edp-agent

# 健康检查
curl http://localhost:8190/.well-known/agent-card.json
```

### 回滚

如果新版本有问题，回滚到旧版本：

```bash
docker stop edp-agent
docker rm edp-agent
docker run -d \
  --name edp-agent \
  --restart=always \
  -p 8190:8190 \
  --env-file /opt/edp-agent/.env \
  edp-agent-java:<old-version>
```

## 监控与巡检

### 日常巡检清单

| 检查项 | 频率 | 命令 | 正常标准 |
|--------|------|------|----------|
| 容器状态 | 每天 | `docker ps` | 状态 Up，healthy |
| 磁盘使用 | 每天 | `df -h` | 日志分区使用率 < 80% |
| Redis 内存 | 每周 | `redis-cli INFO memory` | used_memory 在预期范围内 |
| 错误日志 | 每天 | `docker logs edp-agent 2>&1 \| grep ERROR` | 无异常错误 |
| 大模型连通性 | 每周 | 发送测试消息 | 正常返回响应 |
| Versatile 连通性 | 每周 | 执行业务测试 | 工作流正常执行 |

### 健康检查

EDPAgent 提供两个健康检查端点：

```bash
# Agent Card（Docker HEALTHCHECK 使用此端点）
curl -sf http://localhost:8190/.well-known/agent-card.json

# 如果返回 JSON 且包含 agent 信息，表示服务正常
```

### 发送测试消息验证功能

```bash
curl -N -X POST http://localhost:8190/a2a \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "id": "health-check-'$(date +%s)'",
    "method": "sendStreamingMessage",
    "params": {
      "message": {
        "messageId": "health-check",
        "role": "user",
        "parts": [{"kind": "text", "text": "ping"}]
      }
    }
  }' --max-time 30
```

如果在30秒内收到 SSE 响应（包含 pong 或正常回复），表示服务功能正常。

## 数据备份

### 需要备份的内容

| 数据 | 位置 | 备份方式 |
|------|------|----------|
| 环境变量配置 | `/opt/edp-agent/.env` | 文件备份 |
| 自定义场景 | `/opt/edp-agent/scenarios/` | 目录备份（如果有自定义场景） |
| Redis 数据 | Redis 实例 | Redis BGSAVE/RDB/AOF |

### 备份脚本示例

```bash
#!/bin/bash
BACKUP_DIR=/opt/edp-agent/backups
DATE=$(date +%Y%m%d_%H%M%S)

mkdir -p $BACKUP_DIR

# 备份配置
cp /opt/edp-agent/.env $BACKUP_DIR/.env_$DATE

# 备份自定义场景（如果有）
tar -czf $BACKUP_DIR/scenarios_$DATE.tar.gz /opt/edp-agent/scenarios/

# Redis 备份（如果有密码加 -a 参数）
redis-cli -h your-redis-host BGSAVE
sleep 5
cp /var/lib/redis/dump.rdb $BACKUP_DIR/redis_dump_$DATE.rdb

# 清理30天前的备份
find $BACKUP_DIR -type f -mtime +30 -delete

echo "Backup completed: $DATE"
```

## 常见运维场景

### 场景1：服务无响应

1. 检查容器是否运行：`docker ps`
2. 检查健康状态：`docker inspect edp-agent --format '{{.State.Health.Status}}'`
3. 查看错误日志：`docker logs --tail 200 edp-agent | grep -i error`
4. 检查 Redis 连接是否正常
5. 检查大模型 API 是否可达
6. 如无明显异常，尝试重启：`docker restart edp-agent`

### 场景2：用户反映回复很慢

1. 检查大模型 API 响应时间
2. 检查网络连通性（特别是到外部API的延迟）
3. 查看日志中是否有重试、超时
4. 检查 Redis 响应是否正常（慢查询）
5. 检查容器 CPU/内存是否充足

### 场景3：Skill 不触发

1. 确认 SKILL.yaml 中 description 包含正确的触发词
2. 检查 actrule.yaml 的 allowed_tools 包含 skill_tool
3. 检查 skill_mode 不是 none
4. 重启服务后查看日志中是否有 Skill 加载信息
5. 开启 DEBUG 日志查看 Prompt 中是否包含该 Skill

## 注意事项

1. **不要直接修改 Jar 包内的配置文件**：所有配置通过环境变量覆盖。
2. **重启前确认没有正在进行的关键业务**：重启会中断所有进行中的会话（但 Checkpoint 保存的中断状态可以在Redis TTL内恢复）。
3. **定期清理 Docker 日志**：即使配置了轮转，长时间运行后也要关注磁盘空间。
4. **Redis 数据不持久化不影响核心功能**：Todo 和 Checkpoint 是临时状态，丢失只会导致进行中的会话无法续传，不影响服务启动和新请求。
5. **大模型 API Key 安全**：妥善保管 API Key，不要提交到代码仓，.env 文件权限设置为 600。
