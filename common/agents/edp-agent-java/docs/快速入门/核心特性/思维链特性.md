# 思维链特性

## 特性概述

思维链（Think Chunk）特性实现了Agent思考过程的可视化展示，支持两种展示模式：真实流式推送（real_stream）和固定话术帧模式（fixed_script）。通过精细化的帧控制和阶段话术配置，为用户提供流畅、自然的思考过程反馈，提升交互体验。思维链事件通过EdpaEventRail发射，经EdpaEventStreamAdapter适配为SSE事件流推送到前端。

核心能力包括：
- **think_chunk_scripts配置**：统一的思维链话术配置入口
- **双模式切换**：real_stream（真实LLM流式推送）和fixed_script（固定话术帧模拟）
- **fixed_script精细控制**：chars_per_frame（每帧字符数）、tokens_between_frames（帧间token数）、min_interval_ms（最小间隔）
- **三阶段话术**：planning阶段（首轮规划）、execution阶段（执行反思）、resume阶段（中断续轮）
- **query_patterns关键词匹配**：根据用户query关键词动态选择话术组
- **严格事件配对**：think_start ↔ think_chunk×N ↔ think_end，保证前端状态一致性

## 配置方式

### 1. 思维链全局配置（scriptconfig.yaml）

在`scriptconfig.yaml`的`think_chunk_scripts`中配置思维链模式和参数：

```yaml
scriptconfig:
  # 思维链话术配置，用于思维链展示（替代真实LLM thinking）
  think_chunk_scripts:
    # 模式选择
    # real_stream: 真实LLM流式shard直接推送前端
    # fixed_script: 用预定义固定话术帧替代LLM shard推送前端
    think_chunk_mode: fixed_script
    
    # 固定话术帧配置（仅think_chunk_mode=fixed_script时生效）
    think_chunk_fixed_scripts:
      enabled: true
      chars_per_frame: 4              # 每帧2字符（逐字渲染模式，0=不切分整句推送）
      tokens_between_frames: 2        # 每累积2个LLM token推送一个子帧
      min_interval_ms: 50             # 子帧间最少间隔50ms（0=不限速）
      
      # ── planning阶段话术（第1轮思考）──────────────────────────────
      # 默认话术（query_patterns全未命中时降级使用）
      default_scripts:
        - "正在分析您的需求..."
      
      # 按用户query关键词匹配的话术组（planning阶段，追加策略）
      # 框架提供通用匹配模式，场景追加业务关键词
      # 按声明顺序遍历，首个关键词命中即生效
      query_patterns: []
      
      # ── executing阶段话术（第2轮及后续思考轮次）────────────────────
      # 当Agent在执行工具后进入反思/决策思考时使用
      execution_scripts:
        - "正在分析执行结果..."
      
      # ── resuming阶段话术（Cascade续轮，query="continue"）─────────
      # 当用户在中断后回复触发续轮时使用
      resume_scripts:
        - "当前业务步骤已为您处理完毕"
```

### 2. 配置字段说明

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| think_chunk_mode | string | fixed_script | 模式选择：real_stream/fixed_script |
| think_chunk_fixed_scripts.enabled | boolean | true | 是否启用固定话术帧 |
| chars_per_frame | integer | 4 | 每帧字符数，0表示整句推送不切分 |
| tokens_between_frames | integer | 2 | 每累积多少个LLM token推送一个子帧 |
| min_interval_ms | integer | 50 | 子帧间最小间隔毫秒数，0表示不限速 |
| default_scripts | list\<string\> | ["正在分析您的需求..."] | planning阶段默认话术 |
| query_patterns | list\<QueryPattern\> | [] | 关键词匹配话术组 |
| execution_scripts | list\<string\> | ["正在分析执行结果..."] | execution阶段话术 |
| resume_scripts | list\<string\> | ["当前业务步骤已为您处理完毕"] | resume阶段话术 |

**资源限制字段继承规则**：
- chars_per_frame、tokens_between_frames、min_interval_ms属于资源限制类字段
- 场景配置继承框架配置时取min值，场景不能放宽框架限制
- 话术列表类字段（default_scripts/execution_scripts/resume_scripts）采用替代式覆盖，场景有配置时完全替代框架默认
- query_patterns采用追加策略，框架通用模式 + 场景业务关键词合并

### 3. query_patterns关键词匹配配置示例

场景级配置中可以添加业务关键词匹配：

```yaml
think_chunk_scripts:
  think_chunk_fixed_scripts:
    query_patterns:
      # 理财推荐场景
      - keywords: ["推荐", "理财", "产品", "收益"]
        scripts:
          - "正在为您筛选合适的理财产品..."
          - "分析产品风险等级..."
          - "匹配您的风险偏好..."
      
      # 转账场景
      - keywords: ["转账", "汇款", "转钱"]
        scripts:
          - "正在验证收款账户..."
          - "检查账户余额..."
          - "准备转账信息..."
      
      # 余额查询场景
      - keywords: ["余额", "查询", "多少钱"]
        scripts:
          - "正在连接账户系统..."
          - "查询账户余额..."
```

匹配规则：按声明顺序遍历，首个关键词组命中即生效，未命中任何关键词组则使用default_scripts。

## 使用示例

### 1. real_stream模式事件流

当think_chunk_mode=real_stream时，直接推送LLM返回的reasoning_content：

```
# 第1轮：规划阶段
event: think_start
data: {"timestamp": 1234567890}

event: think_chunk
data: {"content": "用户想要推荐理财产品，我需要先创建任务列表...", "timestamp": 1234567890}

event: think_chunk
data: {"content": "首先调用todo_create，使用product_recommend模板...", "timestamp": 1234567891}

event: think_end
data: {"timestamp": 1234567891}

# （后续调用todo_create、call_versatile等工具）

# 第2轮：执行阶段（工具返回后反思）
event: think_start
data: {"timestamp": 1234567895}

event: think_chunk
data: {"content": "工具返回了5款理财产品，需要整理后展示给用户...", "timestamp": 1234567895}

event: think_end
data: {"timestamp": 1234567896}
```

### 2. fixed_script模式逐字渲染

当think_chunk_mode=fixed_script时，按chars_per_frame切分话术逐帧推送：

```
# planning阶段，话术："正在分析您的需求..."（11个字符，chars_per_frame=4）
event: think_start
data: {"timestamp": 1234567890}

event: think_chunk
data: {"content": "正在", "timestamp": 1234567890}

event: think_chunk
data: {"content": "分析", "timestamp": 1234567940}  # 50ms间隔

event: think_chunk
data: {"content": "您的", "timestamp": 1234567990}

event: think_chunk
data: {"content": "需求", "timestamp": 1234568040}

event: think_chunk
data: {"content": "...", "timestamp": 1234568090}

event: think_end
data: {"timestamp": 1234568090}
```

### 3. 各阶段话术触发时机

| 阶段 | 触发时机 | 使用话术 |
|------|---------|---------|
| planning | 第1轮LLM调用，决定调用todo_create时 | default_scripts或query_patterns匹配的话术 |
| execution | 第2轮及后续，工具执行完反思决策时 | execution_scripts |
| resume | 用户中断后回复"continue"触发续轮时 | resume_scripts |

### 4. planning_start事件

planning_start是无配对事件，表示进入规划阶段，在think_start之前发射：

```
event: planning_start
data: {"content": "我们正在为您进行规划。", "timestamp": 1234567890}

event: think_start
data: {"timestamp": 1234567890}
# ... think_chunk序列 ...
event: think_end
data: {"timestamp": 1234567891}
```

planning_start发射条件（per-request去重）：
- 本轮LLM决定调用todo_create（主动规划）时发射
- PLAN_FIRST拦截（强制规划）时也发射
- 寒暄、超范围、直接作答的轮次不发射

### 5. 延迟think机制

当LLM本轮只调用todo_modify（无业务工具）时，think延迟到todo_end之后发射，保证事件流顺序合理：

```
# 正常顺序：tool_end → todo_end → think → todolist
event: tool_end
data: {"tool": "todo_modify", ...}

event: todo_end
data: {"id": "task-1", "status": "completed", ...}

event: think_start
data: {...}
event: think_chunk
data: {"content": "任务1完成了，接下来..."}
event: think_end
data: {...}

event: todolist_start
data: {...}
event: todolist_item
data: {...}
event: todolist_end
data: {...}
```

## 关键实现机制

### 1. think内容优先级

EdpaEventRail.afterModelCall()选择think_chunk内容时：

1. 优先使用reasoning_content（推理模型的思维链输出）
2. 如果reasoning为空或仅为标点占位（如"."/"。"），回退到content（非推理模型的实际输出）
3. 内容为空时只发think_start/think_end对，不发think_chunk

```java
// 核心逻辑
String reasoning = msg.getReasoningContent();
String content = safe(msg.getContentAsString());
String thinkContent = (reasoning != null && reasoning.strip().length() > 1) ? reasoning : content;
```

代码位置：`EdpaEventRail.java:242-247`

### 2. 事件配对保证机制

EdpaEventRail使用状态标记保证think_start与think_end严格配对：

- `thinkOpen` Map（sessionId→boolean）：标记当前轮think_start是否未闭合
- afterModelCall发think前置true，发think_end后置false
- onModelException时检查thinkOpen，若未闭合则补发think_end，保证配对完整
- 每轮LLM调用严格发一对think_start/think_end，不遗漏不重复

```java
// 异常时保证配对
@Override
public void onModelException(AgentCallbackContext ctx) {
    String sid = sessionId(ctx);
    if (Boolean.TRUE.equals(thinkOpen.get(sid))) {
        emit(ctx, EdpaEventType.THINK_END, Map.of());
        thinkOpen.put(sid, false);
    }
    // ... 发error_event和conversation_end
}
```

代码位置：
- 正常配对：`EdpaEventRail.java:257-271`
- 异常配对：`EdpaEventRail.java:399-418`
- 状态定义：`EdpaEventRail.java:100-104`

### 3. think_chunk与final_answer顺序

严格保证think_end在final_answer_start之前（Rule 3/C4）：

1. ①先处理think：有reasoning→think_start→think_chunk→think_end
2. ②再判断是否为最终回答：finish_reason=stop且无tool_calls→final_answer_start→final_answer_chunk→final_answer_end
3. ③有tool_calls则不发final_answer，由后续工具事件处理

```
# 正确顺序
think_start → think_chunk → think_end → final_answer_start → final_answer_chunk → final_answer_end
```

代码位置：`EdpaEventRail.java:257-279`

### 4. 事件流适配器（EdpaEventStreamAdapter）

EdpaEventStreamAdapter负责将Rail发射的custom OutputSchema转换为前端可消费的SSE帧：

1. custom类型OutputSchema→AgentExecutionResult.output（target=USER）
2. llm_output类型OutputSchema→抑制（不转发），因为LLM内容已通过think_chunk/final_answer_chunk发射，避免重复
3. 其他类型委托给OpenJiuwenStreamAdapter默认处理
4. 序列化为JSON格式推送到前端

```java
// 关键逻辑：抑制裸LLM输出，避免重复
if ("llm_output".equals(chunk.getType())) {
    return null;  // 不转发，内容已在事件中
}
```

代码位置：`EdpaEventStreamAdapter.java:42-67`

### 5. 配置模型（ScriptConfig.java）

ThinkChunkScripts配置模型完整定义了思维链配置结构：

- ThinkChunkScripts：顶层配置，包含mode和fixedScripts
- ThinkChunkFixedScripts：固定话术帧配置，包含所有参数和三阶段话术
- QueryPattern：关键词匹配规则，包含keywords列表和scripts列表

继承策略：
- 资源限制参数（chars_per_frame/tokens_between_frames/min_interval_ms）：取min，场景不能放宽框架限制
- 话术列表（default_scripts/execution_scripts/resume_scripts）：替代式覆盖，场景配置完全替换默认
- query_patterns：追加策略，框架通用模式 + 场景业务关键词合并

代码位置：`ScriptConfig.java:93-171`

### 6. thinkingContent回退逻辑

对于非推理模型（不返回reasoning_content），系统自动回退使用content字段作为think内容：

- reasoning != null且长度>1（非标点占位）：使用reasoning
- 否则：使用content（LLM实际输出内容）
- 保证非推理模型也能展示思维链，不出现空白

## 注意事项

1. **事件严格配对**：每轮LLM调用必须有一对think_start/think_end，异常时也要补发think_end闭合，前端依赖配对做状态管理，不要遗漏。

2. **think_end在final_answer之前**：必须保证think_end先发射，再发射final_answer_start，否则前端渲染顺序错乱。

3. **planning_start不滥发**：只有真正进入规划阶段（todo_create或PLAN_FIRST拦截）时才发射planning_start，寒暄、超范围等不规划的轮次不要发，避免误导用户。

4. **延迟think时机**：只在LLM本轮只调用todo_modify（无业务工具）时延迟think到todo_end之后，保证事件流逻辑顺序：tool_end→todo_end→think→todolist。

5. **裸LLM输出抑制**：EdpaEventStreamAdapter会抑制llm_output类型的原始输出，所有用户可见内容必须通过think_chunk/final_answer_chunk/tool_start/tool_end等事件发射，不要依赖框架默认的llm_output。

6. **资源限制继承**：场景配置chars_per_frame等资源限制时，不能大于框架默认值，继承时取最小值，防止场景配置放宽限制导致前端渲染压力过大。

7. **query_patterns顺序敏感**：关键词匹配按声明顺序遍历，首个命中即生效，请注意业务关键词的优先级顺序，把更具体的关键词组放在前面。

8. **reasoning标点占位处理**：LLM有时返回单个"."或"。"作为reasoning，此时视为无有效推理内容，自动回退到content字段，避免展示无意义标点。

9. **跨轮状态正确清理**：thinkOpen标记在afterInvoke时清理，interruptActive等跨轮状态不清理，保证中断后续轮能正确配对interrupt_end。

10. **fixed_script模式帧参数调优**：
    - chars_per_frame越小，逐字效果越明显，但事件数越多
    - min_interval_ms越小，推送越快，但可能造成前端渲染压力
    - tokens_between_frames控制与LLM输出的同步节奏
    - 建议生产环境使用默认值（4/2/50），根据前端性能调整
