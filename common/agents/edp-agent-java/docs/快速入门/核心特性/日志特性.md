# 日志特性

## 特性概述

EDPAgent 的日志系统由 **LogRail（观测日志 Rail）** 和 **EdpaEventRail（事件流 Rail）** 两部分组成，分别负责系统执行日志记录和前端事件流推送。日志系统支撑端到端穿刺证据收集、token用量统计、异常诊断和审计追踪。

核心实现位于：
- 观测日志 Rail：engine/src/main/java/com/huawei/ascend/edp/rail/LogRail.java
- 事件流 Rail：engine/src/main/java/com/huawei/ascend/edp/rail/EdpaEventRail.java
- 日志配置：engine/src/main/resources/application.yml

核心能力包括：
- **E2E 模型调用日志**：记录模型输入消息数量、最后一条消息、模型完整响应
- **Token 用量统计**：每次模型调用记录 inputTokens/outputTokens/totalTokens，支持成本核算
- **工具调用日志**：记录每个工具调用的完成事件和工具名
- **[EDPA-DIAG] 诊断日志**：事件发射全过程的诊断追踪日志，便于排查事件流问题
- **结构化事件流**：22种事件类型严格配对推送前端，覆盖会话全生命周期
- **异常分类日志**：自动分类模型/工具异常类型（LLM_TIMEOUT/LLM_AUTH_ERROR/TOOL_TIMEOUT等）
- **多环境日志级别**：dev默认DEBUG，prod默认WARN，支持环境变量动态控制

## 日志架构

Rail优先级说明（数字越大越早执行）：
- LogRail 优先级为 **10**（最低），确保在其他业务 Rail 完成处理后记录最终上下文
- EdpaEventRail 优先级为 **80**，保证在 TaskPlanningRail(90) 之后执行，可以读取刷新后的 Todo 缓存

## LogRail 观测日志

LogRail 是核心的执行观测 Rail，记录模型调用和工具调用的关键信息。

### 模型调用前日志（beforeModelCall）

在每次调用 LLM 之前记录：

```
E2E_MODEL_INPUT messageCount={N}, lastMessage={最后一条消息内容}
```

**日志字段说明**：

| 字段 | 含义 |
|------|------|
| `messageCount` | 当前发送给模型的消息总数（含 system prompt、历史对话、工具结果） |
| `lastMessage` | 消息列表中的最后一条消息，通常是用户输入或工具返回结果 |

### 模型调用后日志（afterModelCall）

模型返回响应后记录两条日志：

**1. 模型输出日志**：
```
E2E_MODEL_OUTPUT response={AssistantMessage对象}
```

**2. Token 用量日志**（当响应为 AssistantMessage 且包含 usageMetadata 时）：
```
E2E_MODEL_USAGE inputTokens={N}, outputTokens={M}, totalTokens={K}, model={模型名}
```

如果 usage 为 null，记录 `E2E_MODEL_USAGE null`。

**日志字段说明**：

| 字段 | 含义 |
|------|------|
| `inputTokens` | 输入 token 数（prompt tokens） |
| `outputTokens` | 输出 token 数（completion tokens） |
| `totalTokens` | 总 token 数 |
| `model` | 使用的模型名称（如 deepseek-v4-pro） |

### 工具调用完成日志（afterToolCall）

工具执行完成后记录：
```
LogRail: tool call completed, toolName={工具名}
```

记录所有工具的完成事件，包括框架工具（todo_create/todo_modify等）和业务工具（call_versatile/call_mcp/ask_user/cancel_task）。

## EdpaEventRail 诊断日志 [EDPA-DIAG]

EdpaEventRail 以 `[EDPA-DIAG]` 为前缀输出诊断日志，覆盖事件发射的全流程，是排查前端事件流问题的主要依据。

### 会话生命周期日志

| 时机 | 日志关键词 | 说明 |
|------|---------|------|
| 会话开始 | `beforeInvoke sid={} -> emit conversation_start` | 记录会话启动和初始Todo状态 |
| 模型响应后 | `afterModelCall sid={} -> emit think_/final_answer_` | 记录思维链和最终回答事件发射 |
| 工具调用前 | `beforeToolCall tool={} mode={} -> emit tool_start` | 业务工具调用，mode为real-exec或interrupt-handled |
| PLAN_FIRST拦截 | `PLAN_FIRST_BLOCK (真拦截, 不发 tool_start)` | 规划前置守卫拦截，工具未执行 |
| 工具调用后 | `afterToolCall tool={} -> emit tool_end` | 业务工具完成 |
| Todo变化后 | `afterToolCall todo tool={} -> emitTodoEvents` | Todo状态变更，发射todo事件 |
| Todo加载 | `LOAD_CURRENT_TODOS source=REDIS/FILE/CACHE` | 记录Todo数据来源（Redis优先） |
| 中断恢复 | `interrupt resume -> emit interrupt_end` | ask_user中断恢复 |
| 正常中断 | `ToolInterruptException -> emit interrupt_start` | ask_user触发正常中断 |
| 会话结束 | `afterInvoke sid={} -> emit conversation_end` | 会话正常结束 |
| 出口话术 | `emit exit request_start` | 最终话术发射 |
| 合规拦截 | `compliance gate replaced key={}` | 配置外话术被替换为out_of_scope |

### 异常诊断日志

| 时机 | 日志关键词 | 说明 |
|------|---------|------|
| 模型异常 | `onModelException -> emit error_event(stage=model, error_type={})` | 模型调用异常，包含错误分类 |
| 工具异常 | `onToolException -> emit error_event(stage=tool, error_type={})` | 工具执行异常 |
| Think闭合 | `emit think_end (关闭未闭合 think_start)` | 异常时关闭未闭合事件 |
| Tool闭合 | `emit tool_end(status=failed)` | 异常时关闭未闭合工具事件 |

## 异常分类体系

EdpaEventRail 自动将异常分为6种 `error_type`，对应不同的用户可见话术：

### 模型异常分类（classifyModelError）

| error_type | 触发条件 | 用户可见话术 |
|-----------|---------|------------|
| `LLM_TIMEOUT` | HttpTimeoutException、message含timeout/timed out | "LLM 调用超时，请稍后重试" |
| `LLM_AUTH_ERROR` | BaseError status=MODEL_SERVICE_CONFIG_ERROR、cause含401/unauthorized | "LLM 认证失败，请检查 API Key 配置" |
| `INTERNAL_ERROR` | 其他所有模型异常 | "系统内部错误，请稍后重试" |

分类优先级：框架BaseError.StatusCode枚举 → cause原始异常类型 → message文字匹配 → 兜底。

### 工具异常分类（classifyToolError）

| error_type | 触发条件 | 用户可见话术 |
|-----------|---------|------------|
| `TOOL_TIMEOUT` | 异常class/message含timeout/timed out | "工具执行超时，请稍后重试" |
| `INVALID_TOOL_OUTPUT` | message含json/parse/invalid，class含JsonParse/JsonMapping | "工具返回数据格式错误" |
| `DEPENDENCY_VIOLATION` | message含"依赖"/depend/"循环" | "任务依赖校验失败，存在缺失或循环依赖" |
| `INTERNAL_ERROR` | 其他所有工具异常 | "系统内部错误，请稍后重试" |

## 事件类型与配对规则

EdpaEventRail 发射22种事件类型，所有成对事件严格配对（start数 == end数），异常时自动关闭未闭合事件。

### 生命周期事件

| 事件 | 是否成对 | 说明 |
|------|---------|------|
| `conversation_start` | 与conversation_end配对 | 会话开始 |
| `conversation_end` | 与conversation_start配对 | 会话结束（正常或异常） |
| `request_start` | 无配对 | 出口话术发射（interrupt时不发） |

### 规划事件

| 事件 | 是否成对 | 说明 |
|------|---------|------|
| `planning_start` | 无配对 | 开始任务规划（仅本轮有todo_create时发） |

### 思维链事件

| 事件 | 是否成对 | 说明 |
|------|---------|------|
| `think_start` | 与think_end配对 | 思考开始（每轮LLM调用一对） |
| `think_chunk` | 无配对 | 思考内容片段（reasoning_content或content） |
| `think_end` | 与think_start配对 | 思考结束 |

### 工具事件

| 事件 | 是否成对 | 说明 |
|------|---------|------|
| `tool_start` | 与tool_end配对 | 业务工具开始执行（仅call_versatile/call_mcp） |
| `tool_end` | 与tool_start配对 | 业务工具执行完成（含failed状态） |

### 最终回答事件

| 事件 | 是否成对 | 说明 |
|------|---------|------|
| `final_answer_start` | 与final_answer_end配对 | 最终回答开始 |
| `final_answer_chunk` | 无配对 | 最终回答内容 |
| `final_answer_end` | 与final_answer_start配对 | 最终回答结束 |

### Todo事件

| 事件 | 是否成对 | 说明 |
|------|---------|------|
| `todolist_start` | 与todolist_end配对 | Todo列表刷新开始 |
| `todolist_item` | 无配对 | 单条Todo（N条Todo发N个item） |
| `todolist_end` | 与todolist_start配对 | Todo列表刷新结束 |
| `todo_start` | 与todo_end配对 | 单个任务开始执行（PENDING→IN_PROGRESS） |
| `todo_end` | 与todo_start配对 | 单个任务结束（IN_PROGRESS→COMPLETED/CANCELLED） |

### 中断事件

| 事件 | 是否成对 | 说明 |
|------|---------|------|
| `interrupt_start` | 与interrupt_end配对 | 中断等待用户输入（ask_user触发） |
| `interrupt_end` | 与interrupt_start配对 | 中断恢复（用户回复后） |

### 错误事件

| 事件 | 是否成对 | 说明 |
|------|---------|------|
| `error_event` | 无配对 | 错误事件（stage=model/tool, error_type=分类） |

**事件配对规则**（来自代码注释）：
- Rule 1：conversation_start == conversation_end（即使异常也保证）
- Rule 2：think_start == think_end（每轮LLM一对，异常时自动关闭）
- Rule 3：think_end 在 final_answer_start 之前
- Rule 6：tool_start == tool_end（业务工具才发，异常时以failed状态关闭）
- Rule 7：interrupt_start（本轮末）→ interrupt_end（下轮首）
- Rule 8：异常不破坏配对——从内到外依次关闭未闭合事件
- Rule 9：conversation_start时不发跨轮todolist（前端自行持久化）
- Rule 12：N条todo发N个todolist_item（非tasks数组）

## 日志配置

### Spring Boot 日志级别配置

通过 `application.yml` 中的 `logging.level` 配置包级别日志：

**默认配置（application.yml）**：
```yaml
logging:
  level:
    com.huawei.ascend.edp: DEBUG
    com.openjiuwen.service.adapters.agentcore.ext: DEBUG
    com.openjiuwen.service.app.orchestrator: DEBUG
```

**生产环境配置（application-prod.yml）**：
```yaml
logging:
  level:
    com.huawei.ascend.edp: WARN
    com.openjiuwen: WARN
```

### 环境变量控制

通过 `EDP_AGENT_LOG_LEVEL` 环境变量可以控制日志级别（参考 .env.example）：

```bash
# 开发调试（最详细）
EDP_AGENT_LOG_LEVEL=DEBUG

# 正常运行（默认）
EDP_AGENT_LOG_LEVEL=INFO

# 生产环境（仅警告和错误）
EDP_AGENT_LOG_LEVEL=WARN

# 静默模式（仅错误）
EDP_AGENT_LOG_LEVEL=ERROR
```

可以通过 Java 系统属性方式传入：
```bash
java -Dlogging.level.com.huawei.ascend.edp=${EDP_AGENT_LOG_LEVEL:-INFO} -jar edp-agent.jar
```

### 日志级别建议

| 环境 | com.huawei.ascend.edp | com.openjiuwen | 说明 |
|------|----------------------|----------------|------|
| 开发调试 | DEBUG | DEBUG | 查看所有[EDPA-DIAG]诊断日志和E2E日志 |
| 测试/POC | INFO | INFO | 记录E2E日志和关键诊断信息 |
| 生产 | WARN | WARN | 仅记录警告和错误，减少日志量 |
| 问题排查 | DEBUG | DEBUG | 临时开启，问题解决后恢复 |

## 关键日志关键词

排查问题时，可用以下关键词快速grep日志：

| 关键词 | 用途 |
|--------|------|
| `E2E_MODEL_INPUT` | 查看模型输入消息 |
| `E2E_MODEL_OUTPUT` | 查看模型输出响应 |
| `E2E_MODEL_USAGE` | 查看token消耗 |
| `[EDPA-DIAG]` | 查看所有诊断日志 |
| `ERROR` | 查看所有错误 |
| `error_event` | 查看错误事件详情 |
| `ToolInterruptException` | 查看正常中断（ask_user） |
| `PLAN_FIRST_BLOCK` | 查看规划前置拦截 |
| `LOAD_CURRENT_TODOS` | 查看Todo数据加载来源 |
| `interrupt_start` / `interrupt_end` | 跟踪中断续传流程 |
| `conversation_start` / `conversation_end` | 跟踪会话生命周期 |
| `LLM_TIMEOUT` | 查看模型超时错误 |
| `LLM_AUTH_ERROR` | 查看模型认证错误 |
| `TOOL_TIMEOUT` | 查看工具超时错误 |
| `compliance gate` | 查看合规拦截记录 |

### 常用日志查询命令

**Docker环境查看最近错误**：
```bash
docker logs edp-agent --tail 500 2>&1 | grep -E "ERROR|error_event|E2E_MODEL_USAGE"
```

**查看特定会话的完整日志**：
```bash
docker logs edp-agent --tail 2000 2>&1 | grep "sid=your-session-id"
```

**统计token消耗**：
```bash
docker logs edp-agent 2>&1 | grep "E2E_MODEL_USAGE" | awk -F'totalTokens=' '{sum+=$2} END {print "Total tokens:", sum}'
```

**查看工具调用记录**：
```bash
docker logs edp-agent 2>&1 | grep "tool call completed"
```

## Docker 日志配置

生产部署建议配置 Docker 日志轮转，避免日志文件占满磁盘：

```bash
# daemon.json 配置
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "100m",
    "max-file": "5"
  }
}
```

或 docker run 时指定：
```bash
docker run --log-opt max-size=100m --log-opt max-file=5 ...
```

## Token 用量统计与成本核算

LogRail 每次模型调用都会输出 `E2E_MODEL_USAGE` 日志，可用于统计 token 消耗和核算推理成本。

### 日志示例
```
E2E_MODEL_USAGE inputTokens=2456, outputTokens=312, totalTokens=2768, model=deepseek-v4-pro
```

### 注意事项

1. **DEBUG 级别记录完整模型输入输出**：生产环境务必设为 WARN 或 INFO，避免 Prompt 内容（可能含用户敏感数据）被记录
2. **usage 为 null 的情况**：某些模型或错误场景下不返回 usage，日志会输出 `E2E_MODEL_USAGE null`
3. **reasoning tokens**：reasoning_content（思维链）的 token 会被计入 outputTokens
4. **System Prompt token**：每次调用都会重复发送 System Prompt，其 token 计入 inputTokens

## 端到端穿刺证据收集

LogRail 配合 EdpaEventRail 的诊断日志，可构建完整的端到端穿刺证据链：

### 一次完整请求的日志序列

1. `[EDPA-DIAG] beforeInvoke sid=xxx -> emit conversation_start`
2. `E2E_MODEL_INPUT messageCount=N, lastMessage=...`
3. `[EDPA-DIAG] afterModelCall -> emit think_start/think_chunk/think_end`
4. `[EDPA-DIAG] beforeToolCall tool=xxx -> emit tool_start`（业务工具）
5. `[EDPA-DIAG] afterToolCall tool=xxx -> emit tool_end`
6. `LogRail: tool call completed, toolName=xxx`
7. `E2E_MODEL_OUTPUT response=...`
8. `E2E_MODEL_USAGE inputTokens=..., outputTokens=..., totalTokens=...`
9. （ReAct循环重复步骤2-8）
10. `[EDPA-DIAG] afterModelCall -> emit final_answer_start/final_answer_chunk/final_answer_end`
11. `[EDPA-DIAG] afterInvoke -> emit exit request_start`
12. `[EDPA-DIAG] afterInvoke -> emit conversation_end`

通过以上日志序列，可以完整还原 Agent 的思考、规划、工具调用、结果输出全流程。

## 注意事项

1. **生产环境禁用 DEBUG**：DEBUG 级别会记录完整的模型输入输出（含用户消息和System Prompt），可能泄露业务数据和Prompt工程细节
2. **LogRail低优先级设计**：LogRail优先级为10（最低），确保其他Rail修改上下文后记录的是最终状态
3. **EdpaEventRail仅发SSE事件**：[EDPA-DIAG]是SLF4J日志，event事件本身是SSE推送给前端的，不写入日志文件
4. **异常配对保证**：任何异常情况下EdpaEventRail都会先关闭未闭合的start事件，再发error_event和conversation_end，保证前端事件流不卡死
5. **ToolInterruptException不是错误**：ask_user触发的ToolInterruptException是正常中断机制，不会发error_event，只发interrupt_start
6. **Redis优先数据源**：Todo日志中`LOAD_CURRENT_TODOS source=REDIS`是正常路径，如果看到source=FILE或CACHE说明Redis不可用
7. **日志脱敏预留**：LogRail构造函数接收EdpConfig参数，预留给后续日志脱敏、采样和开关控制使用
8. **Docker日志轮转**：务必配置Docker日志驱动的max-size和max-file，否则长时间运行日志会占满磁盘
9. **[EDPA-DIAG]诊断日志是INFO级别**：即使在INFO级别下也能看到关键诊断信息
