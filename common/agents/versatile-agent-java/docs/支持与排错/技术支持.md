# 技术支持渠道

## 遇到问题时

在寻求技术支持前，请先按以下顺序自查：

1. **查阅文档**
   - [常见问题 FAQ](faq.md) - 先看是否有你的问题
   - [服务启动失败排错](troubleshooting/startup-failure.md) - 服务启动不了按此排查
   - [Versatile调用异常排错](troubleshooting/versatile-error.md) - Versatile调用失败按此排查
   - [版本变更记录](changelog.md) - 检查是否是已知问题或版本变更

2. **查看日志**
   - 收集启动日志和错误日志：`docker logs adapter-versatile --tail 200 > adapter-log.txt`
   - 注意 ERROR 和 WARN 级别的日志
   - 开启 DEBUG 日志复现问题获取更详细信息（默认已开启相关包DEBUG）
   - 重点关注VersatileAgentHandler、HttpClient、SSE节点解析相关日志

3. **基础检查**
   - 确认服务能正常启动：`curl http://localhost:8191/actuator/health`返回UP
   - 确认能直接访问Versatile后端（绕过代理用curl测试）
   - 确认环境变量配置正确（特别是VERSATILE_URL格式）
   - 确认Docker/Java版本满足要求（JDK 17+/Docker JDK 21）
   - 确认VERSATILE_RESULT_NODE配置与Versatile实际返回节点名一致

### 诊断信息收集命令

提交问题前，请先收集以下诊断信息，有助于快速定位问题：

#### 1. 服务版本信息

```bash
# Docker镜像版本
docker inspect adapter-versatile --format '{{.Config.Image}}'

# 查看jar包版本（本地启动）
ls -lh target/adapter-versatile-agent-java-*.jar
unzip -p target/adapter-versatile-agent-java-*.jar META-INF/MANIFEST.MF | grep -i version

# Docker容器启动时间
docker inspect adapter-versatile --format='{{.State.StartedAt}}'
```

#### 2. 健康检查状态

```bash
# Actuator健康检查
curl -s http://localhost:8191/actuator/health
# 预期：{"status":"UP"}

# Agent Card端点
curl -s http://localhost:8191/.well-known/agent-card.json | head -50
# 预期：返回JSON包含agentId、description、skills等

# Docker健康状态
docker inspect adapter-versatile --format='{{json .State.Health}}'
# 等待90秒启动期后，Status应为healthy
```

#### 3. 关键日志收集

```bash
# 最近200行日志
docker logs adapter-versatile --tail 200 > adapter-recent.log

# 错误和异常日志
docker logs adapter-versatile 2>&1 | grep -E "ERROR|Exception|Failed|Caused by" > adapter-errors.log

# 启动日志（前100行）
docker logs adapter-versatile 2>&1 | head -100 > adapter-startup.log

# Versatile调用相关日志
docker logs adapter-versatile 2>&1 | grep -iE "versatile|POST|response|SSE|node|result" > adapter-versatile-calls.log

# 实时日志监控复现问题
docker logs adapter-versatile -f
```

#### 4. Versatile连通性测试

**重要：必须先绕过代理直接测试Versatile，确认Versatile本身正常！**

```bash
# 替换{conversation_id}为测试值，测试Versatile直接连通性
curl -v -X POST "YOUR_VERSATILE_URL" \
  -H "Content-Type: application/json" \
  -H "accept: */*" \
  -H "stream: true" \
  -H "x-invoke-mode: DEBUG" \
  -H "x-language: zh-cn" \
  -d '{"inputs": {"query": "测试查询"}, "stream": true}' \
  --no-buffer

# 记录：
# 1. HTTP状态码
# 2. 返回的SSE流完整内容
# 3. 结果节点的node_name是什么
# 4. 是否正常返回[data: DONE]
```

#### 5. A2A端点测试

```bash
# 测试A2A端点，替换contextId
curl -N -X POST "http://localhost:8191/a2a" \
  -H "Content-Type: application/json" \
  -H "Accept: text/event-stream" \
  -d '{
    "jsonrpc": "2.0",
    "method": "SendStreamingMessage",
    "id": "diagnostic-test-001",
    "params": {
      "metadata": {
        "userId": "diagnostic",
        "agentId": "edp-agent",
        "intent": "test"
      },
      "message": {
        "role": "ROLE_USER",
        "messageId": "msg-diag-001",
        "contextId": "test-conv-diag-001",
        "parts": [
          {"text": "{\"inputs\": {\"query\": \"测试\"}}"}
        ]
      }
    }
  }' 2>&1 | tee a2a-test-result.log
```

#### 6. 环境变量和配置检查（脱敏）

```bash
# 列出所有相关环境变量，隐藏敏感值
docker exec adapter-versatile env | grep -E "VERSATILE|JAVA_OPTS|SERVER_PORT" | sed 's/=.*password.*/=***/'

# 查看容器内配置文件
docker exec adapter-versatile cat /app/application.yml 2>/dev/null || echo "config not in /app"

# 查看端口监听情况
docker exec adapter-versatile apt-get update -qq && docker exec adapter-versatile apt-get install -y -qq net-tools && docker exec adapter-versatile netstat -tlnp
```

#### 7. 容器资源使用

```bash
# 容器状态和资源使用
docker stats adapter-versatile --no-stream

# 容器详细信息
docker inspect adapter-versatile | grep -A 10 -E "Memory|CPU|RestartCount|Status"
```

## 自助排错流程图

```
服务/请求异常
    │
    ├─ 服务能启动吗？（访问/actuator/health是否返回UP）
    │   ├─ 否 → [服务启动失败](troubleshooting/startup-failure.md)
    │   │       ├─ 端口8191被占用？→ 更换端口或结束占用进程
    │   │       ├─ JDK版本错误？→ 安装JDK 17+（Docker用JDK 21镜像）
    │   │       ├─ YAML语法错误？→ 检查配置缩进和格式
    │   │       ├─ 环境变量格式错误？→ 检查VERSATILE_TIMEOUT单位、URL格式
    │   │       ├─ OOM内存不足？→ 增大Docker内存或减小JVM堆
    │   │       └─ Jar包损坏？→ mvn clean package重新构建
    │   │
    │   └─ 是 → Agent Card端点正常吗？（/.well-known/agent-card.json）
    │           ├─ 否 → 等待90秒启动期，或查看日志ERROR
    │           └─ 是 → 直接curl Versatile能通吗？（绕过代理）
    │                   ├─ 不能通 → 是Versatile服务问题，联系Versatile团队排查
    │                   │       ├─ Connection refused → Versatile未启动
    │                   │       ├─ 404 → URL路径错误、占位符未替换、workflow_id错
    │                   │       ├─ 5xx → Versatile内部错误
    │                   │       └─ 超时 → 网络问题或Versatile过载
    │                   │
    │                   └─ 能通 → 通过adapter调用有问题吗？
    │                           ├─ 超时/SSE流中断 → 检查VERSATILE_TIMEOUT设置
    │                           ├─ 所有节点都透传 → 检查resultNodeName配置
    │                           │                     （对比实际返回节点的node_name）
    │                           ├─ 始终触发中断 → SSE流中无结果节点
    │                           │                  检查resultNodeName是否匹配
    │                           ├─ Header未传递 → 检查forward-header-whitelist配置
    │                           └─ 其他问题 → 开启DEBUG日志查看详细信息
    │
    └─ 以上都不能解决 → 收集诊断信息提交Issue
```
