# Versatile Agent 健康检查与日志分析

本文档介绍 adapter-versatile-agent-java 的健康检查机制和日志分析方法。

---

## 1. 健康检查端点

服务提供两个 HTTP 端点用于健康检查：

### 1.1 /actuator/health

| 属性 | 值 |
|------|-----|
| 端口 | 8191 |
| 方法 | GET |
| 认证 | 不需要 |
| 响应类型 | application/json |

**端点配置**：`management.endpoints.web.exposure.include=health,info`，参见 [application.yml:38-42](file:///D:/AgentTool/EDPAgent_java/agent-solution/common/agent/adapter-versatile-agent-java/src/main/resources/application.yml#L38-L42)。

**正常响应**：
```bash
curl http://localhost:8191/actuator/health
```
```json
{"status":"UP"}
```

**异常响应**：
```json
{"status":"DOWN"}
```

**说明**：
- `UP`：Spring Boot 应用上下文正常启动完成
- `DOWN`：应用启动失败或运行异常
- 此端点主要验证 Spring Boot 容器本身是否正常

---

### 1.2 /.well-known/agent-card.json

| 属性 | 值 |
|------|-----|
| 端口 | 8191 |
| 方法 | GET |
| 认证 | 不需要 |

**这是 Docker HEALTHCHECK 使用的检查端点**，参见 [Dockerfile:24-25](file:///D:/AgentTool/EDPAgent_java/agent-solution/common/agent/adapter-versatile-agent-java/deploy/Dockerfile#L24-L25)。

```bash
curl http://localhost:8191/.well-known/agent-card.json
```

此端点验证的内容：
1. 端口 8191 正常监听
2. Spring Web 容器正常运行
3. A2A Agent 路由注册成功
4. Agent Card 元数据可正常返回

**为什么用 Agent Card 做 HEALTHCHECK**：
- 比 `/actuator/health` 更全面，不仅检查 Spring Boot 容器，还验证 A2A 服务可用
- 访问此端点返回 200 说明 Agent 已准备好接收 A2A 请求
- 上游 EDPAgent 也是通过此端点发现 Agent 能力

正常响应包含 Agent 信息：
```json
{
  "name": "versatile-agent",
  "description": "Remote runtime that processes banking and external business workflows through Versatile HTTP/SSE.",
  "skills": [...],
  "capabilities": {...},
  ...
}
```

## 2. Docker HEALTHCHECK 机制详解

### 2.1 HEALTHCHECK 参数

Dockerfile 内置 HEALTHCHECK 指令：

```dockerfile
HEALTHCHECK --interval=10s --timeout=5s --retries=8 --start-period=90s \
  CMD curl -sf http://localhost:8191/.well-known/agent-card.json || exit 1
```

| 参数 | 值 | 含义 | 说明 |
|------|-----|------|------|
| `--interval=10s` | 10 秒 | 检查间隔 | 每 10 秒执行一次健康检查 |
| `--timeout=5s` | 5 秒 | 单次检查超时 | curl 命令 5 秒没返回视为失败 |
| `--retries=8` | 8 次 | 重试次数 | 连续失败 8 次才标记 unhealthy |
| `--start-period=90s` | 90 秒 | 启动宽限期 | 容器启动后 90 秒内的失败不计入 retries |

### 2.2 为什么 start-period 设为 90 秒

Spring Boot 应用启动需要时间：
1. JVM 启动
2. Spring 上下文初始化
3. Bean 加载和依赖注入
4. Web 容器（Tomcat）启动
5. A2A 路由注册

90 秒的宽限期给了应用充足的启动时间，避免启动过程中误判为 unhealthy。

### 2.3 检查逻辑

```bash
curl -sf http://localhost:8191/.well-known/agent-card.json || exit 1
```

- `-s`：静默模式，不输出进度
- `-f`：HTTP 状态码 >= 400 时返回失败（exit code 22）
- 如果 curl 命令执行失败（连接拒绝、超时、HTTP 错误等），返回 exit 1，视为一次检查失败

### 2.4 容器健康状态流转

```
容器启动
    │
    ▼
starting ──► (start-period 90秒内，失败不计入重试)
    │
    ├─ 启动90秒内检查成功 ──► healthy
    │
    └─ 启动90秒后开始累计失败
         │
         ├─ 连续8次失败 ──► unhealthy
         │
         └─ 期间任意一次成功 ──► healthy
```

### 2.5 健康状态查看命令

```bash
# 查看健康状态
docker ps
# 输出中 STATUS 列显示：(healthy)、(unhealthy)、(health: starting)

# 获取健康状态字符串
docker inspect --format='{{.State.Health.Status}}' versatile-agent

# 查看最近 5 次检查日志
docker inspect --format='{{range .State.Health.Log}}[{{.Start.Format "2006-01-02 15:04:05"}}] ExitCode={{.ExitCode}} Output={{.Output}}{{"\n"}}{{end}}' versatile-agent
```

## 3. 日志分析

### 3.1 日志位置

服务日志同时输出到两个地方：

| 位置 | 路径 | 说明 |
|------|------|------|
| 控制台（stdout/stderr） | - | Docker 捕获，通过 `docker logs` 查看 |
| 日志文件 | `/app/logs/run/run.log` | 容器内文件，可挂载到宿主机 |

日志配置参见 [application.yml:44-46](file:///D:/AgentTool/EDPAgent_java/agent-solution/common/agent/adapter-versatile-agent-java/src/main/resources/application.yml#L44-L46)：
```yaml
logging:
  file:
    name: logs/run/run.log
```

### 3.2 日志级别

默认日志级别配置（参见 [application.yml:47-50](file:///D:/AgentTool/EDPAgent_java/agent-solution/common/agent/adapter-versatile-agent-java/src/main/resources/application.yml#L47-L50)）：

| 包路径 | 级别 | 说明 |
|--------|------|------|
| `com.huawei.ascend.versatile` | DEBUG | 本模块的配置、Handler 日志 |
| `com.openjiuwen.service.adapters.versatile` | DEBUG | Versatile 适配器、HTTP 客户端日志 |
| `com.openjiuwen.service.app` | DEBUG | A2A 应用框架日志 |
| 其他包 | INFO | Spring Boot 默认 INFO 级别 |

日志级别从低到高：`TRACE < DEBUG < INFO < WARN < ERROR`。

### 3.3 关键日志关键词

排查问题时搜索以下关键词：

| 关键词 | 说明 |
|--------|------|
| `Started VersatileAgentApplication` | 应用启动成功标志 |
| `Tomcat started on port(s)` | Tomcat 启动完成，端口监听 |
| `AgentHandler` | Agent Handler 注册相关 |
| `versatile-agent` | Agent ID 相关日志 |
| `VersatileHttpClient` | Versatile HTTP 调用日志 |
| `A2A request` | A2A 请求处理日志 |
| `conversation_id` | 会话 ID 替换日志 |
| `Result node` | 结果节点识别日志 |
| `ERROR` | 错误日志 |
| `Exception` | 异常堆栈 |
| `Timeout` | 超时相关错误 |
| `ConnectException` | 网络连接失败 |
| `401\|403\|404\|500` | HTTP 错误状态码 |

## 4. 问题诊断

当服务 unhealthy 或功能异常时，按以下步骤排查：

### 4.1 排查步骤流程图

```
服务异常
    │
    ▼
1. 检查容器状态 ──► docker ps -a
    │
    ├─ 容器未运行 (Exited) ──► 查看 docker logs 找启动错误
    │
    └─ 容器运行中但 unhealthy
         │
         ▼
2. 检查端口监听 ──► netstat -tlnp / curl localhost:8191
    │
    ├─ 端口未监听 ──► 应用未启动成功，看启动日志
    │
    └─ 端口监听正常
         │
         ▼
3. 检查健康端点 ──► curl /.well-known/agent-card.json
    │
    ├─ 连接被拒绝 ──► 防火墙/网络问题
    ├─ 返回 404 ──► A2A 路由未注册
    ├─ 返回 5xx ──► 应用内部错误
    │
    └─ 端点能访问
         │
         ▼
4. 检查 Versatile 连通性 ──► curl 测试 Versatile URL（替换{conversation_id}）
    │
    ├─ 连接失败 ──► Versatile 地址/网络问题
    ├─ 超时 ──► Versatile 服务慢或网络延迟
    ├─ 401/403 ──► 认证问题
    ├─ 404 ──► URL 路径错误
    │
    └─ Versatile 连通正常
         │
         ▼
5. 检查详细日志 ──► 看是否有业务异常/配置错误
```

### 4.2 常见问题与解决方案

#### 问题 1：容器启动后一直是 starting，然后变成 unhealthy

**可能原因**：
1. Spring Boot 启动太慢，超过 90 秒
2. 启动过程中抛出异常

**排查**：
```bash
# 查看启动日志
docker logs versatile-agent

# 查找异常关键词
docker logs versatile-agent 2>&1 | grep -A 20 "Exception\|Error starting"
```

**解决方案**：
- 如果是 JVM 内存不足，增加 JVM 堆内存：`-e JAVA_OPTS="-Xms512m -Xmx1024m"`
- 如果是配置错误（如 VERSATILE_URL 格式错误），修正环境变量重启

---

#### 问题 2：curl /actuator/health 返回 UP，但 Docker 显示 unhealthy

**排查**：
```bash
# 直接在宿主机 curl （注意端口映射）
curl http://localhost:8191/.well-known/agent-card.json

# 进入容器内部 curl
docker exec -it versatile-agent sh
curl -v http://localhost:8191/.well-known/agent-card.json
```

**可能原因**：
- 端口映射配置错误
- 容器内 curl 未安装（正常情况下 Dockerfile 已安装 curl，参见 [Dockerfile:14-16](file:///D:/AgentTool/EDPAgent_java/agent-solution/common/agent/adapter-versatile-agent-java/deploy/Dockerfile#L14-L16)）
- A2A 路由注册失败导致 Agent Card 端点 404

---

#### 问题 3：调用 Versatile 时连接失败 (ConnectException: Connection refused)

**排查**：
```bash
# 1. 检查 VERSATILE_URL 配置
docker exec versatile-agent env | grep VERSATILE_URL

# 2. 在容器内测试连通性（替换 {conversation_id} 为测试值）
docker exec -it versatile-agent sh
curl -v "http://your-versatile-host:port/.../conversations/test123?..."
```

**常见原因**：
1. VERSATILE_URL 中 hostname/port 错误
2. Versatile 服务未启动
3. 网络不通（如容器访问不到宿主机服务，需用 `host.docker.internal`）
4. 防火墙拦截

**解决方案**：
- 如果 Versatile 在宿主机，URL 使用 `http://host.docker.internal:port/...`
- 如果 Versatile 在其他容器，加入同一 Docker 网络
- 检查 URL 中的 `{conversation_id}` 是占位符（运行时自动替换），但手动测试时需要替换成真实字符串

---

#### 问题 4：调用超时 (TimeoutException)

**现象**：A2A 请求等待很久后返回超时错误。

**排查**：
1. 检查 VERSATILE_TIMEOUT 配置是否足够（默认 600s）
2. 手动 curl Versatile URL 看响应时间
3. 查看日志中是否有 Versatile 响应慢的记录

**解决方案**：
- 增大超时：`-e VERSATILE_TIMEOUT=900s`
- 检查 Versatile 服务性能
- 检查网络延迟

---

#### 问题 5：Result node not found 警告

**现象**：日志显示 `Result node GXZQAResponseNode not found`。

**排查**：
1. 检查 VERSATILE_RESULT_NODE 配置是否与工作流定义一致
2. 查看 Versatile 返回的所有节点名称

**解决方案**：
- 确认工作流中结果节点的准确名称
- 设置正确的 `-e VERSATILE_RESULT_NODE=YourResultNodeName`

---

### 4.3 获取诊断信息

遇到问题时，收集以下信息便于定位：

```bash
# 1. 容器状态
docker ps -a --filter name=versatile-agent

# 2. 容器配置和环境变量
docker inspect versatile-agent > diagnose-inspect.json

# 3. 完整日志（最近 500 行）
docker logs --tail 500 versatile-agent > diagnose-logs.log 2>&1

# 4. 健康检查历史
docker inspect --format='{{json .State.Health}}' versatile-agent

# 5. 健康端点响应
curl -v http://localhost:8191/actuator/health
curl -v http://localhost:8191/.well-known/agent-card.json

# 6. Versatile 连通性测试（替换为实际 URL 和测试 conversation_id）
curl -v "http://your-versatile-url/.../conversations/test-conv-001?..." -m 30
```
