# 项目背景与演进说明

> 本文整理 OpenClaw Security Hardening Skill 的来龙去脉，帮助读者理解项目发展历程。

---

## 起源

本项目源自 OpenClaw 社区在生产部署过程中遇到的实际安全问题。OpenClaw 是一款容器化部署的 AI Agent 网关，默认部署配置存在明显的安全短板（弱 Token、宽松权限、公网暴露等）。随着越来越多的用户将 OpenClaw 部署到生产环境，社区需要一个**标准化、可审计、可回滚**的安全加固工具来应对这些常见风险。

最初的 v1 版本是面向 Rocky Linux 宿主机部署场景的基础加固脚本。随着 OpenClaw 被容器化（Docker/Podman/LXC）部署的比例逐渐增加，项目逐步扩展到多发行版容器环境，并经历了多次重构和功能增强，演进至当前的 v8.07 版本。

## 各阶段核心贡献

### 阶段一：基础功能（v1–v4）

- 建立 OpenClaw 安全加固的基本脚本框架
- 针对 Rocky Linux 宿主机的基础加固项
- 暴露了**手动加固的痛点**：无统一基线、无回滚能力、无审计

### 阶段二：容器化适配（v5–v6）

- 引入容器运行模式检测（`detect_container_mode`）
- 容器内自动跳过不适用的检查（systemd、SELinux、firewalld）
- 扩展到 Debian/Ubuntu，引入包管理器自动适配
- 修复大量 v4 遗留问题

### 阶段三：安全加固（v7）

- 修复 `safe_node_write` / `safe_node_read` 命令注入漏洞：将 shell 变量拼接改为 stdin 传递
- 备份文件增加 SHA256 完整性校验
- 备份自动保留策略（保留最新 5 个，防止磁盘占满）
- 恢复前 SHA256 验证，防止恢复损坏的备份
- 新增 health-check.sh 8 项日常健康检查
- 新增 Gateway HTTP 健康检查
- 新增 `DEPLOY-GUIDE.md` 和 `PRODUCT-OVERVIEW.md`

⚠️ **v7 的重构事故**：将 `safe_node_read` 批量重命名为 `safe_node_get` 时，4 个脚本中产生了 `}fe_node_read() {` 语法残留，导致后验证、前后对比、报告、恢复验证四个核心脚本全部无法运行。

### 阶段四：修复与优化（v8）

- 修复 v7 的 4 个脚本致命语法错误
- 修复 `safe_node_get` 调用参数（旧 JS 代码表达式 → 纯键路径格式）
- health-check.sh 优先使用 `jq`（速度提升 10x+），无 jq 时自动回退 node
- 健康检查扩展到 10 项（新增 Gateway HTTP 可达性 + 进程运行时长）
- JSON 空值兜底（bind_address、config_permissions 等字段规范化）
- 算术表达式增加正则校验，防止空字符串导致 crash

## 整合为 Skill 提交

当前 v8.07 已经是一个功能完整、稳定可部署的工具包。为了融入 openJiuwen 的 Agent 生态，我们将其整理为符合 [agent-solution](https://gitcode.com/openJiuwen/agent-solution) 规范的 Skill，主要工作包括：

1. **重构 SKILL.md**：按照 Skill 规范重写，补充触发条件、工作流、能力边界、输出格式说明
2. **目录重组**：将原文档归档到 `references/`，保留 `scripts/` 和 `templates/` 作为可执行资产
3. **统一版本号**：对外标记为 Skill v1.0.0，脚本内部保留 v8.07 标记
4. **许可证转换**：从 MIT 改为 Apache 2.0，与 agent-solution 仓库对齐
5. **补充概述**：在 SKILL.md 开头清晰阐述"为什么需要"、"痛点"、"核心价值"、"预期效果"

## 后续演进方向（未实施）

以下方向在 `PRODUCT-OVERVIEW.md` 中已有讨论，但当前版本尚未实现：

- **镜像扫描集成**：接入 Trivy 等工具扫描容器镜像 CVE
- **Windows 版本**：PowerShell 实现的 Windows 宿主加固工具包
- **加固策略模板**：针对金融、医疗、政务等行业提供预设策略
- **Web 管理界面**：可视化 Dashboard，非技术人员可操作
- **远程编排**：批量管理多台目标机器
- **实时威胁检测**：集成 fail2ban 等轻量 IDS

这些方向将作为未来版本的候选特性，根据社区反馈和资源投入情况逐步推进。
