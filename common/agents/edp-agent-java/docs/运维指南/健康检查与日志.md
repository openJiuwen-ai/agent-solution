# 健康检查与日志分析

## 健康检查机制

EDPAgent 提供 Docker 原生 HEALTHCHECK 支持，通过访问 Agent Card 端点判断服务健康状态。

### Docker HEALTHCHECK 配置

在 [deploy/Dockerfile](file:///D:/AgentTool/EDPAgent_java/branch_edpa_java/agent-store/edp-agent-java/deploy/Dockerfile) 中已预配置：

```dockerfile
HEALTHCHECK --interval=10s --timeout=5s --retries=5 --start-period=60s \
  CMD curl -sf http://localhost:8190/.well-known/agent-card.json || exit 1
```

| 参数 | 值 | 说明 |
|------|-----|------|
| `--interval=10s` | 每10秒检查一次 | 检查频率 |
| `--timeout=5s` | 单次检查超时5秒 | 单次超时阈值 |
| `--retries=5` | 连续5次失败标记为unhealthy | 失败重试次数 |
| `--start-period=60s` | 启动后60秒开始计算失败 | 给服务启动留足时间 |

### 健康状态查看

```bash
# 查看容器健康状态
docker inspect edp-agent --format '{{.State.Health.Status}}'
# 返回值：starting / healthy / unhealthy

# 查看健康检查历史
docker inspect edp-agent --format '{{json .State.Health.Log}}' | python -m json.tool

# 查看容器列表（包含健康状态）
docker ps
```

健康状态说明：
- `starting`：服务正在启动中（start-period内）
- `healthy`：健康检查连续成功，服务正常
- `unhealthy`：连续5次健康检查失败，服务异常

### 手动健康检查

不依赖 Docker，可以手动执行健康检查：

```bash
# 检查 Agent Card 端点
curl -sf http://localhost:8190/.well-known/agent-card.json > /dev/null
if [ $? -eq 0 ]; then
  echo "✅ 服务健康"
else
  echo "❌ 服务异常"
fi

# 带响应内容检查
curl -s http://localhost:8190/.well-known/agent-card.json | python -c "
import sys, json
try:
    data = json.load(sys.stdin)
    if 'agent' in data and data['agent'].get('id') == 'edp-agent':
        print('✅ 服务正常，Agent ID:', data['agent']['id'])
        sys.exit(0)
    else:
        print('❌ 返回格式异常')
        sys.exit(1)
except Exception as e:
    print('❌ 解析失败:', e)
    sys.exit(1)
"
```

## 日志体系

### 日志位置

- **Docker 部署**：通过 `docker logs` 查看（标准输出/标准错误）
- **本地启动**：日志输出到控制台，如配置了文件输出则在 `engine/logs/` 目录

### 日志格式

EDPAgent 使用 SLF4J + Logback 日志框架，标准日志格式：

```
2026-07-09 10:30:45.123  INFO 1 --- [           main] c.h.a.edp.EdpaApplication  : Started EdpaApplication in 12.345 seconds
2026-07-09 10:31:00.456  INFO 1 --- [ctor-http-nio-2] c.h.a.edp.rail.EdpaTodoRail : Todo created: task-xxx
2026-07-09 10:31:01.789  WARN 1 --- [ctor-http-nio-2] c.h.a.edp.rail.McpInterruptRail : MCP master connection failed, trying standby...
2026-07-09 10:31:02.012 ERROR 1 --- [ctor-http-nio-2] c.h.a.edp.rail.VersatileInterruptRail : Versatile call timeout
```

格式说明：
```
时间戳  级别  线程ID --- [ 线程名           ] 类名                        : 日志消息
```

### 日志级别配置

通过环境变量 `EDP_AGENT_LOG_LEVEL` 设置：

| 级别 | 启用的日志 | 适用场景 |
|------|------------|----------|
| `ERROR` | 仅错误 | 高并发生产环境 |
| `WARN` | 警告+错误 | 稳定生产环境 |
| `INFO` | 信息+警告+错误 | **推荐生产默认** |
| `DEBUG` | 所有级别 | 开发调试、问题排查 |

### 启动失败常见日志

**Redis 连接失败**：
```
ERROR c.h.a.edp.config.RedisConfig : Redis connection failed: Connection refused
```
👉 检查 Redis 是否启动、地址端口密码是否正确。

**配置解析错误**：
```
ERROR c.h.a.edp.config.GovernanceConfigLoader : Failed to parse planrule.yaml: ...
```
👉 检查 YAML 语法是否正确，缩进是否一致。

**API Key 无效**（启动时不会报错，请求时报错）：
```
ERROR c.h.a.edp.rail.... : Model API call failed: 401 Unauthorized
```
👉 检查 `EDP_AGENT_MODEL_API_KEY` 是否正确。

### 错误日志

**模型调用错误**：
```
ERROR c.h.a.edp.rail.... : Model API error: status=429, message=Rate limit exceeded
```
👉 API 限流，检查配额或增大重试间隔。

```
ERROR c.h.a.edp.rail.... : Model API timeout after 30s
```
👉 网络问题或模型服务过载。

**工具执行错误**：
```
ERROR c.h.a.edp.rail.McpInterruptRail : MCP call failed: Connection refused to http://mcp-server:8080
```
👉 MCP 服务不可达，检查 MCP 配置和服务状态。

```
ERROR c.h.a.edp.rail.VersatileInterruptRail : Versatile returned error: 500 Internal Server Error
```
👉 Versatile 服务端错误，查看 Versatile 日志。

**执行限制触发**：
```
WARN c.h.a.edp.rail.ExecutionLimitRail : Max steps reached (100/100), forcing task completion
```
👉 任务可能陷入死循环，检查 Governance 配置 max_steps 是否合理。

```
WARN c.h.a.edp.rail.ExecutionLimitRail : Tool call limit reached: call_versatile (50/50)
```
👉 工具调用次数达到上限。

## 日志分析技巧

### 按会话追踪

每个 SSE 请求对应一个会话（conversationId），追踪单一会话流程：

```bash
# 按 conversationId 过滤（需要DEBUG日志）
docker logs edp-agent 2>&1 | grep "conv-xxx"
```

### 统计错误数量

```bash
# 统计 ERROR 日志数量
docker logs edp-agent 2>&1 | grep -c "ERROR"

# 统计各类错误
docker logs edp-agent 2>&1 | grep "ERROR" | awk '{print $NF}' | sort | uniq -c | sort -rn
```

### 查找慢请求

```bash
# 查找耗时较长的模型调用（需要 DEBUG 日志）
docker logs edp-agent 2>&1 | grep "Model.*took" | sort -t'took' -k2 -rn | head 20
```

### 查看工具调用统计

```bash
# 统计各工具调用次数
docker logs edp-agent 2>&1 | grep "Tool called:" | awk -F'Tool called: ' '{print $2}' | awk -F',' '{print $1}' | sort | uniq -c | sort -rn
```

## 监控集成建议

### 简单监控脚本

```bash
#!/bin/bash
# health_check.sh - EDPAgent 健康检查脚本

ALERT_WEBHOOK="your-alert-webhook"
HEALTH_URL="http://localhost:8190/.well-known/agent-card.json"
LOG_FILE="/var/log/edp-agent-health.log"

check() {
    response=$(curl -sf -o /dev/null -w "%{http_code}" --max-time 10 $HEALTH_URL)
    if [ "$response" = "200" ]; then
        echo "[$(date)] OK: HTTP $response" >> $LOG_FILE
        return 0
    else
        echo "[$(date)] FAIL: HTTP $response" >> $LOG_FILE
        # 发送告警
        curl -s -X POST $ALERT_WEBHOOK -d "EDPAgent health check failed: HTTP $response"
        return 1
    fi
}

check
```

加入 crontab 定时执行：
```bash
# 每5分钟检查一次
*/5 * * * * /opt/edp-agent/health_check.sh
```

### 日志收集

生产环境建议使用 ELK（Elasticsearch + Logstash + Kibana）或 Loki + Grafana 收集日志：

1. 配置 Docker 日志驱动为 syslog 或 fluentd
2. 将日志收集到集中存储
3. 配置告警规则（如 5 分钟内 ERROR 日志超过 10 条触发告警）
4. 建立仪表盘展示请求量、错误率、响应时间等指标

### 关键告警指标

| 指标 | 告警阈值 | 说明 |
|------|----------|------|
| 健康检查失败 | 连续3次 | 服务不可用 |
| ERROR 日志突增 | 5分钟内 > 10条 | 可能有异常 |
| Redis 连接失败 | 任意1次 | 状态存储异常 |
| 模型 API 错误率 | > 10% | 影响用户体验 |
| 容器重启次数 | 1小时内 > 3次 | 服务不稳定 |

## 常见健康问题排查

### unhealthy 状态排查

1. **查看启动日志**：
```bash
docker logs edp-agent --tail 200
```

2. **手动执行健康检查**：
```bash
docker exec edp-agent curl -v http://localhost:8190/.well-known/agent-card.json
```

3. **检查端口监听**：
```bash
docker exec edp-agent netstat -tlnp | grep 8190
```

4. **检查 JVM 内存**：
```bash
docker exec edp-agent jstat -gc 1
```

### 日志中频繁出现 Redis 错误

1. 检查 Redis 服务是否正常运行
2. 检查网络连通性和防火墙
3. 检查 Redis 密码配置
4. 检查连接池配置（超时设置）
5. 查看 Redis 日志是否有慢查询或内存不足

### 大模型调用频繁超时

1. 检查网络带宽和延迟
2. 检查模型服务状态（可能是高峰期限流）
3. 适当增大超时时间（当前在代码中配置）
4. 考虑配置模型 API 备节点（如有）

### 内存使用持续增长

1. 检查是否有内存泄漏（长时间运行后 OOM）
2. 适当调整 JVM 堆内存参数
3. 定期重启（如每周一次滚动重启）
4. 检查 Redis 中是否有大量未过期的 key
