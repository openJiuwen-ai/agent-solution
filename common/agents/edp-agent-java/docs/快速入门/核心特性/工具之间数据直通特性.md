# 工具之间数据直通特性

## 特性概述

在 Agent 执行过程中，前序工具（如 call_mcp、call_versatile）的输出结果需要传递给后续工具使用。如果依赖 LLM 在 thought 中转述这些结构化数据，可能会出现数据丢失、格式错误或幻觉注入等问题。

EDPAgent 通过 **ToolDataChannel** 机制实现工具间数据直通——前序工具的执行结果自动写入数据通道，后续工具可通过 `input_key` 参数直接读取，无需经过 LLM 转述，保证数据准确性和完整性。

## 核心概念

### ToolDataChannel

ToolDataChannel 是一个会话级别的键值存储，用于在工具调用之间传递结构化数据。数据按作用域（Scope）隔离，生命周期与会话绑定。

### ToolDataScope

数据作用域分为三个级别：

| 作用域 | 说明 | 生命周期 |
|--------|------|----------|
| `SKILL` | Skill 级别，仅在当前 Skill 执行上下文中可见 | Skill 执行结束 |
| `SESSION` | 会话级别，同一 conversationId 下所有工具可见 | 会话结束 |
| `GLOBAL` | 全局级别，跨会话可见（当前版本未完全启用） | 服务重启 |

### ToolDataKey

数据键由三部分组成：`scope` + `toolName` + `key`，确保不同工具、不同作用域的数据互不干扰。

通过 ToolDataKeyFactory 可以创建标准格式的键：
- 自动前缀作用域标识
- 关联工具名称避免冲突
- 支持自定义业务键名

## 工作机制

### 1. 自动写入

MCP 和 Versatile 调用完成后，McpInterruptRail 和 VersatileInterruptRail 会自动将执行结果写入 ToolDataChannel，包括：
- MCP 脚本执行返回的结构化数据
- Versatile 工作流返回的业务结果
- 工具调用状态和元数据

### 2. 显式读取

后续工具调用时，通过 `input_key` 参数指定要读取的数据键，Rail 拦截器在执行前自动从通道中取出数据注入到工具参数中。

```
┌─────────────┐     ┌──────────────────┐     ┌─────────────┐
│ call_mcp    │────▶│ ToolDataChannel  │────▶│ call_versa- │
│ (执行脚本)   │写入 │  (会话级KV存储)   │读取 │ tile (使用)  │
└─────────────┘     └──────────────────┘     └─────────────┘
                           ↑
                           │ 自动写入/显式读取
                    ┌──────┴──────┐
                    │ 其他工具输出  │
                    └─────────────┘
```

## 使用方式

### 在 call_mcp 中写入数据

MCP 脚本执行完成后，返回的 JSON 结果会自动写入 ToolDataChannel，默认键格式为 `mcp_result_{script_command}`。

在 Skill 的 Python 脚本中，也可以显式写入数据到通道（通过返回特定格式）：

```python
# run_xxx.py 示例
def main(params):
    # 执行业务逻辑
    result = query_products(params)
    
    # 返回结果会自动写入 ToolDataChannel
    return {
        "products": result,
        "total_count": len(result),
        "_channel_key": "recommended_products"  # 指定写入的键名
    }
```

### 在 call_versatile 中读取数据

在 call_versatile 时通过 `input_key` 指定读取前序数据：

```json
{
  "name": "call_versatile",
  "arguments": {
    "query_description": "根据推荐结果进行产品购买",
    "query_intent": "执行理财产品购买流程",
    "input_key": "recommended_products",
    "response_template_keys": ["product_buy_success"]
  }
}
```

VersatileInterruptRail 会自动从 ToolDataChannel 读取 `recommended_products` 对应的数据，注入到 Versatile 请求中。

### 在 Skill 间传递数据

Skill 执行时也可以访问 ToolDataChannel 中的数据，实现跨 Skill 数据传递：

1. product_recommend_skill 执行后，推荐结果写入通道
2. product_select_skill 触发时，通过 `input_key` 读取推荐结果
3. 无需用户重复输入或 LLM 重复生成

## 使用示例

### 场景：理财推荐→购买流程

**步骤1：产品推荐（call_mcp + Skill 话术）**

Agent 调用 call_mcp 执行推荐脚本，返回产品列表后自动写入 ToolDataChannel：

```
ToolDataChannel["SESSION"]["mcp_result_recommend_products"] = {
  "products": [
    {"id": "P001", "name": "稳健理财A", "rate": "3.2%"},
    {"id": "P002", "name": "增值理财B", "rate": "4.1%"}
  ]
}
```

**步骤2：用户选择产品（ask_user）**

Agent 调用 ask_user 询问用户购买哪个产品。

**步骤3：执行购买（call_versatile）**

Agent 调用 call_versatile，通过 input_key 读取推荐数据：

```json
{
  "name": "call_versatile",
  "arguments": {
    "query_description": "执行产品购买",
    "query_intent": "根据用户选择的产品ID执行购买",
    "input_key": "mcp_result_recommend_products"
  }
}
```

VersatileInterruptRail 自动将产品列表数据注入到 Versatile 请求中，购买工作流可以直接获取到完整的产品信息。

## 配置项

ToolDataChannel 本身无需额外配置，数据存储在内存中（会话级别）。Redis 中持久化的是 Todo 和 Checkpoint 数据，ToolDataChannel 数据不持久化。

## 注意事项

1. **数据生命周期**：ToolDataChannel 数据是会话级内存存储，服务重启或会话过期后数据丢失。如需持久化，请通过 Versatile 或业务系统存储。

2. **数据大小限制**：单条数据建议不超过 100KB，过大的数据传递会增加内存占用和序列化开销。

3. **键名规范**：业务键名使用小写字母+下划线，建议包含业务前缀（如 `recommend_products`、`balance_query_result`）避免冲突。

4. **空值处理**：如果 `input_key` 对应的数据不存在，Rail 不会报错，而是正常执行工具（但数据字段为空）。业务脚本需要做好空值判断。

5. **并发场景**：同一迭代中多个工具写入同一键名时，后写入的值会覆盖先写入的值。建议不同工具使用不同键名。

6. **与 LLM 的关系**：ToolDataChannel 是给工具/Rail 用的，LLM 本身不能直接读取通道内容。LLM 看到的是工具返回的 result 字段（如果 Rail 配置了返回给 LLM）。
