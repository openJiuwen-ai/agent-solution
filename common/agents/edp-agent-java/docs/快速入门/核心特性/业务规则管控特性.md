# 业务规则管控特性

## 特性概述

业务规则管控特性通过多层治理机制约束Agent的行为边界，确保Agent在授权范围内安全、可控地执行业务。该特性包括业务范围控制（scope）、工具白名单（allowed_tools）、工具调用次数限制（tool_limits）、任务循环控制（enable_task_loop）和执行步数限制（ExecutionLimitRail），以及话术控制，共同形成全方位的行为治理体系。

核心能力包括：
- **scope业务范围控制**：通过allowed/denied列表定义允许和禁止的业务范围
- **allowed_tools工具白名单**：精确控制Agent可调用的工具集合
- **tool_limits调用次数上限**：防止单个工具被无限循环调用
- **enable_task_loop任务循环控制**：控制是否允许任务循环执行
- **ExecutionLimitRail步数限制**：强制限制工具调用次数，防止失控

## 配置方式

### 1. 业务范围配置（planrule.yaml）

在`planrule.yaml`中配置scope的allowed和denied列表：

```yaml
planrule:
  # Agent职责边界
  scope:
    # 允许的业务范围列表（逗号分隔的关键词或正则）
    allowed: "理财推荐,余额查询,转账,产品购买"
    
    # 禁止的业务范围列表（逗号分隔的关键词或正则）
    denied: "股票交易,期货,外汇,贷款"
```

**工作模式**：
- 当scope已配置业务范围（allowed或denied非空）时：业务范围内的请求必须走「todo_create → call_versatile → todo_modify → final_answer」标准流程
- 当scope未配置业务范围（allowed和denied均为空）时：先判断请求是否匹配Skill Routing中的任意Skill，匹配则走标准流程；若不匹配且无法提供有效帮助，直接说明无可用业务能力配置，禁止强行创建空任务

### 2. 工具白名单配置（actrule.yaml）

在`actrule.yaml`的`allowed_tools`中配置允许调用的工具列表：

```yaml
actrule:
  # 允许调用的工具列表（白名单机制）
  allowed_tools:
    - bash
    - skill_tool
    - call_versatile
    - call_mcp
    - ask_user
    - todo_create
    - todo_modify
    - todo_list
    - todo_get
    - cancel_task
```

只有在此列表中的工具才会被注册到Agent的工具注册表中，不在列表中的工具无法被LLM调用。

### 3. 工具调用次数限制配置（actrule.yaml）

在`actrule.yaml`的`tool_limits`中配置单个工具的调用次数上限：

```yaml
actrule:
  # 单个工具调用次数上限（key: 工具名, value: 上限值）
  tool_limits:
    call_versatile: 50
    call_mcp: 50
    ask_user: 50
    execute_cmd: 50
```

未配置的工具默认上限为100次。

### 4. 任务循环和步数配置（actrule.yaml）

```yaml
actrule:
  # 限制单层最大子任务数量
  max_subtasks: 50
  
  # 限制最大执行步数
  max_steps: 100
  
  # 是否启用任务循环
  enable_task_loop: true
  
  # Skill加载模式
  # - all: 列出所有Skill，Agent直接判断匹配
  # - auto_list: 动态搜索Skill，Agent先调用list_tool查询
  # - none: 关闭Skill系统
  skill_mode: all
```

## 使用示例

### 1. 场景级业务范围配置示例

以hz-zhidaitong场景为例，在场景的governance/planrule.yaml中配置特定业务范围：

```yaml
planrule:
  role: 你的身份是通用动态规划智能体
  description: "你负责智贷通业务场景，包括贷款申请、额度查询、还款计划等。"
  scope:
    allowed: "贷款申请,额度查询,还款计划,利率咨询,贷款进度"
    denied: "理财推荐,基金购买,转账汇款"
  supplementary_prompt:
    additional_prompt: |
      你是智贷通业务助手，只处理贷款相关业务。
      对于理财、转账等非贷款业务，直接告知用户不支持。
```

### 2. 工具白名单配置示例（受限场景）

对于安全性要求高的场景，可以收紧工具白名单：

```yaml
actrule:
  allowed_tools:
    - call_versatile
    - ask_user
    - todo_create
    - todo_modify
    - todo_list
    - todo_get
    - cancel_task
  # 移除bash、call_mcp等高风险工具
  tool_limits:
    call_versatile: 10  # 收紧调用次数上限
    ask_user: 5
```

### 3. 触发工具限制的场景

当工具调用次数达到上限时，ExecutionLimitRail会强制结束：

```
# 日志输出
ExecutionLimitRail: tool call_versatile call count 51 > limit 50, forcing finish

# Agent收到强制结束指令
{
  "message": "工具调用次数已达上限"
}
```

### 4. 超范围请求处理示例

当用户请求超出scope配置的业务范围时：

```
用户：帮我推荐几只股票
Agent：正在学习中，暂不支持该业务。
（不创建todo，不调用任何业务工具，直接返回out_of_scope话术）
```

## 关键实现机制

### 1. Scope业务范围控制

scope配置通过prompt注入引导LLM判断业务范围：

**planrule.yaml中的base_protocol明确规定**：
- 有scope配置时：范围内请求必须走标准todo流程
- 无scope配置时：先匹配Skill，无匹配则直接说明不支持，禁止创建空任务

**out_of_scope话术处理**：
- ScriptsRail.complianceGate()进行合规把关
- 配置外话术自动替换为out_of_scope（"正在学习中，暂不支持该业务。"）

代码位置：
- 配置定义：`planrule.yaml:17-22`
- 协议注入：`planrule.yaml:33-38, 97-98`
- 合规把关：`ScriptsRail.java:253-261`

### 2. allowed_tools工具白名单

工具白名单在工具注册阶段生效：

**EdpaToolRegistry**根据actrule.allowed_tools过滤可用工具：
- 只有在白名单中的工具才会被注册
- 不在白名单中的工具对LLM不可见，无法被调用

**默认工具集**（actrule.yaml:25-35）：
- 业务工具：bash、skill_tool、call_versatile、call_mcp
- 交互工具：ask_user、cancel_task
- Todo工具：todo_create、todo_modify、todo_list、todo_get

代码位置：`actrule.yaml:24-35`

### 3. tool_limits工具调用次数限制

ExecutionLimitRail实现了会话级别的工具调用计数：

**实现机制**：
1. 使用`ConcurrentHashMap<sessionId, Map<toolName, count>>`维护计数器
2. beforeToolCall时以sessionId隔离，按工具名累加调用次数
3. 超过配置上限时调用`ctx.requestForceFinish()`强制结束
4. 未配置的工具默认上限为100次

**代码逻辑**（ExecutionLimitRail.java:60-80）：
```java
@Override
public void beforeToolCall(AgentCallbackContext ctx) {
    if (!(ctx.getInputs() instanceof ToolCallInputs inputs)) {
        return;
    }
    String toolName = inputs.getToolName();
    int limit = getToolLimit(toolName);  // 优先读配置，默认100
    
    String sessionId = ctx.getSession() != null ? ctx.getSession().getSessionId() : "_default";
    Map<String, Integer> sessionCounts = toolCallCounts.computeIfAbsent(sessionId, k -> new ConcurrentHashMap<>());
    int currentCount = sessionCounts.merge(toolName, 1, Integer::sum);
    
    if (currentCount > limit) {
        LOGGER.info("ExecutionLimitRail: tool {} call count {} > limit {}, forcing finish",
                toolName, currentCount, limit);
        ctx.requestForceFinish(Map.of("message", "工具调用次数已达上限"));
    }
}
```

代码位置：`ExecutionLimitRail.java:29-98`

### 4. max_subtasks子任务数量限制

EdpaTodoRail在todo_create前校验子任务数量：

**实现机制**：
1. beforeToolCall时解析todo_create的tasks参数
2. 统计任务数量，与actrule.max_subtasks比较
3. 超过上限时设置_skip_tool=true，返回合成错误结果，不执行真实的todo_create

**错误返回**：
```json
{
  "error": "MAX_SUBTASKS_EXCEEDED",
  "message": "子任务数量 51 超过上限 50，请精简任务列表后重试。"
}
```

代码位置：`EdpaTodoRail.java:213-229`

### 5. enable_task_loop任务循环控制

enable_task_loop控制是否允许Agent在任务完成后重新规划循环：

- `enable_task_loop: true`（默认）：允许Agent根据执行结果动态调整和新增任务
- `enable_task_loop: false`：任务列表一次性创建完成后不允许新增，只能按原计划执行或取消

### 6. 规划前置守卫（Plan-First Guard）

EdpaTodoRail.enforcePlanBeforeBusinessTool()实现了强制规划机制，根治"LLM跳过规划直接调工具"的问题：

**机制**：
1. 业务工具（call_mcp/call_versatile）调用前检查是否已规划todo
2. 未规划时：设置_skip_tool=true和_plan_first_block=true
3. 返回PLAN_FIRST合成错误结果，跳过真实工具执行
4. 通过pushSteering强制LLM先调用todo_create规划

**错误返回**：
```json
{
  "error": "PLAN_FIRST",
  "message": "BLOCKED: 业务工具 call_versatile 被拦截。你必须先调用 todo_create 按 catalog_id 创建任务列表，规划完整执行步骤后，才能调用业务工具。"
}
```

**与tool_start/tool_end的配合**：
- 被PLAN_FIRST拦截的工具不发射tool_start/tool_end事件（EdpaEventRail检查_plan_first_block标记）

代码位置：
- 守卫实现：`EdpaTodoRail.java:279-302`
- 事件配合：`EdpaEventRail.java:299-309, 340-344`

## 注意事项

1. **白名单机制是硬约束**：不在allowed_tools中的工具完全不会被注册，LLM无法看到也无法调用，这是比prompt更可靠的安全机制。

2. **会话隔离计数**：tool_limits的计数器以sessionId为key隔离，不同A2A请求不会互相影响计数。

3. **超限强制结束**：工具调用次数达到上限时调用requestForceFinish()，会立即终止当前会话，Agent无法绕过。

4. **scope通过prompt引导而非硬拦截**：当前scope配置主要通过prompt注入引导LLM判断，并非代码层面硬拦截；超范围请求的拦截依赖LLM遵循指令和out_of_scope话术兜底。

5. **规划前置是硬拦截**：业务工具执行前必须有todo，这个是EdpaTodoRail代码层面硬拦截，不依赖LLM自觉性，根治跳过规划问题。

6. **max_steps框架级限制**：max_steps对应框架的maxIterations配置，在框架层面限制最大迭代轮次，防止无限循环。

7. **skill_mode配置**：
   - all模式：所有Skill都列在prompt中，适合Skill数量少的场景
   - auto_list模式：需要Agent主动调用list_tool查询，适合Skill数量多的场景
   - none模式：关闭Skill系统，只使用基础工具

8. **默认限制值**：未在tool_limits中配置的工具默认上限是100次，max_subtasks默认50，max_steps默认100，请根据业务场景调整。
