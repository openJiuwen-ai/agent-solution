# 产品介绍

---

## 1. 产品定位

### 1.1 是什么

Adapter-Versatile-Agent 是一个**独立部署的 A2A（Agent-to-Agent）代理服务**，基于 Spring Boot 4.0.6 + JDK 17+ 构建，核心依赖 agent-runtime-java 框架。它将 EDPAgent 等 A2A 客户端发来的请求，适配转换为 Versatile REST/SSE 工作流调用，并对 Versatile 的流式响应进行智能节点分流处理。

本服务本质是一个**协议适配器 + 响应路由器**，专注于解决 A2A 协议与 Versatile 工作流系统之间的对接问题。

### 1.2 解决什么问题

EDPAgent 通过 A2A 协议调用业务工作流时，面临以下适配问题：

| 问题 | Adapter-Versatile-Agent 的解决方案 |
|------|-----------------------------------|
| **协议差异**：A2A 使用 JSON-RPC over SSE，Versatile 使用 HTTP/SSE REST API | 双向协议转换，将 A2A `sendStreamingMessage` 映射为 Versatile HTTP 请求 |
| **响应类型多样**：Versatile 工作流包含菜单、表单、提示、结果等多种节点类型 | 节点类型智能分流：交互节点给用户、结果节点给 LLM、待输入触发中断 |
| **流式交互**：工作流执行过程中需要实时推送中间状态 | SSE 流式透传，确保用户及时看到菜单、表单等交互内容 |
| **Header 透传**：需要透传调用模式、语言等上下文信息 | Header 白名单机制，安全转发 `x-invoke-mode`、`x-language` 等头 |

## 2. 核心能力

### 2.1 A2A 协议端点

本服务提供标准的 A2A 协议端点：

| 端点 | 方法 | 说明 |
|------|------|------|
| `/.well-known/agent-card.json` | GET | Agent Card 发现端点，返回 Agent 元信息、能力列表、Skills |
| `/a2a` | POST (SSE) | `sendStreamingMessage` 流式消息端点，处理 A2A 请求并返回 SSE 流 |

**Agent Card 默认配置**（定义于 [application.yml:11-16](file:///D:/AgentTool/EDPAgent_java/agent-solution/common/agent/adapter-versatile-agent-java/src/main/resources/application.yml#L11-L16)）：
- Agent ID: `versatile-agent`
- 描述: "Remote runtime that processes banking and external business workflows through Versatile HTTP/SSE."
- 默认 Skill: `banking_workflow`，处理账户查询、转账准备等银行业务工作流

### 2.2 Versatile HTTP/SSE 适配

核心适配能力由 agent-runtime-java 的 `VersatileAgentHandler` 提供：

- **URL 模板占位符替换**：支持 `{conversation_id}`、`{workflow_id}` 等占位符自动替换
  - 默认 URL 模板：`http://localhost:30001/v1/0/agent-manager/workflows/mock_workflow/conversations/{conversation_id}?type=controller&workspace_id=10`
- **Header 白名单透传**：仅转发白名单内的 Header，保证安全
  - 默认白名单：`stream`、`x-invoke-mode`、`x-language`
- **超时控制**：默认超时 600 秒（10分钟），适应长工作流场景
- **默认 Header 模板**：
  ```yaml
  Content-Type: application/json
  accept: "*/*"
  stream: "true"
  x-invoke-mode: DEBUG
  x-language: zh-cn
  ```

### 2.3 节点类型分流（核心逻辑）

这是本服务最核心的能力，复用 agent-runtime-java 的 `VersatileAgentHandler`（配置于 [VersatileAgentConfiguration.java:32-34](file:///D:/AgentTool/EDPAgent_java/agent-solution/common/agent/adapter-versatile-agent-java/src/main/java/com/huawei/ascend/versatile/VersatileAgentConfiguration.java#L32-L34)），按节点类型（`node_type` + `node_name`）区分处理：

| 节点类型 | 处理目标 | 说明 |
|----------|----------|------|
| **非结果节点** | `Target.USER` | 菜单（menu）、提示（notice）、表单（form）等交互节点，完整 JSON 流式透传给前端用户展示 |
| **结果节点** | `Target.LLM` | 节点类型为 `QA` 且节点名匹配 `GXZQAResponseNode`（可通过 `VERSATILE_RESULT_NODE` 配置），提取结果内容作为工具调用结果返回给上游 LLM |
| **缺失 End 节点 / 需要用户输入** | `Interrupt` | 工作流暂停等待用户输入时，触发 `ToolInterruptException` 中断，保存上下文等待用户输入后续传 |

**结果节点匹配规则**：只有 `node_type` 和 `node_name` 同时命中时，才会作为结果回灌上游 Agent，其余 message 节点完整 JSON 透传前端。

> 💡 **默认结果节点配置**：`GXZQAResponseNode`（node_type=QA, node_name=GXZQAResponseNode），对应 mock 服务 versatile_main.py 的结果帧格式。

### 2.4 Header 透传机制

采用**白名单机制**安全转发上游请求头到 Versatile 服务：

配置位置：[application.yml:20-33](file:///D:/AgentTool/EDPAgent_java/agent-solution/common/agent/adapter-versatile-agent-java/src/main/resources/application.yml#L20-L33)

```yaml
openjiuwen:
  service:
    versatile:
      # 上游 a2a metadata 允许透传到 REST 请求头的白名单
      passthrough-headers:
        - x-invoke-mode
        - x-language
      # 从 a2a metadata 合并进 REST body inputs 的字段
      input-metadata-keys:
        - intent
        - wap_userName
```

- `passthrough-headers`：指定哪些 Header 从 A2A 请求透传到 Versatile REST 请求
- `input-metadata-keys`：指定哪些 A2A metadata 字段合并到 Versatile 请求 body 的 inputs 中

### 2.5 多意图路由（endpoints 配置）

> ⚠️ **注意**：当前版本基于 `VersatileAgentHandler` 单实例配置，支持通过 `endpoints` 配置按 `intent` 分发到不同 Versatile URL（agent-runtime-java 框架能力）。

通过 A2A 请求 metadata 中的 `intent` 字段，可以路由到不同的 Versatile 工作流端点。

## 4. 与 EDPAgent 的关系

Adapter-Versatile-Agent 是 EDPAgent 生态系统中的**专用适配器**，与 EDPAgent 的分工如下：

| 维度 | EDPAgent | Adapter-Versatile-Agent |
|------|----------|-------------------------|
| **核心职责** | ReAct 推理规划、Skill 路由、多工具编排 | Versatile 协议适配、节点分流处理 |
| **协议入口** | 接收用户对话、多渠道接入 | 接收 A2A 请求、暴露 Agent Card |
| **工作流处理** | 决策调用哪个工具/Skill | 专注调用 Versatile，处理 SSE 流分流 |
| **交互对象** | 终端用户 + LLM | EDPAgent (上游) + Versatile (下游) + 前端用户 |

### 对接方式

EDPAgent 通过以下方式对接本服务：

1. **直接 A2A 调用**：将本服务作为标准 A2A Agent 调用，通过 A2A 协议通信
2. **工具调用**：通过 `call_versatile` 工具间接调用，EDPAgent 配置 `EDP_AGENT_VERSATILE_A2A_URL` 指向本服务地址

**典型调用链路**：
```
用户 → EDPAgent (ReAct规划) → 选择 versatile-agent Skill → A2A调用本服务
→ 本服务调用 Versatile → 分流响应：交互节点给前端 / 结果返回EDPAgent
→ EDPAgent 继续处理 → 回复用户
```

## 5. 适用场景

### 5.1 银行理财业务工作流代理

典型场景：
- 账户余额查询
- 转账准备与确认
- 理财产品购买流程（风险测评 → 产品选择 → 确认 → 结果）
- 客户信息查询与补全

这些场景通常包含多轮交互（菜单选择、表单填写、确认提示），需要用户输入后才能继续执行，正好匹配本服务的三分流能力。

### 5.2 需要用户交互的业务流程

任何包含以下特征的业务流程：
- ✅ 流程中需要向用户展示菜单/选项
- ✅ 需要用户填写表单信息
- ✅ 需要用户确认操作
- ✅ 流程有明确的最终结果返回给上层 Agent

### 5.3 Versatile 工作流系统的 A2A 化接入

如果你的业务系统已经在使用 Versatile 工作流引擎，并且需要：
- 接入 EDPAgent 等 A2A 生态
- 让 LLM 能够编排调用业务工作流
- 保留原有 Versatile 工作流的交互能力

那么部署本服务作为 A2A 代理是最快的接入方式，无需改造现有 Versatile 工作流。
