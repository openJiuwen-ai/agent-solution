EvoAgent · TF-GRPO Test Environment

TF-GRPO 测试环境部署指南

面向测试人员：在 clone 代码后，从零搭起一套可提交 tf_grpo 优化任务的联调环境。 交付验收以 EvoAgent 服务化接口（FastAPI POST /optimize）为主路径。

场景 tf_grpo 主入口 POST /optimize 依赖 Adapter + 业务 Agent + LLM
架构依赖 前置条件 端口与拓扑 环境变量 启动顺序 EDPAgent 示例 数据集准备 验收检查 冒烟与最小任务 常见问题

1. 需要依赖哪些系统

TF-GRPO 不是单进程算法：优化服务改写 Skill，必须经 Adapter 热更到业务 Agent 侧，并真实对话打分。

系统职责仓库位置（通用）
EvoAgent API 优化编排服务：接单、跑 TF-GRPO、写 artifacts、SSE 进度 common/agent-evolve/evoagent/
EvoAgentAdapter Sidecar：代理对话、轨迹采集、Skill 热更（常对接沙箱文件 API） common/agent-evolve/evoagent-adapter/
业务 Agent 实际执行 Skill（如读 SKILL.md、调脚本）；Adapter 配置中的 agent_name 按项目提供（示例拓扑见 Adapter sandbox 实验目录）
运行时沙箱（若启用） Skill 真源目录（如 /tmp/skills）；须与业务 Agent 使用同一 sandbox 随业务 Agent / jiuwenbox 类组件部署
LLM 服务 变体生成、rollout 摘要、语义优势、经验库更新（OpenAI 兼容或内网 Provider） 外部服务；密钥只放本地 .env（勿提交）
测试人员 / 平台 │ POST /optimize (scenario = tf_grpo) ▼ EvoAgent API ──EVO_ADAPTER_URL──► Adapter Sidecar │ │ │ ├──► 业务 Agent（对话 / 工具） │ └──► 沙箱 Skill 目录（热更真源） └──► LLM API（优化器模型）
仓库内场景绑定：common/agent-evolve/evoagent/scenarios/tf_grpo/（含 scenario.yaml）。默认示例 skill 名为 audit-business、agent_name 需与 Adapter 注册名一致（常见为 edp_agent）。

2. 前置条件

  • 代码：已 clone 含 common/agent-evolve/evoagent 与 evoagent-adapter 的仓库分支。
  • 运行时：Docker（推荐完整联调）或本机 Python 3.12+ + 项目依赖（见 evoagent Makefile / README）。
  • 网络：EvoAgent 能访问 Adapter；Adapter 能访问业务 Agent 与沙箱；EvoAgent 能访问 LLM。
  • 密钥：有效的 LLM API Key（或内网 Provider 凭证）；写入本地 env，不要提交到 git。
  • 数据：自备 cases 文件（JSON）；API 要求路径落在白名单根目录下（见第 5 节）。

下表为常见联调约定，实际以你们 compose / 运维分配为准。容器互通时请用服务名，不要用容器内的 localhost 指另一服务。

4. 环境变量与配置注意事项

4.1 EvoAgent（必填）

从模板复制本地配置（模板在仓库内；复制后的文件通常不入库）：

cd common/agent-evolve/evoagent/deployment
cp config/.env.example config/.env
# 编辑 config/.env 填入真实值
变量作用注意
EVO_ADAPTER_URL Adapter Sidecar 根地址 未配置时 POST /optimize 直接 500
EVO_LLM_API_KEY 优化器 LLM 密钥 勿提交；勿写进报告 HTML
EVO_LLM_BASE_URL OpenAI 兼容 Base URL 与 Provider 文档一致
EVO_OPTIMIZER_MODEL / EVO_TARGET_MODEL 优化侧 / 目标侧模型名 需账号有权限
EVO_ALLOWED_DATA_ROOTS API dataset_path 允许的根路径列表（逗号分隔） 路径不在白名单 → 422
EVO_REMOTE_TIMEOUT 远程调用超时（秒） 长对话 / 慢 Agent 建议 ≥ 300
完整模板字段见 deployment/config/.env.example。若使用内网 ICBC Provider，按模板启用 EVO_ICBC_*（与 OpenAI 模式二选一）。

4.2 场景默认超参（可被 API 覆盖）

文件：scenarios/tf_grpo/scenario.yaml。常用字段：

字段含义测试建议
group_size每轮变体数冒烟可改为 1（无组内对比，会跳过经验库更新）
cases_per_variant每轮共用 train case 数冒烟可改为 1
num_epochs训练轮数冒烟 1；完整回归再加大
num_parallelrollout 并发环境弱时降为 1～2
skills待优化 skill 列表须已部署到业务 Agent / 沙箱
adapter_url场景内默认 Adapter运行时以进程环境 EVO_ADAPTER_URL / 请求装配为准，避免指错环境

4.3 Adapter 侧关键项

  • agent 注册名须与请求里 agent_name 一致。
  • 沙箱模式下确认 skill_backend、jiuwenbox_url、remote_skills_dir、sandbox_id_resolve（推荐 from_logs；多 ready 沙箱时勿用盲目 list_ready）。
  • 日志采集：见第 6.6 节（路径映射与 log_pattern 一致性）。

5. 建议启动顺序

  1. 创建/确认 Docker 网络（多容器须在同一网络）。
  2. 启动沙箱组件（若拓扑需要）→ 健康检查通过。
  3. 启动业务 Agent → 确认已加载目标 Skill，并记录 / 解析 sandbox_id（若适用）。
  4. 启动Adapter → 配置指向业务 Agent 与沙箱；/health 200。
  5. 配置 EvoAgent deployment/config/.env（至少 Adapter URL + LLM）。
  6. 启动 EvoAgent：
    • Docker：在 evoagent/deployment 按 operations-guide.md 执行 build.sh / run.sh；或
    • 本地：安装依赖后 uvicorn evo_agent.api.app:app --host 0.0.0.0 --port 8000（须能加载上述 env）。
  7. 执行第 8 节验收检查，再提交优化任务。
启动 EvoAgent 时工作目录 / --env-file 必须能读到含 EVO_ADAPTER_URL 的配置；否则会出现「健康检查正常，但 POST /optimize 500」。

6. 以 EDPAgent 为例：部署注意事项

下面以常见联调组合 EDPAgent（sandbox）+ 沙箱 File API + Adapter（注册名 edp_agent）+ EvoAgent tf_grpo 说明。其他业务 Agent 可对照同一检查清单迁移。

6.1 名称必须对齐的三处

位置字段 / 约定示例（可改，但三处一致）
Adapter 配置 agents[].name、agent_url edp_agent → 指向 EDP 服务地址
EvoAgent 请求 / 场景 agent_name；scenarios/tf_grpo/scenario.yaml 的 skills agent_name=edp_agent；skill audit-business
EDPAgent 规则与场景 激活的业务场景、todolist / 路由绑定的 skill 名 场景内 skill 名 = audit-business（与热更目录名一致）
任一处拼写不一致，会出现「能对话但从不读目标 Skill」「热更成功但 Agent 仍走旧场景」等假阳性。

6.2 AgentRule（根规则）

EDPAgent 根目录 AgentRule.md：YAML frontmatter → 运行配置；Markdown body → 注入系统提示。测 TF-GRPO（业扩审核示例）时重点检查：

配置项建议说明
scenario_discovery.base_path skills/scenarios（相对 EDPAgent 根） 场景文件目录；构建镜像时须打进包
scenario_discovery.active_scenario 指向审核场景名，例如 AgentRule_audit_business 也可用环境变量覆盖（若工程支持 ACTIVE_SCENARIO）；勿仍指向理财等无关场景
scope 允许「办理时限 / 业扩审核」类意图 过窄会直接拒答，优化 rollout 全失败
planning_steps / 正文约束 要求先读 Skill、按契约输出（如 <answer>JSON</answer>） 与 dataset evaluator 字段约定一致
todolist_steps（若启用） skill 字段与真实 skill 目录名一致 占位 skill 名会导致运行期快速失败或空跑
允许的工具 至少覆盖读 Skill / 执行条款脚本所需工具（如读文件、执行命令） 工具被禁则无法按 SKILL.md 流程打分

6.3 Scenario 场景文件

  • 在 skills/scenarios/ 下提供与 active_scenario 同名的场景 Markdown（如 AgentRule_audit_business.md）。
  • 场景 frontmatter 通常包含：scope、todolist_steps、skill_routing（trigger → skill）、可选 tools。
  • 路由 / todolist 绑定的 skill 必须等于沙箱中的目录名（示例 audit-business），并与 EvoAgent tf_grpo 场景 skills 列表一致。
  • 部分工程可能对 skills/*/ 做 ignore：测试镜像构建时请确认场景文件与 skill 包已打入镜像或挂载进容器，不要假设「clone 后磁盘上一定有」。

6.4 Skill 包（以 audit-business 为例）

  • 目录结构需满足 Agent 与 Adapter 约定，常见为沙箱内 /tmp/skills/<skill名>/SKILL.md（及条款文档、脚本等）。
  • 首次部署：随 EDPAgent 初始化上传 / 解压到沙箱；之后由 Adapter update_skill 热更同一路径。
  • EvoAgent 优化前会 restore_skill：依赖 Adapter 能读到可恢复的初始内容（本地 meta 快照或远端可读）。
  • 输出契约字段（如 responsibility）须与测试集期望、评估抽取策略一致。

6.5 沙箱与环境变量（EDP 侧）

项注意
SANDBOX_URL指向沙箱服务；容器间用服务名或 host.docker.internal，避免错误 localhost
Skill 落盘根与 Adapter remote_skills_dir 一致（常见 /tmp/skills）
sandbox_idAdapter 与 EDP 必须同一沙箱；推荐 Adapter sandbox_id_resolve: from_logs，多 ready 沙箱时慎用盲目 list_ready
网络EDP、沙箱、Adapter 同一 Docker network（参见 Adapter deployment/sandbox-experiment/README.md）

6.6 日志路径映射与读取一致性（轨迹采集）

TF-GRPO 打分依赖 Adapter 从业务 Agent 进程日志采集轨迹。日志是「Agent 写、Adapter 读」的另一条链路，与 Skill 沙箱热更相互独立，但都必须对齐。

EDPAgent 容器 写日志 ──► <宿主机日志目录> ──只读挂载──► Adapter 容器内 log_dir （如 /data/logs/edp_agent） log_pattern 匹配 process*.log → 解析 traces → EvoAgent 评估
检查项要求
挂载同一目录 宿主机上 EDP 日志目录 = Adapter compose/卷挂载源；容器内路径写在 Adapter agents[].log_dir
log_pattern 与真实文件名一致（如 process.log / process_N.log → 常用 process*.log）；匹配不到则无 traces
权限 Adapter 对日志目录只读即可；目录空或权限拒绝会导致空提取与重试风暴
轮询 关注 Adapter poll_interval；对话刚结束立即取 trace 可能尚未落盘
与 Skill 路径解耦 改 Skill 热更不会自动修好日志映射；两条线分别验收
联调时可先单独打一轮 Adapter 对话，确认 traces 接口或输出目录有新记录，再跑优化任务。

6.7 EDP 侧上线前检查清单

  1. AgentRule.md 的 active_scenario 已切到审核场景（非理财等默认场景）。
  2. 对应 skills/scenarios/*.md 存在于运行镜像/挂载中，且 skill 路由指向目标 skill。
  3. 沙箱中存在 /tmp/skills/<skill>/SKILL.md，Adapter skill_content 可读、可 update_skill。
  4. 日志目录已挂到 Adapter，log_pattern 能匹配到最新日志文件。
  5. Adapter 注册名 = EvoAgent 请求 agent_name；EvoAgent tf_grpo 的 skills 列表 = 实际 skill 名。
  6. 手工问一句业务题：Agent 会 read_file Skill 并按契约输出，而非闲聊拒答。

7. 数据集如何准备（不依赖仓库内样本）

workspace/ 与本地数据集通常不随仓库分发。测试人员需自备 cases 文件。

项要求
格式JSON 数组；含用例 id、输入消息、期望标签等（与项目 Case 约定一致）
规模划分后 train、val 至少各 1 条 → 物理上最少 2 条；完整回归再加大
路径绝对路径，且落在 EVO_ALLOWED_DATA_ROOTS 某根之下
切分API 请求 train_split + val_split = 1.0，且均 > 0
评估器服务化 API 默认 type=metric（默认 exact_match，可配 extract）；语义评估须显式 type=llm + prompt。CLI 则在 dataset.yaml 声明 evaluator

9. 冒烟：提交 TF-GRPO 任务

9.1 接单后取消（验证链路，不跑满）

curl -s -X POST http://<evoagent-host>:8000/optimize \
  -H "Content-Type: application/json" \
  -d '{
    "task_name": "tfgrpo-smoke-cancel",
    "agent_name": "<adapter注册名>",
    "optimizer_type": "skill",
    "skills": ["<skill名>"],
    "dataset_path": "<白名单内绝对路径/cases.json>",
    "optimizer_template": {
      "name": "tf_grpo",
      "scenario": "tf_grpo",
      "hyperparams": {
        "num_epochs": 1,
        "group_size": 1,
        "cases_per_variant": 1,
        "num_parallel": 1
      },
      "train_split": 0.5,
      "val_split": 0.5
    },
    "evaluator_template": {
      "name": "exact",
      "scenario": "tf_grpo",
      "type": "metric",
      "metric": "exact_match"
    }
  }'
# 期望: {"job_id":"...","status":"queued",...}

curl -s -X POST http://<evoagent-host>:8000/optimize/<job_id>/cancel
curl -s http://<evoagent-host>:8000/optimize/<job_id>

9.2 最小全流程（跑完 1 轮）

在 9.1 请求体基础上不要 cancel；轮询至 completed / failed：

curl -s http://<evoagent-host>:8000/optimize/<job_id>
# 或 SSE:
# curl -N http://<evoagent-host>:8000/optimize/<job_id>/stream
  • group_size=1 时组内无分数方差，跳过经验库更新属预期，仍应完成基线 → 变体 → rollout → val 门控。
  • 产物默认写在 EvoAgent 配置的 artifact 目录（容器内常挂载到部署目录下的 workspace）；以 Job 返回与服务日志为准。

10. 常见问题

现象可能原因处理
POST /optimize 500，提示 Adapter 未配置 未加载 EVO_ADAPTER_URL 检查 deployment/config/.env 与启动方式
422 dataset path 路径不在白名单 / 文件不存在 调整 EVO_ALLOWED_DATA_ROOTS 或移动数据
422 train/val split 两比例之和 ≠ 1 或一侧为 0 例如 0.5 / 0.5；且总 case ≥ 2
热更后分数不变 / 读旧 Skill sandbox_id 不一致；skill 路径不一致 核对 Adapter 与业务 Agent 沙箱；看 Adapter 日志
大量空提取 / 重试 轨迹采集失败或输出格式不符 检查第 6.6 节日志挂载与 log_pattern；核对 Agent 输出契约与超时
Agent 拒答 / 不读 Skill active_scenario 错误；场景未打进镜像；scope 过窄 按第 6.2～6.4 节核对 AgentRule / scenario / skill 包
任务极慢 group_size / cases / epochs / 并发过大 先按第 9 节最小化超参冒烟
Windows 终端中文乱码 控制台编码 使用较新 evoagent（含 UTF-8 stdio 修复）或终端切 UTF-8
