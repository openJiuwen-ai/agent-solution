# 常见问题 FAQ

## 部署类

### Q: Docker 启动后容器状态一直是 starting，healthy 过了很久才变？
**A:** 这是正常现象。Dockerfile 中配置了 `--start-period=60s`，启动后 60 秒才开始计算健康检查失败。加上 JVM 启动和 Spring Boot 初始化需要 10-30 秒，首次变为 healthy 可能需要 1-2 分钟。如果超过 3 分钟仍未 healthy，请查看日志排查问题。

### Q: 必须用 Docker 部署吗？可以直接 java -jar 运行吗？
**A:** 可以。EDPAgent 是标准 Spring Boot 应用，支持 `java -jar` 直接运行。但直接运行需要：
- 本地安装 JDK 21
- 本地安装 Python 3 并安装 requirements-mcp.txt 中的依赖（如果用 MCP）
- 通过环境变量配置所有参数

推荐生产环境使用 Docker 部署，避免环境差异问题。

### Q: 需要独立部署 Redis 吗？用 Docker 启动一个 Redis 可以吗？
**A:** 测试/POC 可以用 Docker 启动 Redis，但生产环境建议独立部署 Redis（哨兵或集群模式），原因：
- Docker 中 Redis 默认无持久化配置，容器重建数据丢失
- 资源隔离，避免互相影响
- Redis 作为状态存储需要稳定运行

### Q: 支持 K8s/Helm 部署吗？
**A:** 当前版本（v0.2.0）仅提供单机 Docker 部署方案，K8s Helm Chart 在后续版本规划中。

## 配置类

### Q: 如何切换业务场景？
**A:** 修改 `EDP_AGENT_SCENARIO_HOME` 环境变量指向目标场景目录，然后重启服务。Docker 部署时如果场景是自定义的，需要将场景目录挂载到容器内。

```bash
# 切换到智贷通场景
EDP_AGENT_SCENARIO_HOME=/app/scenarios/hz-zhidaitong
docker restart edp-agent
```

### Q: 修改 Governance YAML 后需要重启吗？
**A:** 需要。当前版本不支持热加载，任何配置变更（planrule.yaml、actrule.yaml、scriptconfig.yaml、SKILL.yaml）都需要重启服务才能生效。

### Q: 如何新增一个自定义场景？
**A:** 参考以下步骤：
1. 在 `scenarios/` 目录下创建新场景目录，如 `scenarios/my-scenario/`
2. 创建 `governance/` 子目录，放入 planrule.yaml、actrule.yaml、scriptconfig.yaml（只写需要覆盖的差异配置）
3. 如需 Skill，创建 `skills/` 子目录，放入自定义 Skill
4. 设置 `EDP_AGENT_SCENARIO_HOME` 指向新场景目录
5. 重启服务

参考 [Governance 配置指南](../developer-guide/配置指南.md) 和 [Skill 开发指南](../developer-guide/技能开发指南.md)。

### Q: 如何关闭某个工具（如 call_mcp）？
**A:** 在场景的 actrule.yaml 中不包含该工具即可（注意是**不包含**，不是移除——因为 allowed_tools 是叠加合并的，需要在场景配置中覆盖整个 allowed_tools 列表，或在框架配置中移除）。当前版本的叠加策略下，移除工具需要在框架层 actrule.yaml 中不列出该工具。

### Q: 支持哪些大模型？只能用 DeepSeek 吗？
**A:** 支持所有 OpenAI 兼容接口的模型，包括：
- DeepSeek（默认）
- OpenAI GPT 系列
- Azure OpenAI（需确认接口兼容性）
- 私有部署模型（vLLM、Ollama、OneAPI 等提供 OpenAI 兼容接口的）
- 通义千问、智谱等提供 OpenAI 兼容接口的国产模型

通过配置 `EDP_AGENT_MODEL_BASE_URL`、`EDP_AGENT_MODEL_NAME`、`EDP_AGENT_MODEL_API_KEY` 切换。注意模型必须支持 Tool Calling（Function Calling）和 System Prompt。

## 使用类

### Q: Agent 回复"正在学习中，暂不支持该业务"是什么意思？
**A:** 这是 `out_of_scope` 话术，表示用户请求不在当前场景的业务范围（scope.allowed）内，也不匹配任何 Skill。检查：
1. planrule.yaml 中的 `scope.allowed` 是否包含该业务
2. 对应的 Skill 是否正确加载（查看启动日志）
3. SKILL.yaml 的 description 中是否包含正确的触发词

### Q: Agent 不按 Governance 规则执行怎么办？
**A:** 可能原因：
1. additional_prompt 写得不够明确，LLM 没有严格遵守
2. 模型能力问题，小参数模型可能遵循指令能力较弱
3. 规则之间有冲突，LLM 无法判断优先级

解决方法：
- 在 additional_prompt 中使用"必须"、"严禁"、"违反此规则将被视为错误"等强约束词汇
- 使用遵循能力更强的模型
- 开启 DEBUG 日志查看实际发给 LLM 的 System Prompt，确认规则已正确加载

### Q: Skill 不触发怎么办？
**A:** 排查步骤：
1. 查看启动日志，确认 Skill 被正确加载（"Loaded skill: xxx"）
2. 检查 SKILL.yaml 的 description 是否包含用户触发词
3. 检查 actrule.yaml 的 skill_mode 不是 `none`
4. 检查 actrule.yaml 的 allowed_tools 包含 `skill_tool`
5. 开启 DEBUG 日志，查看 System Prompt 中是否包含该 Skill 的描述
6. 尝试在 description 中增加更多触发词和排除说明

### Q: 如何让 Agent 每次只执行一个工具调用，不并发？
**A:** 在场景的 additional_prompt 中明确要求：
```yaml
additional_prompt: |
  ## 强制规则
  - 每次只执行一个 tool_call，不要并发调用多个工具
```

参考 wealth-demo 场景的配置。

### Q: ask_user 多次追问同一个问题怎么办？
**A:** 可能是用户回复后 Agent 没有正确解析 answer 字段。确保：
1. 用户回复通过正确的 A2A 消息格式传入
2. conversationId 一致（续传需要相同会话 ID）
3. 检查日志中是否收到用户回复内容
4. 在 additional_prompt 中提醒 LLM 检查工具返回结果，不要重复提问

## 运维类

### Q: 如何查看进行中的任务？
**A:** 连接 Redis 查看 Todo 数据：
```bash
redis-cli KEYS edpa:todo:*
redis-cli GET edpa:todo:<conversationId>
```

### Q: 如何强制清理某个会话的状态？
**A:** 删除 Redis 中对应的 key：
```bash
redis-cli DEL edpa:todo:<conversationId>
redis-cli DEL edpa:checkpoint:<conversationId>
```
⚠️ 这会导致该会话的所有进行中任务丢失。

### Q: 日志占用太多磁盘空间怎么办？
**A:** 1. 配置 Docker 日志轮转（已在部署指南中推荐）；2. 降低日志级别为 WARN 或 ERROR；3. 定期清理旧日志。

### Q: 服务运行一段时间后变慢怎么办？
**A:** 可能原因和解决方法：
1. 内存泄漏：检查 JVM 内存使用，必要时重启
2. Redis 中数据过多：检查 TTL 设置，确保旧数据自动过期
3. 大模型 API 变慢：检查模型服务状态，不是 EDPAgent 本身问题
4. 日志量过大：降低日志级别

### Q: 如何备份配置和数据？
**A:** 需要备份的内容：
1. `.env` 环境变量配置文件
2. 自定义场景目录（如果有）
3. Redis 数据（如果需要保留会话状态，使用 Redis BGSAVE）

## 开发类

### Q: Python 脚本能用第三方库吗？
**A:** 可以。需要将依赖添加到 `deploy/requirements-mcp.txt`，然后重新构建 Docker 镜像。如果使用外部 MCP 服务，在 MCP 环境中安装即可。

### Q: Skill 的 Python 脚本如何调试？
**A:** 本地直接运行脚本测试：
```bash
cd scenarios/wealth-demo/skills/product_recommend_skill
python scripts/run_product_recommend_skill.py '{"risk_level":"R2","amount":50000}'
```

### Q: 如何添加自定义 Java 工具？
**A:** 当前版本（v0.2.0）的工具扩展需要修改 Java 代码，通过 EdpaToolRegistry 注册新工具。后续版本将提供 SPI 插件机制。

### Q: 如何不重启更新 Skill 话术？
**A:** 当前版本不支持热加载。如果只需要修改 SKILL.yaml 中的话术文本，修改后必须重启服务。

## 协议类

### Q: A2A 协议和普通 REST API 有什么区别？
**A:** A2A（Agent-to-Agent）是专为 Agent 间通信设计的协议，基于 JSON-RPC over SSE：
- 长连接流式响应（SSE），可以实时推送 Agent 思考过程、工具调用、最终答案
- 支持中断续传（ask_user 后用户回复继续）
- 标准 Agent Card 发现机制
- 与 REST API 不同，A2A 是会话式的，不是请求/响应式的

### Q: 可以用 Postman 测试 A2A 接口吗？
**A:** 可以，但需要注意：
1. 使用 POST 请求，Content-Type: application/json
2. 响应是 SSE 流，Postman 可以显示但不会实时更新（需要用 curl -N 或编写客户端）
3. 必须按照正确的 JSON-RPC 格式发送

建议使用 curl 测试：
```bash
curl -N -X POST http://localhost:8190/a2a \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc":"2.0",
    "id":"test-1",
    "method":"sendStreamingMessage",
    "params":{"message":{"messageId":"msg-1","role":"user","parts":[{"kind":"text","text":"你好"}]}}
  }'
```

### Q: 如何在对话中继续之前中断的任务？
**A:** 发送消息时使用相同的 conversationId（在 SSE 事件流的上下文中返回）。用户回复后，Agent 自动从 Checkpoint 恢复执行。

## 其他

### Q: EDPAgent 和其他 Agent 框架（LangChain、AutoGPT）有什么区别？
**A:**
- EDPAgent 是**企业级运行时**，不是开发框架。它提供完整的 Governance 配置体系、Rail 拦截器链、场景隔离、A2A 协议接入，目标是开箱即用。
- LangChain 等是开发库，需要开发者自己写代码组装 Agent。
- EDPAgent 的核心特色是**配置驱动**——不需要写 Java/Python 代码，通过 YAML 配置即可定义 Agent 行为和业务场景。
- EDPAgent 内置 Versatile 和 MCP 集成，面向企业工作流场景设计。

### Q: EDPAgent 本身支持多轮对话吗？
**A:** 支持。enable_task_loop 为 true（默认）时，所有任务完成后用户可以继续提问，Agent 会重新规划新任务。会话上下文通过 Redis 维护。

### Q: 数据安全如何保障？
**A:**
- API Key、Redis 密码等敏感信息通过环境变量传入，不硬编码在配置文件中
- MCP 脚本在沙箱中运行（如果使用外部 MCP 服务）
- 无内置用户认证（当前版本），生产部署应在内网或通过 API 网关接入
- 不记录用户敏感数据到日志（但 DEBUG 日志会记录 Prompt 内容，生产环境不要开启 DEBUG）
