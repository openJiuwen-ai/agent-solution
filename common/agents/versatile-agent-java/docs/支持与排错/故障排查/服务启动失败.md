# 服务启动失败排错指南

## 问题描述

adapter-versatile-agent-java服务（Spring Boot 4.0.6应用）启动过程中失败，进程退出或Docker容器立即退出，无法正常提供A2A代理服务。

**重要特性**：
- 这是**轻量级无状态代理服务**，启动时**不依赖Redis、数据库、MCP或Python**
- 启动时**不会**检查Versatile后端服务连通性（Versatile不可达不影响启动，只影响运行时调用）
- 仅需JRE和正确配置即可启动
- 默认端口8191

启动流程依次完成：Spring Boot初始化、VersatileProperties配置绑定、VersatileAgentHandler Bean注册、Web容器（Tomcat）启动、Actuator端点注册。

**重要提示**：Versatile服务不可达不会导致启动失败，但会导致运行时A2A请求失败。

## 常见症状

1. **容器立即退出**：`docker run`后容器状态为Exited，`docker ps`看不到运行中的容器
2. **JVM启动失败**：出现`Error: Could not create the Java Virtual Machine`、`UnsupportedClassVersionError`等错误
3. **端口被占用**：日志显示`Port 8191 is already in use`、`Address already in use`
4. **配置解析错误**：YAML语法错误、配置文件找不到、环境变量格式错误、VersatileProperties绑定失败
5. **Bean创建失败**：`VersatileAgentHandler` Bean创建失败、配置校验失败
6. **启动后健康检查一直失败**：进程在但健康检查端点返回错误，Docker状态显示unhealthy
7. **ClassNotFoundException/NoSuchMethodError**：依赖缺失或版本冲突、jar包损坏
8. **OOM启动失败**：堆内存不足，JVM无法启动
9. **URL格式校验失败**：VERSATILE_URL格式不合法导致启动失败

## 可能原因

| 分类 | 可能原因 | 关键特征 |
|------|----------|----------|
| 端口冲突 | 端口8191被占用 | `Address already in use`、`Port 8191 was already in use` |
| Java版本 | JDK/JRE版本不满足 | `UnsupportedClassVersionError`，本地需要JDK 17+，Docker用JDK 21 |
| 配置文件 | YAML配置语法错误 | `ScannerException`、`while parsing a block mapping`、缩进问题 |
| 配置绑定 | VersatileProperties绑定失败 | `Failed to bind properties`、配置格式错误、Duration解析失败 |
| 配置校验 | VERSATILE_URL格式不合法 | URL格式错误、缺少协议头 |
| 内存配置 | 堆内存不足 | `OutOfMemoryError: Java heap space`、`Could not reserve enough space` |
| 依赖问题 | Jar包损坏/依赖冲突 | `ClassNotFoundException`、`NoSuchMethodError`、jar文件不完整 |
| 权限问题 | 文件权限不足 | `Permission denied`、无法读取配置文件 |
| Docker问题 | 镜像构建失败/架构不匹配 | `exec format error`、`no such file or directory` |
| 环境变量 | 环境变量格式错误 | `Duration parse error`（如VERSATILE_TIMEOUT格式错误）、`NumberFormatException` |

**注意**：以下问题**不会**导致启动失败，但会导致运行时错误：
- Versatile后端服务未启动/不可达
- VERSATILE_URL指向的地址无法访问（只要URL格式正确）
- resultNodeName配置与实际Versatile节点不匹配
- Header白名单配置问题

## 排查步骤

### 步骤1：使用docker logs查看完整启动日志

这是排查启动失败的第一步：

```bash
# 查看容器日志（即使容器已退出）
docker logs adapter-versatile

# 查看最后200行日志
docker logs adapter-versatile --tail 200

# 实时查看日志
docker logs adapter-versatile -f
```

本地启动（java -jar或mvn spring-boot:run）：
- 日志直接输出到控制台
- 如果配置了文件日志，查看`logs/run/run.log`文件（默认配置）

**关键日志节点（正常启动顺序）**：
```
  .   ____          _            __ _ _
 /\\ / ___'_ __ _ _(_)_ __  __ _ \ \ \ \
( ( )\___ | '_ | '_| | '_ \/ _` | \ \ \ \
 \/  ___)| |_)| | | | | || (_| |  ) ) ) )
  '  |____| .__|_| |_|_| |_\__, | / / / /
 =========|_|==============|___/=/_/_/_/
 :: Spring Boot ::                (v4.0.6)

Starting VersatileAgentApplication v0.1.0-SNAPSHOT ...
...
Tomcat started on port(s): 8191 (http)
Started VersatileAgentApplication in XX.XXX seconds
```

找到第一个`ERROR`或`Exception`，那里就是启动失败的根本原因。

### 步骤2：检查端口占用

如果日志显示`Address already in use`或`Port 8191 was already in use`：

**Windows：**
```powershell
# 查看8191端口占用
netstat -ano | findstr :8191
# 查看对应进程
tasklist | findstr <PID>
# 结束占用进程（谨慎）
taskkill /F /PID <PID>
```

**Linux/macOS：**
```bash
# 查看8191端口占用
lsof -i :8191
netstat -tlnp | grep 8191
# 结束占用进程
kill -9 <PID>
```

**Docker端口映射冲突：**
```bash
# 查看端口映射
docker ps
# 如果主机8191被占用，映射到其他端口
docker run -p 8192:8191 ...  # 主机8192→容器8191
```

或修改服务监听端口：
```bash
VERSATILE_AGENT_PORT=8080
```

### 步骤3：验证Java版本

adapter-versatile-agent-java的版本要求：
- **本地运行/编译**：需要JDK 17或更高版本（pom.xml中配置`java.version=17`）
- **Docker运行**：使用eclipse-temurin:21-jre-jammy基础镜像（JDK 21）

检查Java版本：
```bash
java -version
# 本地开发预期输出包含 openjdk version "17.x.x" 或更高
# Docker容器内预期为 openjdk version "21.x.x"
```

如果版本不对：
- 本地开发：安装JDK 17+，设置`JAVA_HOME`
- Docker：使用官方Dockerfile（已配置JDK 21），不要修改基础镜像版本

典型错误：
```
Exception in thread "main" java.lang.UnsupportedClassVersionError: 
  com/huawei/ascend/versatile/VersatileAgentApplication has been compiled by a more recent version of the Java Runtime
```
→ 这表示运行时JRE版本低于编译时JDK版本（需要17+）。

### 步骤4：检查YAML配置语法

YAML对缩进和语法要求严格，常见错误：

1. **缩进不一致**（必须用空格，不能用Tab）
2. **冒号后缺少空格**：`key:value` ❌ → `key: value` ✅
3. **特殊字符未转义**：URL中的特殊字符、密码含特殊字符
4. **列表项格式错误**
5. **Duration格式错误**：超时配置需要单位（s/m/ms）

验证YAML语法：
- 使用在线YAML验证工具：https://yaml-online-parser.appspot.com/
- 使用IDE的YAML插件检查
- 使用Python验证：`python -c "import yaml; yaml.safe_load(open('application.yml'))"`

关键配置文件：
- `src/main/resources/application.yml`（主配置）
- `src/main/resources/application.yaml`（备用配置）

典型错误日志：
```
Caused by: org.yaml.snakeyaml.scanner.ScannerException: while scanning a simple key
  in 'reader', line 5, column 1:
    key:value
    ^
could not find expected ':'
```

### 步骤5：检查VersatileProperties配置绑定

配置绑定失败常见原因：
1. Duration格式错误（`VERSATILE_TIMEOUT`）
2. URL格式不合法
3. 类型不匹配（数字字段传入字符串）

**Duration格式正确示例**：
```bash
VERSATILE_TIMEOUT=600s    # 600秒（默认）
VERSATILE_TIMEOUT=10m     # 10分钟
VERSATILE_TIMEOUT=30000ms # 30000毫秒
```

**Duration格式错误示例**：
```bash
VERSATILE_TIMEOUT=600     # ❌ 缺少单位
VERSATILE_TIMEOUT=10min   # ❌ 错误单位（应该是m）
```

**URL格式检查**：
- 必须包含协议头：`http://`或`https://`
- 必须包含`{conversation_id}`占位符（虽然启动时不会校验连通性，但URL格式必须合法）
- 不能有空格或非法字符

检查配置绑定相关日志：
```bash
docker logs adapter-versatile 2>&1 | grep -E "Bind|Properties|versatile"
```

### 步骤6：检查内存配置

如果JVM无法分配足够内存，会启动失败：

错误特征：
```
Error occurred during initialization of VM
Could not reserve enough space for object heap
OutOfMemoryError: Java heap space
```

排查和解决：
1. **检查Docker内存限制**：Docker Desktop默认内存可能不足，建议设置≥2GB（该服务是轻量级代理，不需要太大内存）
2. **调整JVM堆内存参数**：
   ```bash
   # Docker运行时设置
   docker run -e JAVA_OPTS="-Xms256m -Xmx512m" ...
   
   # 或直接在JAVA_TOOL_OPTIONS中设置
   JAVA_TOOL_OPTIONS="-Xms128m -Xmx256m"
   ```
3. **检查系统可用内存**：`free -h`（Linux）或任务管理器（Windows）
4. **降低内存使用**：该服务是轻量级代理，默认堆内存设置即可满足需求

### 步骤7：查看容器退出码

```bash
# 查看容器退出码
docker inspect adapter-versatile --format='{{.State.ExitCode}}'
```

常见退出码：
| 退出码 | 含义 | 可能原因 |
|--------|------|----------|
| 0 | 正常退出 | 应用主动退出（可能是启动后立即关闭，检查是否有daemon线程问题） |
| 1 | 通用错误 | 应用异常，查看日志中的Exception |
| 137 | OOM被Kill | 内存不足，被Docker/K8s OOM Killer杀死 |
| 139 | Segmentation Fault | JVM崩溃、native代码问题 |
| 143 | SIGTERM | 被正常终止（如docker stop） |

退出码137→**内存不足**，增大Docker内存或减小JVM堆。

### 步骤8：检查Docker镜像和架构问题

如果是`exec format error`：
```
exec /usr/local/bin/java: exec format error
```
这表示镜像架构与宿主机不匹配（如在ARM Mac上运行x86镜像，或反之）。

解决：
1. 确认Dockerfile基础镜像支持当前架构（eclipse-temurin:21-jre-jammy支持多架构）
2. 构建镜像时指定平台：`docker build --platform linux/amd64 ...`
3. 或在对应架构机器上重新构建

检查镜像：
```bash
docker inspect adapter-versatile-agent-java:latest --format='{{.Architecture}}'
uname -m  # 查看宿主机架构
```

### 步骤9：检查Jar包完整性

如果出现`ClassNotFoundException`、`NoSuchMethodError`：
1. 重新构建项目：`mvn clean package -DskipTests`
2. 检查Maven依赖是否完整：`mvn dependency:tree`
3. 删除本地Maven仓库中损坏的包，重新下载
4. 确认target目录下jar包完整：
   ```bash
   ls -lh target/adapter-versatile-agent-java-*.jar
   # 文件大小应该在几十MB（包含依赖的fat jar）
   ```

**Maven构建命令**（在项目根目录执行）：
```bash
cd adapter-versatile-agent-java
mvn clean package -DskipTests
# 构建成功后jar包在target/目录下
```

### 步骤10：验证健康检查配置

Docker HEALTHCHECK配置：
- 检查间隔：10秒
- 超时：5秒
- 重试次数：8次
- 启动等待期：90秒
- 检查URL：`http://localhost:8191/.well-known/agent-card.json`

如果容器启动后一直显示starting/health: starting，等待90秒后再检查。如果超过2分钟仍未healthy：

1. 手动测试健康检查端点：
   ```bash
   curl -v http://localhost:8191/.well-known/agent-card.json
   curl -v http://localhost:8191/actuator/health
   ```

2. 查看容器内curl是否可用（Dockerfile已安装curl）：
   ```bash
   docker exec adapter-versatile which curl
   ```

3. 检查应用是否真的在监听8191端口：
   ```bash
   docker exec adapter-versatile netstat -tlnp 2>/dev/null || docker exec adapter-versatile apt-get update && docker exec adapter-versatile apt-get install -y net-tools && docker exec adapter-versatile netstat -tlnp
   ```

### 步骤11：检查文件权限

Linux/Docker环境下可能出现权限问题：
```
Permission denied
java.io.FileNotFoundException: /app/app.jar (Permission denied)
```

解决：
```bash
# 检查文件权限
ls -la target/*.jar
ls -la deploy/

# 构建镜像时确保权限正确（参考Dockerfile，jar包复制到/app/app.jar）
# 不要以root用户以外的用户运行（除非正确配置了权限）
```

## 解决方案

### 方案1：快速启动最小可用环境

**本地开发快速启动：**
```bash
# 1. 进入项目目录
cd adapter-versatile-agent-java

# 2. Maven构建
mvn clean package -DskipTests

# 3. 使用默认配置启动（连接本地Mock Versatile）
# 默认配置已指向 http://localhost:30001/...，可先启动Versatile Mock
mvn spring-boot:run
```

**Docker快速启动（不依赖外部服务即可启动）：**
```bash
# 1. 先构建jar包
mvn clean package -DskipTests

# 2. 构建Docker镜像
docker build -t adapter-versatile-agent-java:latest -f deploy/Dockerfile .

# 3. 启动容器（即使Versatile未启动也能正常启动服务）
docker run -d --name adapter-versatile \
  -p 8191:8191 \
  adapter-versatile-agent-java:latest

# 4. 查看启动日志
docker logs adapter-versatile -f

# 5. 等待90秒后验证健康状态
docker inspect adapter-versatile --format='{{.State.Health.Status}}'
```

**注意**：服务启动后，调用/a2a端点时才会实际连接Versatile。

### 方案2：更换端口

如果8191端口被占用：

**方式1：映射到其他主机端口**
```bash
docker run -p 8192:8191 ...  # 主机8192→容器8191
```

**方式2：修改容器内监听端口**
```bash
docker run -e VERSATILE_AGENT_PORT=8080 -p 8080:8080 ...
```

**本地启动：**
```bash
VERSATILE_AGENT_PORT=8080 mvn spring-boot:run
```

修改端口后，健康检查URL和A2A端点端口也要相应改变。

### 方案3：使用正确的JDK版本

**本地开发：**
1. 下载JDK 17或21：https://adoptium.net/
2. 设置JAVA_HOME：
   ```bash
   # Windows
   set JAVA_HOME=C:\Program Files\Eclipse Adoptium\jdk-17.x.x-hotspot
   # Linux/macOS
   export JAVA_HOME=/Library/Java/JavaVirtualMachines/temurin-17.jdk/Contents/Home
   ```
3. 验证：`java -version`

**Docker：**
使用项目提供的deploy/Dockerfile（基础镜像为eclipse-temurin:21-jre-jammy），不要修改基础镜像版本。

### 方案4：修复YAML配置错误

YAML排错技巧：
1. **复制到在线验证工具**检查语法
2. **统一缩进**：使用2空格缩进，不要混用Tab
3. **对照默认配置**：参考src/main/resources/application.yml
4. **逐步回滚**：最近修改的配置最可能是问题原因
5. **使用IDE**：IntelliJ IDEA/Eclipse有YAML语法检查

**最简可用配置（使用默认值）**：
```yaml

  port: ${VERSATILE_AGENT_PORT:8191}

    agent-id: versatile-agent
    query:
      legacy-path-enabled: true

      url-template: ${VERSATILE_URL:http://localhost:30001/v1/0/agent-manager/workflows/mock_workflow/conversations/{conversation_id}?type=controller&workspace_id=10}

        - stream

management:
  endpoints:
    web:
      exposure:
        include: health,info

logging:
  file:
    name: logs/run/run.log
  level:
    com.huawei.ascend.versatile: DEBUG
```

## 相关配置/日志关键词

### 关键启动日志（按顺序）
1. Spring Boot Banner → Spring启动
2. `Starting VersatileAgentApplication` → 应用启动
3. Spring Bean注册日志（包括`versatileAgentHandler`）
4. `Tomcat started on port(s): 8191` → Web容器启动
5. `Started VersatileAgentApplication` → **启动成功标志**

### 常见错误关键词
- `Address already in use` → 端口占用
- `UnsupportedClassVersionError` → JDK版本错误
- `ScannerException`/`YAML` → YAML语法错误
- `Failed to bind properties` → 配置绑定失败
- `FileNotFoundException` → 文件/路径错误
- `OutOfMemoryError`/`Could not reserve` → 内存不足
- `ClassNotFoundException` → 依赖问题/Jar损坏
- `Permission denied` → 权限问题
- `exec format error` → 架构不匹配
- `Duration parse` → 超时配置格式错误
- `Invalid URL` / `MalformedURLException` → URL格式错误

### 配置项
- `VERSATILE_AGENT_PORT` - 服务端口（默认8191）
- `VERSATILE_URL` - Versatile URL模板（格式正确即可，启动时不检查连通性）
- `VERSATILE_TIMEOUT` - 超时时间（默认600s，注意格式）

## 预防措施

1. **启动前检查脚本**：编写启动前检查脚本，验证端口、Java版本
   ```bash
   # 示例预检脚本
   java -version 2>&1 | grep -E "version \"(17|21)" || { echo "JDK 17+ required"; exit 1; }
   ! netstat -tlnp | grep :8191 || { echo "Port 8191 in use"; exit 1; }
   ```

2. **使用Docker Compose**：如果需要与Versatile一起编排
   ```yaml
   services:
     adapter-versatile:
       image: adapter-versatile-agent-java:latest
       ports:
         - "8191:8191"
       environment:
         - VERSATILE_URL=http://versatile:30001/...
       # 注意：不需要depends_on versatile，因为启动不依赖Versatile可达
   ```

3. **配置健康检查**：利用Docker HEALTHCHECK（已默认配置）或K8s liveness/readiness探针

4. **配置中心化管理**：使用ConfigMap/Nacos/Apollo管理配置，避免手动修改YAML出错

5. **CI/CD验证**：
   - PR构建时自动验证配置文件语法
   - 自动化启动测试（验证能正常启动并通过健康检查）
   - 镜像扫描和架构验证

6. **资源配额**：
   - 该服务是轻量级代理，JVM堆设置256-512MB足够
   - 容器内存limits设置为512MB-1GB即可

7. **日志收集**：
   - 启动日志持久化存储（logs/run/run.log）
   - 监控启动失败告警（容器重启、CrashLoopBackOff）

8. **版本管理**：
   - 使用明确版本标签，不使用latest
   - 升级前在测试环境验证启动
   - 保留回滚镜像版本
