# 常见问题 FAQ

## 部署类

### Q: 必须用Docker部署吗？可以直接java -jar运行吗？
**A:** 可以。adapter-versatile-agent-java是标准Spring Boot应用，支持`java -jar`直接运行。但需要：
- 本地安装JDK 17或更高版本
- 通过环境变量配置参数（可选，有默认值）
- 先执行`mvn clean package -DskipTests`构建jar包

推荐生产环境使用Docker部署，避免环境差异问题。

参考：[服务启动失败排错](./troubleshooting/startup-failure.md)

### Q: 需要Redis吗？需要数据库吗？需要Python环境吗？
**A:** 都不需要。这是**纯Java轻量级无状态代理服务**：
- ❌ 无Redis依赖（无状态，不需要会话存储）
- ❌ 无数据库依赖
- ❌ 无MCP依赖
- ❌ 无Python依赖（不需要安装Python或任何Python包）
- ✅ 仅需JRE运行环境

这是与EDPAgent主应用的重要区别——EDPAgent需要Redis存储会话状态，但adapter作为无状态代理不需要。

### Q: 默认端口是多少？可以修改吗？
**A:** 默认端口是**8191**。可以通过环境变量`VERSATILE_AGENT_PORT`修改：
```bash
# 修改为8080端口
VERSATILE_AGENT_PORT=8080
```

Docker部署时也可以通过端口映射修改主机端口：
```bash
# 主机8192映射到容器8191
docker run -p 8192:8191 adapter-versatile-agent-java:latest
```

### Q: Docker启动后容器状态一直是starting，多久能变成healthy？
**A:** 这是正常现象。Dockerfile中配置了`--start-period=90s`，启动后90秒才开始计算健康检查失败。加上JVM启动和Spring Boot初始化需要10-30秒，首次变为healthy可能需要1.5-2分钟。如果超过3分钟仍未healthy，请查看日志排查问题。

健康检查通过`/.well-known/agent-card.json`端点进行。

### Q: 支持K8s/Helm部署吗？
**A:** 当前版本（v0.1.0-SNAPSHOT）提供单机Docker部署方案，K8s Helm Chart在后续版本规划中。由于是无状态服务，K8s部署相对简单，可自行编写Deployment/Service manifest。

### Q: 可以水平扩展吗？多实例部署需要注意什么？
**A:** 可以水平扩展。因为服务是**完全无状态**的：
- 会话状态不存储在本地（由上游EDPAgent通过Redis维护）
- 每次请求都包含完整的contextId/conversationId
- 不依赖本地内存或本地文件

多实例部署注意事项：
1. 前面配置负载均衡（如Nginx、K8s Service）
2. 确保所有实例配置相同（VERSATILE_URL、超时等）
3. 会话亲和性不是必须的，但建议配置以获得更好的调试体验

## 配置类

### Q: 如何配置Versatile服务地址？
**A:** 通过环境变量`VERSATILE_URL`配置，这是**URL模板**，必须包含`{conversation_id}`占位符：

```bash
# 示例
VERSATILE_URL=http://versatile-host:30001/v1/0/agent-manager/workflows/your_workflow/conversations/{conversation_id}?type=controller&workspace_id=10
```

**URL模板说明：**
- `{conversation_id}`：运行时自动替换为A2A请求中的contextId（会话ID）
- 可以包含其他查询参数（如`type=controller&workspace_id=10`）
- 必须包含协议头（`http://`或`https://`）

默认值（用于本地开发Mock）：
```
http://localhost:30001/v1/0/agent-manager/workflows/mock_workflow/conversations/{conversation_id}?type=controller&workspace_id=10
```

参考：[Versatile调用异常排错](./troubleshooting/versatile-error.md)

### Q: resultNodeName是什么？如何配置？
**A:** `resultNodeName`（环境变量`VERSATILE_RESULT_NODE`）用于识别SSE流中的**结果节点**：

- 匹配此节点名的SSE节点会被识别为**最终结果**，回灌给LLM（Target.LLM）
- 不匹配的其他节点会作为**透传节点**直接发送给前端（Target.USER），用于显示菜单、提示文本、中间过程等
- 如果整个SSE流结束都没找到结果节点，会触发中断（Interrupt）

默认值是`GXZQAResponseNode`（对应Versatile Mock服务的QA结果节点）。

配置方式：
```bash
VERSATILE_RESULT_NODE=YourActualResultNodeName
```

**如何找到正确的resultNodeName？**
1. 直接curl Versatile API查看返回的SSE流
2. 开启DEBUG日志查看每个节点的`node_name`字段
3. 找到最终答案所在节点的node_name

### Q: 如何按intent路由到不同的Versatile工作流？
**A:** 当前版本支持通过`input-metadata-keys`提取A2A请求metadata中的`intent`字段，配合endpoints配置可以实现多工作流路由。

1. A2A请求metadata中携带intent字段：
   ```json
   {
     "params": {
       "metadata": {
         "intent": "product_recommendation",
         "userId": "xxx"
       },
       "message": {...}
     }
   }
   ```

2. intent会自动注入到Versatile请求body中，Versatile服务端根据intent路由到不同工作流。

3. 如果需要更复杂的路由（不同intent→不同URL），需要扩展VersatileAgentHandler或在Versatile侧做路由。

### Q: Header白名单是什么？怎么配置自定义Header透传？
**A:** adapter只透传白名单内的Header给Versatile服务，避免无关Header泄露或干扰。

默认白名单（`forward-header-whitelist`）：
- `stream`
- `x-invoke-mode`
- `x-language`

如果需要透传自定义Header（如认证token、自定义追踪ID），需要修改`application.yml`中的白名单配置：
```yaml
openjiuwen:
  service:
    versatile:
      forward-header-whitelist:
        - stream
        - x-invoke-mode
        - x-language
        - authorization
        - x-trace-id
```

默认请求头（`headers-template`）会自动添加：
```
Content-Type: application/json
accept: */*
stream: true
x-invoke-mode: DEBUG
x-language: zh-cn
```

### Q: 如何调整超时时间？
**A:** 通过环境变量`VERSATILE_TIMEOUT`配置，默认600秒（10分钟）。

```bash
# 设置为10分钟（默认）
VERSATILE_TIMEOUT=600s

# 设置为20分钟
VERSATILE_TIMEOUT=1200s

# 设置为5分钟
VERSATILE_TIMEOUT=5m
```

支持的时间单位：
- `ms`：毫秒
- `s`：秒
- `m`：分钟

注意：超时时间应该略大于Versatile工作流的最大预期执行时间。超时过短会导致SSE流中断，过长可能导致请求堆积。

### Q: 配置修改后需要重启吗？
**A:** 需要。当前版本不支持热加载，任何配置变更（环境变量、application.yml）都需要重启服务才能生效。

## 使用类

### Q: adapter-versatile-agent-java和EDPAgent是什么关系？
**A:** 它们是**协作关系**：
- **EDPAgent**：主Agent运行时，包含LLM推理、工具调用、Governance配置、Redis状态存储等完整能力
- **adapter-versatile-agent-java**：独立部署的A2A代理适配器，专门负责A2A协议到Versatile REST/SSE协议的转换和节点分流

EDPAgent通过A2A协议调用adapter-versatile-agent-java的`/a2a`端点，adapter再转发请求给Versatile服务。这种架构的好处：
1. 进程隔离：Versatile调用问题不会影响EDPAgent主进程
2. 独立扩展：可以单独扩容adapter实例
3. 语言无关：任何支持A2A协议的Agent都可以通过这个adapter调用Versatile
4. 独立迭代：adapter可以单独升级不影响EDPAgent

### Q: 为什么需要独立部署？能不能集成到EDPAgent里？
**A:** 技术上可以集成（EDPAgent也有REST直连模式），但独立部署有以下优势：
1. **故障隔离**：Versatile调用超时、OOM、连接泄漏等问题不会搞垮EDPAgent主进程
2. **资源隔离**：Versatile调用是IO密集型，可以单独分配资源
3. **多实例共享**：多个EDPAgent实例或其他Agent可以共享同一个adapter集群
4. **独立升级**：Versatile协议变更时只需升级adapter，不需要升级所有EDPAgent实例
5. **语言异构**：adapter是纯Java，未来可以用其他语言重写而不影响EDPAgent

EDPAgent中也有REST直连Versatile的模式，但A2A适配器模式是推荐的生产部署方式。

### Q: 节点类型怎么区分？Target.USER/Target.LLM/Interrupt分别是什么？
**A:** Versatile返回的SSE节点会按类型进行**三分流处理**：

| 分流目标 | 触发条件 | 用途 | 前端表现 |
|----------|----------|------|----------|
| **Target.USER** | node_name不匹配resultNodeName | 透传中间节点：菜单、确认框、提示文本、进度信息等 | 用户可见的中间交互内容（如"请确认购买"菜单） |
| **Target.LLM** | node_name匹配resultNodeName | 最终结果节点：业务流程完成后的答案/数据 | 结果回灌给LLM，由LLM整理后回复用户 |
| **Interrupt** | SSE流结束但未找到结果节点，或Versatile返回input_required状态 | 工作流中断，需要用户补充信息 | 触发A2A中断机制，等待用户输入后可续传 |

简单理解：
- 菜单、提示→**直接给用户看**（透传）
- 最终答案→**给LLM处理**（结果）
- 需要用户输入→**中断等待**（Interrupt）

### Q: 什么情况下触发中断？中断后如何续传？
**A:** 触发中断的场景：
1. Versatile SSE流正常结束（收到`[DONE]`）但全程没有匹配到resultNodeName
2. Versatile返回明确的`input_required`状态（需要用户选择/输入）
3. SSE流异常中断且没有收到结果节点

中断后续传流程：
1. adapter通过A2A协议返回`TASK_STATE_INPUT_REQUIRED`状态
2. EDPAgent的VersatileInterruptRail捕获中断，保存上下文
3. 用户在前端看到透传的菜单/提示，进行选择或输入
4. EDPAgent使用**相同的conversationId**发送新的A2A请求
5. adapter将用户输入转发给Versatile，继续执行工作流

注意：续传状态的维护由**上游EDPAgent+Redis**负责，adapter本身是无状态的。

### Q: "非结果节点透传"是什么意思？为什么不直接把所有节点都给LLM？
**A:** "透传"是指SSE节点不经过LLM处理，直接原样发送给前端展示。这样设计的原因：

1. **保留结构化信息**：菜单、卡片、按钮等是结构化数据，经过LLM处理可能会丢失格式（如按钮变成普通文本）
2. **实时交互体验**：透传节点可以实时渲染，用户能看到工作流进度和中间步骤，不需要等LLM"转述"
3. **减少LLM调用次数**：中间节点不需要LLM理解和生成，节省token和延迟
4. **前端控制交互**：菜单选择、按钮点击等交互由前端控制，LLM只需要处理最终结果和业务逻辑

典型透传节点示例：
```json
{"node_type":"menu","menu_type":"confirm","text":"确认购买该产品？","options":["确认","取消"]}
```
这个菜单节点直接透传给前端，前端渲染成确认按钮，而不是让LLM把它变成自然语言问用户。

### Q: 为什么Versatile返回的内容有些是直接显示，有些是被LLM"润色"后才显示？
**A:** 这就是三分流机制的体现：
- **透传节点**（菜单、提示、进度）：直接显示给用户，保持原格式
- **结果节点**：发送给LLM，LLM可以根据上下文对结果进行整理、解释、补充，然后回复用户

例如Versatile返回结果节点：
```json
{"node_type":"QA","node_name":"GXZQAResponseNode","answer":"R2稳健型，收益率3.5%"}
```
LLM收到后可能回复："根据您的风险承受能力，为您推荐稳健理财A，风险等级R2，预期年化收益率3.5%，适合稳健型投资者..."

## 运维类

### Q: 日志在哪里？怎么查看？
**A:** 日志位置和查看方式：

**文件日志（默认配置）：**
- 路径：`logs/run/run.log`
- Docker容器内路径：`/app/logs/run/run.log`
- 日志级别：默认DEBUG（`com.huawei.ascend.versatile`和`com.openjiuwen.service.adapters.versatile`包）

**查看日志：**
```bash
# Docker实时查看日志
docker logs adapter-versatile -f

# 查看最后200行
docker logs adapter-versatile --tail 200

# 搜索错误日志
docker logs adapter-versatile 2>&1 | grep -E "ERROR|Exception|Failed"

# 查看容器内文件日志
docker exec adapter-versatile tail -f logs/run/run.log
```

**本地运行日志：**
- 控制台直接输出
- 同时写入`logs/run/run.log`

### Q: 服务是无状态的吗？重启会丢失数据吗？
**A:** 是**完全无状态**的：
- 不存储会话数据
- 不使用本地内存缓存业务数据
- 不写本地文件（除了日志）
- 所有请求都携带完整上下文

重启服务**不会丢失任何业务数据**，会话状态由上游EDPAgent保存在Redis中。重启后新的请求会正常处理。

### Q: 如何排查问题？排错流程是什么？
**A:** 标准排错流程：
1. **检查服务是否启动**：`curl /actuator/health` → 如果不通过，参考[启动失败排错](./troubleshooting/startup-failure.md)
2. **直接测试Versatile连通性**：用curl绕过adapter直接调用Versatile → 如果失败，是Versatile的问题
3. **查看adapter日志**：开启DEBUG日志，查看请求URL、Headers、SSE节点内容
4. **检查resultNodeName匹配**：确认配置的节点名与Versatile实际返回一致
5. **检查Header透传**：确认需要的Header在白名单内
6. **测试A2A端点**：用curl发送A2A请求查看SSE输出

参考：[技术支持渠道](./support.md)中的自助排错流程图。

### Q: 有哪些管理端点？
**A:** 通过Spring Boot Actuator暴露的管理端点（端口同服务端口8191）：

| 端点 | 用途 |
|------|------|

| `/actuator/info` | 应用信息（版本等） |
| `/.well-known/agent-card.json` | A2A标准Agent Card，用于服务发现和健康检查 |

注意：actuator端点只暴露了health和info，其他端点默认未开启。

### Q: 日志级别可以调整吗？生产环境用什么级别？
**A:** 可以调整。默认配置是DEBUG级别，便于开发调试：
```yaml

  level:
    com.huawei.ascend.versatile: DEBUG
    com.openjiuwen.service.adapters.versatile: DEBUG
```

**生产环境建议调整为INFO级别**，减少日志量：

## 协议类

### Q: A2A端点是什么？怎么调用？
**A:** A2A端点是`http://host:8191/a2a`，使用JSON-RPC 2.0 over SSE协议，方法名是`SendStreamingMessage`（大小写不敏感）。

**调用示例：**
```bash
curl -N -X POST "http://localhost:8191/a2a" \
  -H "Content-Type: application/json" \
  -H "Accept: text/event-stream" \
  -d '{
    "jsonrpc": "2.0",
    "method": "SendStreamingMessage",
    "id": "test-001",
    "params": {
      "metadata": {
        "userId": "test-user",
        "agentId": "edp-agent",
        "intent": "balance_query"
      },
      "message": {
        "role": "ROLE_USER",
        "messageId": "msg-test-001",
        "contextId": "test-conv-001",
        "parts": [
          {"text": "{\"inputs\": {\"query\": \"查询余额\"}}"}
        ]
      }
    }
  }'
```

关键参数说明：
- `contextId`：会话ID，用于替换URL中的`{conversation_id}`占位符
- `metadata.intent`：意图，路由到不同工作流
- `message.parts[].text`：JSON字符串，包含inputs等业务参数

### Q: Agent Card有什么用？
**A:** Agent Card是A2A协议的标准发现机制，端点是`/.well-known/agent-card.json`，作用：
1. **服务发现**：Agent可以通过这个端点发现adapter提供的能力
2. **健康检查**：Docker HEALTHCHECK和K8s探针都用这个端点
3. **能力声明**：返回支持的skills、agent描述、端点信息等
4. **兼容性验证**：客户端可以先获取Agent Card确认协议兼容

返回示例：
```json
{
  "agentId": "versatile-agent",
  "description": "Remote runtime that processes banking workflows...",
  "skills": [...],
  "url": "http://host:8191/a2a",
  "version": "0.1.0-SNAPSHOT"
}
```

### Q: SSE流格式是什么？
**A:** A2A SSE响应是标准Server-Sent Events格式，每个事件以`data: `开头，事件之间用空行分隔：

```
data: {"jsonrpc":"2.0","method":"artifactUpdate","params":{"artifact":{"parts":[{"kind":"text","text":"{\"node_type\":\"menu\",...}"}]}}}

data: {"jsonrpc":"2.0","method":"artifactUpdate","params":{"artifact":{"parts":[{"kind":"text","text":"{\"node_type\":\"QA\",\"node_name\":\"GXZQAResponseNode\",...}"}]}}}

data: {"jsonrpc":"2.0","method":"statusUpdate","params":{"status":{"state":"TASK_STATE_COMPLETED"}}}

data: [DONE]
```

事件类型：
- `artifactUpdate`：消息/内容更新（透传节点和结果节点都通过这个发送）
- `statusUpdate`：状态更新（完成、中断等）
- `[DONE]`：流结束标记

### Q: 支持A2A的哪些方法？
**A:** 当前版本主要支持：
- ✅ `SendStreamingMessage`：流式消息发送（核心方法）
- ❌ 其他A2A方法（如CancelTask、GetTask等）在后续版本规划中

对于代理适配器场景，`SendStreamingMessage`是最核心的方法，已覆盖主要使用场景。

## 开发类

### Q: 本地如何调试？
**A:** 本地调试步骤：

**1. 启动Versatile Mock（可选，用于端到端测试）：**
```bash
cd ../../edp-agent-java/mock
pip install fastapi uvicorn python-dotenv loguru
python versatile_main.py
# Mock服务监听 http://localhost:30001
```

**2. IDE中启动：**
- 用IntelliJ IDEA打开项目
- 找到`VersatileAgentApplication.java`
- 右键→Debug 'VersatileAgentApplication'
- 默认配置即可启动，连接本地Mock

**3. 远程调试Docker容器：**
```bash
# 启动时开启debug端口
docker run -d -p 8191:8191 -p 5005:5005 \
```

**4. 发送测试请求：**
使用curl发送A2A请求（参考Q: A2A端点是什么？），在IDE中打断点调试。

### Q: 如何对接自定义Versatile服务？
**A:** 对接自定义Versatile服务步骤：

1. **确认Versatile API格式：**
   - 请求是HTTP POST
   - Content-Type: application/json
   - 支持SSE流式响应（stream=true时）
   - URL包含conversation_id路径参数
   - 返回SSE格式：`data: {json}\n\ndata: [DONE]\n\n`

2. **配置VERSATILE_URL：**
   ```bash
   VERSATILE_URL=http://your-versatile/api/path/conversations/{conversation_id}?your_params=value
   ```

3. **确认结果节点名：**
   - 调用你的Versatile服务查看返回的SSE节点
   - 找到最终结果的node_name
   - 配置`VERSATILE_RESULT_NODE=YourResultNodeName`

4. **配置Header（如需要）：**
   - 在`headers-template`中添加默认Header（如认证token）
