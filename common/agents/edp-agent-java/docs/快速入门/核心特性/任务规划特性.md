# 任务规划特性

## 特性概述

任务规划是EDP Agent的核心特性，采用Openjiuwen DeepAgent循环模式处理用户请求。该特性通过Todo状态机管理任务生命周期，支持catalog_id预定义模板、动态路径调整、依赖关系（DAG）管理，并通过执行限制防止任务失控。

核心能力包括：
- **DeepAgent循环**：规划-执行-观察-反思的闭环处理
- **Todo状态机**：严格的状态流转控制（PENDING→IN_PROGRESS→COMPLETED/CANCELLED）
- **Todo工具集**：todo_create/todo_modify/todo_list/todo_get四个专用工具
- **catalog_id模板**：预定义任务模板快速创建任务
- **动态路径调整**：根据执行结果动态跳过或调整任务路径
- **依赖关系（DAG）**：自动解析任务依赖，保证执行顺序
- **执行限制**：最大步数、最大子任务数限制

## 配置方式

### 1. 状态机配置（planrule.yaml）

在`planrule.yaml`的`supplementary_prompt.base_protocol`中定义状态机规则：

```yaml
planrule:
  supplementary_prompt:
    base_protocol: |
      ### 状态机（严格枚举）
      任务状态只能通过 `todo_modify` 工具切换，取值只能是：
        PENDING → IN_PROGRESS → COMPLETED
                    └→ CANCELLED
      - 开始执行某任务前：todo_modify 将其置为 IN_PROGRESS
      - 任务完成：todo_modify 将其置为 COMPLETED（并填 result_summary）
      - 任务取消（含路径跳过）：todo_modify 将其置为 CANCELLED（并填 result_summary）
```

### 2. 执行限制配置（actrule.yaml）

在`actrule.yaml`中配置最大子任务数和最大步数：

```yaml
actrule:
  # 限制单层最大子任务数量
  max_subtasks: 50
  
  # 限制最大执行步数
  max_steps: 100
  
  # 是否启用任务循环
  enable_task_loop: true
```

### 3. Todo工具白名单配置

在`actrule.yaml`的`allowed_tools`中启用Todo相关工具：

```yaml
actrule:
  allowed_tools:
    - todo_create
    - todo_modify
    - todo_list
    - todo_get
    - cancel_task
    # ... 其他工具
```

## 使用示例

### 1. 使用catalog_id创建任务

```python
# 调用todo_create工具，使用预定义catalog_id
todo_create({
  tasks: [
    {catalog_id: "default_task"},
    {catalog_id: "product_recommend"}
  ]
})
```

使用catalog_id时，系统自动填充content/description/skill字段，无需手动填写。

### 2. 状态流转示例

```python
# 1. 开始执行任务 - 将任务置为IN_PROGRESS
todo_modify({
  action: "update",
  todo_id: "task-1",
  status: "IN_PROGRESS"
})

# 2. 任务完成 - 将任务置为COMPLETED并填写结果
todo_modify({
  action: "update",
  todo_id: "task-1",
  status: "COMPLETED",
  result_summary: "产品推荐完成，共推荐5款理财产品"
})

# 3. 动态路径调整 - 取消跳过的任务
todo_modify({
  action: "cancel",
  todo_id: "task-3",
  result_summary: "用户无需风险评估，跳过此步骤"
})
```

### 3. 批量状态更新

完成任务时可批量更新：一次todo_modify同时标记当前任务COMPLETED + 下一任务IN_PROGRESS：

```python
todo_modify({
  action: "update",
  updates: [
    {todo_id: "task-1", status: "COMPLETED", result_summary: "步骤1完成"},
    {todo_id: "task-2", status: "IN_PROGRESS"}
  ]
})
```

### 4. 最终回答同步更新

所有任务执行完毕后，必须在输出final_answer的同一迭代中调用todo_modify标记最后一个任务为COMPLETED：

```python
# 同一迭代中：todo_modify + final_answer
todo_modify({
  action: "update",
  todo_id: "task-last",
  status: "COMPLETED",
  result_summary: "全部任务完成"
})
# 紧接着输出final_answer
final_answer("【需求概述】...【规划过程】...【结果汇总】...")
```

## 关键实现机制

### 1. ReAct循环流程

Agent采用「规划—执行—观察—反思」循环：

1. **规划阶段**：分析用户请求，调用todo_create创建任务列表
2. **执行阶段**：按顺序执行任务，调用业务工具（call_versatile/call_mcp）
3. **观察阶段**：分析工具返回结果，判断是否满足路径切换条件
4. **反思阶段**：决定下一步操作，更新任务状态，继续下一轮循环

### 2. 规划前置守卫（EdpaTodoRail）

`EdpaTodoRail.beforeToolCall()`实现了强制规划守卫：
- 业务工具（call_mcp/call_versatile）执行前必须已规划todo
- 若未规划，拦截工具执行并返回PLAN_FIRST错误
- 通过steering强制LLM先调用todo_create

代码位置：`EdpaTodoRail.java:279-302`

### 3. 依赖闭环（DAG自动解析）

`EdpaTodoRail.afterToolCall()`在todo_create后自动完成依赖闭环：

1. 从todos的`meta_data.catalog_id`建立catalog_id→UUID映射（anchors）
2. 根据catalog配置的depends_on，将catalog_id引用替换为真实UUID
3. 写回Redis/文件存储，保证depends_on是合法DAG

代码位置：`EdpaTodoRail.java:364-472`

### 4. 动态路径规则注入

`EdpaTodoRail.buildPathRulesPrompt()`将dynamic_paths配置注入为prompt，引导LLM在适当时机取消任务：

```
## 动态路径选择规则
你在执行任务列表时，可以根据业务执行结果动态调整路径。
### 路径1：无需风险评估（skip_risk_assessment）
- 触发条件：用户已完成风险评估或选择低风险产品
- 操作：
  1. 调用 todo_modify 将 risk_assessment 对应的任务标记为 cancelled
```

代码位置：`EdpaTodoRail.java:548-582`

### 5. 子任务数量限制

`EdpaTodoRail.beforeToolCall()`在todo_create时校验子任务数量：
- 超过`actrule.max_subtasks`时拦截并返回MAX_SUBTASKS_EXCEEDED错误
- 默认上限为50个子任务

代码位置：`EdpaTodoRail.java:213-229`

## 注意事项

1. **禁止旧协议**：禁止在thought中输出```todolist```或```todo_update```markdown代码块，一切任务操作必须通过todo_create/todo_modify工具调用完成。

2. **todo_modify调用约束**：
   - 调用时必须提供action参数（取值：update/cancel）
   - 返回成功后不要重复调用相同的状态更新
   - 完成最后一个任务后无需重新规划，直接todo_modify + final_answer

3. **禁止在最终回答中提问**：如果需要用户补充信息，必须调用ask_user工具，不能在final_answer中用文本提问。

4. **无依赖工具可并行**：同一迭代中可执行多个无依赖的工具调用（如skill_tool + todo_create），但有数据依赖的工具必须分步执行。

5. **依赖关系自动解析**：LLM无需手动填写depends_on字段，系统在todo_create后自动通过catalog_id映射完成依赖闭环。

6. **Redis优先存储**：启用Redis时，Todo状态优先存储在Redis中，文件存储仅作为Core框架内部使用，会话结束后自动清理。
