# 外部集成指南

EDPAgent 作为智能体运行时，需要与多个外部系统集成。本文档介绍如何配置和对接这些外部系统。

## 集成架构

```
                          ┌─────────────────┐
                          │   大模型 API     │
                          │ (DeepSeek/OpenAI)│
                          └────────▲────────┘
                                   │
┌───────────┐              ┌───────┴────────┐              ┌───────────────┐
│  客户端    │── A2A ──────▶│   EDPAgent     │── REST/A2A ─▶│   Versatile   │
│ (上层系统) │◀── SSE ──────│   Engine       │              │ (业务工作流)   │
└───────────┘              └───────┬────────┘              └───────────────┘
                                   │
                          ┌────────▼────────┐
                          │     Redis       │
                          │ (Todo/Checkpoint)│
                          └─────────────────┘
                                   │
                          ┌────────▼────────┐
                          │    MCP SSE      │
                          │  (脚本沙箱)      │
                          └─────────────────┘
```

EDPAgent 的外部集成点包括：
1. **大模型 API**：LLM 推理服务
2. **Redis**：任务状态和 Checkpoint 存储
3. **Versatile**：业务工作流委托
4. **MCP SSE**：脚本沙箱执行
5. **客户端/上层系统**：通过 A2A 协议接入

## 1. 大模型集成

EDPAgent 默认使用 DeepSeek API，但支持所有 OpenAI 兼容的接口。

### 配置项

在 `application.yml` 或环境变量中配置：

```yaml
edpa:
  agent:
    model:
      provider: ${EDP_AGENT_MODEL_PROVIDER:OpenAI}
      name: ${EDP_AGENT_MODEL_NAME:deepseek-v4-pro}
      base-url: ${EDP_AGENT_MODEL_BASE_URL:https://api.deepseek.com/v1}
      api-key: ${EDP_AGENT_MODEL_API_KEY:}
```

| 环境变量 | 说明 | 默认值 |
|----------|------|--------|
| `EDP_AGENT_MODEL_PROVIDER` | 模型提供者，OpenAI 兼容接口统一填 `OpenAI` | `OpenAI` |
| `EDP_AGENT_MODEL_NAME` | 模型名称 | `deepseek-v4-pro` |
| `EDP_AGENT_MODEL_BASE_URL` | API Base URL | `https://api.deepseek.com/v1` |
| `EDP_AGENT_MODEL_API_KEY` | API Key（必填） | - |

### 切换到其他模型

**使用 OpenAI GPT**：

```bash
EDP_AGENT_MODEL_PROVIDER=OpenAI
EDP_AGENT_MODEL_NAME=gpt-4o
EDP_AGENT_MODEL_BASE_URL=https://api.openai.com/v1
EDP_AGENT_MODEL_API_KEY=sk-your-openai-key
```

**使用私有部署模型（如 vLLM/Ollama 等 OpenAI 兼容服务）**：

```bash
EDP_AGENT_MODEL_PROVIDER=OpenAI
EDP_AGENT_MODEL_NAME=your-model-name
EDP_AGENT_MODEL_BASE_URL=http://your-private-llm:8000/v1
EDP_AGENT_MODEL_API_KEY=your-key-if-needed
```

**使用 Azure OpenAI**：

需要确认 Azure endpoint 是否完全兼容 OpenAI 接口格式，Base URL 格式通常为：
`https://{resource}.openai.azure.com/openai/deployments/{deployment}/`

### 注意事项

1. **模型必须支持 Tool Calling（Function Calling）**：EDPAgent 重度依赖工具调用能力，请确认所选模型支持。
2. **模型必须支持 System Prompt**：Governance 配置通过 System Prompt 生效，不支持 System Prompt 的模型无法使用。
3. **流式输出**：EDPAgent 使用流式调用（stream=true），模型服务必须支持 SSE 流式响应。
4. **思维链模式**：如果使用 fixed_script 模式（scriptconfig.yaml 中配置），模型不需要返回 thinking；如果使用 real_stream 模式，模型需要支持 thinking 输出。

## 2. Redis 集成

Redis 用于存储 Todo 任务状态和会话 Checkpoint（中断恢复点）。

### 配置项

```yaml
spring:
  data:
    redis:
      host: ${EDPA_REDIS_HOST:localhost}
      port: ${EDPA_REDIS_PORT:6379}
      password: ${EDPA_REDIS_PASSWORD:}
      database: ${EDPA_REDIS_DB:0}

edpa:
  agent:
    redis:
      mode: ${EDPA_REDIS_MODE:single}
      host: ${EDPA_REDIS_HOST:localhost}
      port: ${EDPA_REDIS_PORT:6379}
      password: ${EDPA_REDIS_PASSWORD:}
      database: ${EDPA_REDIS_DB:0}
      connect-timeout-ms: ${EDPA_REDIS_CONNECT_TIMEOUT:5000}
      socket-timeout-ms: ${EDPA_REDIS_SOCKET_TIMEOUT:10000}
      checkpointer-ttl-minutes: ${EDPA_REDIS_CHECKPOINTER_TTL:60}
      todo:
        key-prefix: ${EDPA_REDIS_KEY_PREFIX:edpa}
        ttl-seconds: ${EDPA_REDIS_TODO_TTL:3600}
        refresh-on-read: ${EDPA_REDIS_REFRESH_ON_READ:true}
```

| 环境变量 | 说明 | 默认值 |
|----------|------|--------|
| `EDPA_REDIS_MODE` | 部署模式：`single`/`sentinel`/`cluster` | `single` |
| `EDPA_REDIS_HOST` | Redis 主机 | `localhost` |
| `EDPA_REDIS_PORT` | Redis 端口 | `6379` |
| `EDPA_REDIS_PASSWORD` | Redis 密码 | 空 |
| `EDPA_REDIS_DB` | 数据库编号 | `0` |
| `EDPA_REDIS_CONNECT_TIMEOUT` | 连接超时（毫秒） | `5000` |
| `EDPA_REDIS_SOCKET_TIMEOUT` | Socket 超时（毫秒） | `10000` |
| `EDPA_REDIS_CHECKPOINTER_TTL` | Checkpoint TTL（分钟） | `60` |
| `EDPA_REDIS_KEY_PREFIX` | Key 前缀 | `edpa` |
| `EDPA_REDIS_TODO_TTL` | Todo 数据 TTL（秒） | `3600` |
| `EDPA_REDIS_REFRESH_ON_READ` | 读取时自动续期 | `true` |

### Redis 部署模式

#### 单机模式（single）

默认模式，直接连接单个 Redis 实例：

```bash
EDPA_REDIS_MODE=single
EDPA_REDIS_HOST=redis-server
EDPA_REDIS_PORT=6379
```

#### 哨兵模式（sentinel）

配置哨兵模式需要额外的哨兵节点信息（当前版本在 application.yml 中配置 sentinel 节点列表，环境变量支持基础配置）。

#### 集群模式（cluster）

集群模式下客户端自动发现集群节点。

### Key 格式

EDPAgent 在 Redis 中使用以下 Key 格式：

| Key 格式 | 说明 | TTL |
|----------|------|-----|
| `edpa:todo:{conversationId}` | Todo 列表数据 | 3600秒 |
| `edpa:checkpoint:{conversationId}` | 会话 Checkpoint（中断恢复） | 60分钟 |

Key 前缀通过 `EDPA_REDIS_KEY_PREFIX` 配置（默认 `edpa`）。

### 注意事项

1. **Redis 版本**：建议 Redis 6.0+，使用 RESP3 协议。
2. **持久化**：生产环境建议开启 RDB 或 AOF 持久化，避免重启丢失 Checkpoint 导致中断续传失败。
3. **内存**：Todo 和 Checkpoint 数据量不大（每会话几 KB），2GB 内存可支持上万并发会话。
4. **TTL 策略**：Todo 默认 1 小时过期，Checkpoint 默认 60 分钟过期。超时后会话无法续传，但不影响新请求。

## 3. Versatile 集成

Versatile 是业务工作流引擎，EDPAgent 通过 `call_versatile` 工具将业务操作委托给 Versatile 执行。

### 两种调用模式

#### REST 直连模式（默认）

EDPAgent 直接调用 Versatile 的 REST API。

**配置项**：

```yaml
edpa:
  agent:
    versatile:
      url: ${EDP_AGENT_VERSATILE_URL:http://localhost:30001/v1/0/agent-manager/workflows/{workflow_id}/conversations/{conversation_id}}
      timeout: ${EDP_AGENT_VERSATILE_TIMEOUT:30s}
      url-variables:
        workflow_id: mock_workflow
      query-params:
        type: controller
        workspace_id: "10"
      headers:
        content-type: application/json
        stream: "true"
```

| 环境变量 | 说明 | 默认值 |
|----------|------|--------|
| `EDP_AGENT_VERSATILE_URL` | REST API 地址模板，支持 `{workflow_id}`、`{conversation_id}` 路径变量 | - |
| `EDP_AGENT_VERSATILE_TIMEOUT` | 调用超时时间 | `30s` |

URL 模板中的路径变量在运行时由框架自动填充：
- `{workflow_id}`：根据 query_intent 路由到对应的工作流 ID（默认 `mock_workflow`，场景可配置路由规则）
- `{conversation_id}`：当前会话 ID，用于 Versatile 侧上下文关联

#### A2A 适配器模式

通过 A2A 协议与 Versatile 的 A2A 适配器通信。

**配置项**：

```yaml
edpa:
  agent:
    versatile:
      adapter-a2a-url: ${EDP_AGENT_VERSATILE_A2A_URL:http://localhost:8191/a2a}
```

| 环境变量 | 说明 | 默认值 |
|----------|------|--------|
| `EDP_AGENT_VERSATILE_A2A_URL` | Versatile A2A 适配器地址 | `http://localhost:8191/a2a` |

### Versatile 接口规范

REST 模式下，EDPAgent 向 Versatile 发送 POST 请求，请求体包含：

```json
{
  "query": "用户原始请求",
  "intent": "query_intent 值",
  "description": "query_description 值",
  "context": {
    "conversation_id": "会话ID",
    "tool_data": {}  // 从 ToolDataChannel 读取的前序数据
  }
}
```

Versatile 应返回 SSE 流式响应，支持以下响应类型：

| 响应类型 | 说明 |
|----------|------|
| 正常完成 | 返回最终结果 JSON，包含业务数据 |
| 需要用户输入 | 返回中断信号，携带需要追问的问题 |
| 错误 | 返回错误信息 |

### 使用 Mock 服务器测试

代码仓提供了 Mock Versatile 服务器用于开发测试：

```bash
cd mock
pip install flask requests  # 安装依赖
python versatile_main.py
```

Mock 服务器默认启动在 30001 端口，提供模拟的工作流：
- `wealth_recommend` - 理财推荐
- `product_buy` - 产品购买
- `balance_query` - 余额查询
- `transfer_round1`/`transfer_confirmed` - 转账流程
- `fund_recommend` - 基金推荐

配置环境变量指向 Mock：

```bash
EDP_AGENT_VERSATILE_URL=http://localhost:30001/v1/0/agent-manager/workflows/{workflow_id}/conversations/{conversation_id}
```

### 对接生产 Versatile

对接生产 Versatile 时需要确认：
1. REST API 地址和认证方式
2. 工作流 ID 与 intent 的映射规则
3. 请求/响应格式
4. 中断续传协议
5. 超时时间设置

## 4. MCP SSE 集成

MCP（Model Context Protocol）SSE 服务提供 Python 脚本沙箱执行环境。

### 配置项

```yaml
edpa:
  agent:
    mcpsse:
      master-url: ${EDP_MCP_MASTER_URL:}
      standby-url: ${EDP_MCP_STANDBY_URL:}
      access-token: ${EDP_MCP_ACCESS_TOKEN:}
      app-name: ${EDP_MCP_APP_NAME:}
```

| 环境变量 | 说明 |
|----------|------|
| `EDP_MCP_MASTER_URL` | MCP SSE 主服务 URL |
| `EDP_MCP_STANDBY_URL` | MCP SSE 备服务 URL（可选，主节点故障时自动切换） |
| `EDP_MCP_ACCESS_TOKEN` | 鉴权 Token（可选） |
| `EDP_MCP_APP_NAME` | 应用名称标识（可选） |

### 主备切换

MCP 客户端实现了自动故障切换：

```
请求 → 主URL ──成功──→ 返回结果
   │
   └──失败──→ 备URL ──成功──→ 返回结果
                │
                └──失败──→ 返回错误
```

每次调用都先尝试主节点，主节点不可用时自动切换到备节点。

### Python 依赖

MCP 沙箱需要的 Python 依赖在 [deploy/requirements-mcp.txt](file:///D:/AgentTool/EDPAgent_java/branch_edpa_java/agent-store/edp-agent-java/deploy/requirements-mcp.txt) 中定义。Docker 构建时会自动安装这些依赖。

如果使用外部 MCP 服务，请确保 MCP 环境中已安装所需的 Python 包。

### 不配置 MCP 的情况

如果场景不需要执行 Python 脚本（即不使用 call_mcp 工具），可以不配置 MCP：
1. 从 actrule.yaml 的 allowed_tools 中移除 `call_mcp`
2. 不设置 EDP_MCP_* 环境变量

## 5. 客户端集成（A2A 协议）

上层系统（客户端、其他 Agent）通过 A2A 协议与 EDPAgent 通信。

### 协议概述

- **端点**：`http://{host}:{port}/a2a`
- **协议**：JSON-RPC 2.0 over HTTP POST
- **响应**：SSE（Server-Sent Events）流式响应
- **发现端点**：`http://{host}:{port}/.well-known/agent-card.json`

### Agent Card 发现

访问 `/.well-known/agent-card.json` 获取 Agent 元数据：

```bash
curl http://localhost:8190/.well-known/agent-card.json
```

返回示例：

```json
{
  "agent": {
    "id": "edp-agent",
    "name": "通用动态规划智能体",
    "description": "企业级动态规划智能体运行时",
    "url": "http://localhost:8190/a2a",
    "version": "0.2.0"
  }
}
```

### 发送消息

**请求格式**：

```json
{
  "jsonrpc": "2.0",
  "id": "request-id-001",
  "method": "sendStreamingMessage",
  "params": {
    "message": {
      "messageId": "msg-001",
      "role": "user",
      "parts": [
        {
          "kind": "text",
          "text": "帮我推荐一些理财产品"
        }
      ]
    },
    "configuration": {
      "scenario": "wealth-demo"
    }
  }
}
```

**curl 示例**：

```bash
curl -N -X POST http://localhost:8190/a2a \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "id": "req-001",
    "method": "sendStreamingMessage",
    "params": {
      "message": {
        "messageId": "msg-001",
        "role": "user",
        "parts": [{"kind": "text", "text": "你好"}]
      }
    }
  }'
```

### SSE 事件流

响应为 SSE 格式，每个事件以 `event:` 和 `data:` 开头：

```
event: status
data: {"type":"processing","state":"planning"}

event: message
data: {"jsonrpc":"2.0","result":{...}}

event: message
data: {"jsonrpc":"2.0","result":{...}}
```

客户端需要解析 SSE 事件流，根据事件类型更新 UI。

### 中断续传

当 Agent 调用 ask_user 触发中断时，SSE 流中会返回中断事件。用户回复后，客户端使用相同的 conversationId 发送新消息，Agent 从断点恢复执行。

### Python 客户端示例

```python
import requests
import json
import sseclient

def send_message(url, message, conversation_id=None):
    headers = {
        "Content-Type": "application/json",
        "Accept": "text/event-stream"
    }
    
    payload = {
        "jsonrpc": "2.0",
        "id": "req-1",
        "method": "sendStreamingMessage",
        "params": {
            "message": {
                "messageId": "msg-1",
                "role": "user",
                "parts": [{"kind": "text", "text": message}]
            }
        }
    }
    
    if conversation_id:
        payload["params"]["configuration"] = {"conversationId": conversation_id}
    
    response = requests.post(url, json=payload, headers=headers, stream=True)
    client = sseclient.SSEClient(response)
    
    for event in client.events():
        data = json.loads(event.data)
        print(f"Event: {event.event}, Data: {data}")
        
        # 检查是否是中断事件（需要用户输入）
        if event.event == "interrupt":
            print("需要用户输入:", data)
            break

# 使用
send_message("http://localhost:8190/a2a", "帮我推荐理财产品")
```

### Java 客户端示例

使用 OkHttp 或 Spring WebClient 处理 SSE 流。

### 注意事项

1. **长连接**：SSE 是长连接，客户端需要设置合理的超时时间（建议 5 分钟以上）。
2. **会话管理**：客户端需要维护 conversationId，同一对话的后续消息需要携带相同的 conversationId。
3. **断点续传**：中断后重连时使用相同 conversationId，Agent 从 Checkpoint 恢复。
4. **错误处理**：HTTP 4xx/5xx 错误表示请求格式错误或服务异常；SSE 流中的 error 事件表示业务错误。

## 6. 配置文件锚点

除了环境变量，EDPAgent 还支持通过配置锚点文件加载配置：

```yaml
edpa:
  agent:
    config-path: ${EDPA_AGENT_CONFIG_PATH:src/main/resources/edp-config.yaml}
```

`edp-config.yaml` 是配置锚点文件，定义了场景发现和 Governance 配置加载路径。Docker 中位于 `/app/config/edp-config.yaml`。

## 健康检查

Docker 内置 HEALTHCHECK，每 10 秒访问 Agent Card 端点：

```dockerfile
HEALTHCHECK --interval=10s --timeout=5s --retries=5 --start-period=60s \
  CMD curl -sf http://localhost:8190/.well-known/agent-card.json || exit 1
```

`--start-period=60s` 表示启动 60 秒后开始健康检查，给服务启动留出时间。
